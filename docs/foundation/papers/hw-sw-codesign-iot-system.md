# IoT系统软硬件协同设计方法论
> **难度**：🟡 中级 | **领域**：系统设计方法 | **阅读时间**：约 20 分钟

## 引言

假设你要装修一套房子。如果先做完所有硬装(水电、墙地),再让家具进场,很可能发现插座被沙发挡住了,或者空调位置对着床头。好的装修应该是硬装和软装一起规划,每个位置都考虑最终的使用场景。IoT 系统设计也一样:硬件和软件必须协同设计,否则就会在集成阶段不断返工。

## 1 软硬件协同设计的概念

### 1.1 什么是协同设计

软硬件协同设计(HW/SW Co-Design)是指在系统设计阶段,硬件和软件团队共同参与架构决策,以整体最优为目标进行联合设计:

- 不是先做硬件再做软件的串行流程
- 不是各自设计最后拼接
- 而是从需求阶段就一起权衡取舍

### 1.2 传统流程的问题

传统的瀑布式开发流程:

```
需求 -> 硬件设计 -> 硬件验证 -> 软件开发 -> 系统集成 -> 测试
```

问题清单:

1. 硬件过度设计: 不了解软件需求,留过多余量
2. 硬件欠设计: 软件需要的功能硬件没支持
3. 接口不匹配: 寄存器映射、时序对不上
4. 返工成本高: 硬件改版周期 4-8 周,费用数万
5. 上市时间延迟: 问题在集成阶段集中爆发

### 1.3 协同设计流程

```
系统需求
   |
   v
架构探索(联合)
   |
   v
HW/SW 划分(联合决策)
   |
   v
接口规范(签署合约)
   |
   +--------+--------+
   |                  |
   v                  v
硬件开发          软件开发
(基于接口规范)   (基于接口规范)
   |                  |
   +--------+---------+
            |
            v
       集成验证
```

## 2 为什么 IoT 特别需要协同设计

### 2.1 资源约束极紧

IoT 设备的特点:低成本、低功耗、小体积。这意味着:

- MCU 选型直接影响软件能做什么
- 内存大小决定能否用 RTOS、能否跑协议栈
- 功耗预算决定无线通信策略

### 2.2 硬件选择影响软件架构

| 硬件决策 | 对软件的影响 |
|----------|-------------|
| 选 MCU vs MPU | 裸机 vs Linux |
| RAM 大小 | 能否跑蓝牙/WiFi 协议栈 |
| 有无硬件加密 | 软件是否需要软件加密库 |
| ADC 精度 | 软件是否需要额外滤波 |
| 电源管理 IC | 休眠策略和唤醒方式 |

### 2.3 软件需求影响硬件选择

| 软件需求 | 对硬件的要求 |
|----------|-------------|
| OTA 固件更新 | 双 Bank Flash 或足够 RAM |
| TLS 安全连接 | 硬件加密加速或足够 CPU 算力 |
| 实时响应(< 1ms) | 中断优先级架构、DMA 支持 |
| 多传感器并发 | 足够的 ADC 通道、DMA 通道 |
| 低功耗运行 | 低功耗外设、快速唤醒能力 |

## 3 划分决策

### 3.1 什么放硬件,什么放软件

划分原则:

- **延迟关键**: 放硬件(加密加速、PWM 生成、DMA 传输)
- **灵活性需求高**: 放软件(协议解析、业务逻辑、UI)
- **功耗敏感**: 分析具体场景,有时硬件更省电(硬件加密 vs CPU 软加密),有时软件更灵活(动态调频)

### 3.2 功耗预算分配

典型 IoT 节点的功耗预算分解:

```
总功耗预算: 10 mW (钮扣电池供电)

MCU 活跃:    3 mW (30%)
无线通信:    4 mW (40%)
传感器:      1 mW (10%)
其他模拟:    1 mW (10%)
余量:        1 mW (10%)
```

如果无线通信选 BLE 而非 WiFi,无线功耗可从 4mW 降到 0.5mW,但这限制了数据吞吐量。这个决策需要软硬件团队共同做出。

### 3.3 决策矩阵

对每个功能模块做如下分析:

| 功能 | 硬件实现 | 软件实现 | 决策依据 |
|------|----------|----------|----------|
| ADC 采样 | 必需 | 触发和读取 | 硬件 |
| 数字滤波 | FIR 加速器 | CMSIS-DSP | 看阶数和采样率 |
| 加密 | AES 硬件 | mbedTLS 软件 | 有硬件就用硬件 |
| 协议栈 | 无 | 必需 | 软件 |
| PWM 控制 | 硬件定时器 | 软件位操作 | 硬件 |

## 4 接口规范

### 4.1 HW/SW 边界定义

在并行开发之前,必须签署一份接口规范,涵盖:

1. **寄存器映射**: 每个寄存器的地址、位域定义、读写属性
2. **中断行为**: 触发条件、清除方式、优先级建议
3. **DMA 通道**: 可用通道、传输方向、缓冲区对齐要求
4. **时序约束**: 建立时间、保持时间、最大响应延迟
5. **上电顺序**: 哪些模块先使能、哪些需要延迟等待

### 4.2 寄存器映射示例

```
基地址: 0x4000_1000

偏移   名称        位域          描述
0x00   CTRL       [0] EN        模块使能
                  [1] START     启动转换
                  [2] DMA_EN    DMA 使能
0x04   STATUS     [0] BUSY      转换进行中
                  [1] DONE      转换完成(写1清除)
                  [2] OVERRUN   溢出标志
0x08   DATA       [15:0]        转换结果
0x0C   CONFIG     [2:0] CH_SEL  通道选择
                  [5:3] RATE    采样率选择
```

### 4.3 硬件抽象层 HAL 作为合约

HAL 是软硬件之间的合约层:

```c
// HAL 接口(软件团队定义,硬件团队确认)
typedef struct {
    void (*init)(void);
    void (*start_conversion)(uint8_t channel);
    bool (*is_busy)(void);
    uint16_t (*read_data)(void);
    void (*register_callback)(void (*cb)(uint16_t));
} adc_hal_t;
```

硬件团队确保 HAL 底层实现正确,软件团队只通过 HAL 接口操作,不直接碰寄存器。这样即使硬件改版(寄存器地址变了),软件只需要更新 HAL 实现,上层代码不受影响。

## 5 架构探索工具

### 5.1 SystemC 系统级建模

SystemC 可以在 RTL 之前做系统级仿真:

```cpp
SC_MODULE(adc_model) {
    sc_in<bool> start;
    sc_out<bool> done;
    sc_out<sc_uint<16>> data;

    void convert() {
        if (start.read()) {
            wait(10, SC_US);  // 模拟转换延迟
            data.write(sample_value++);
            done.write(true);
        }
    }
    SC_CTOR(adc_model) {
        SC_METHOD(convert);
        sensitive << start;
    }
};
```

软件团队可以基于 SystemC 模型提前开发驱动,不等硬件。

### 5.2 QEMU 虚拟原型

QEMU 可以模拟目标处理器,运行实际固件:

- 在硬件完成前就能运行和调试软件
- 可以注入故障测试错误处理路径
- 支持 GDB 远程调试

### 5.3 FPGA 原型验证

对于有自定义逻辑的设计:

- 在 FPGA 上实现硬件模块
- 软件在真实处理器上运行
- 验证 HW/SW 接口和时序

## 6 实战案例: IoT 振动监测器

### 6.1 需求

- 3 轴加速度计,采样率 10 kHz
- 256 点 FFT,每 100ms 一次
- 特征频率检测和报警
- BLE 传输数据到手机
- 电池供电,续航 1 年

### 6.2 划分分析: FFT 硬件 vs 软件

**软件方案**: CMSIS-DSP 库在 Cortex-M4 上跑 256 点 FFT:

```
Cortex-M4 @ 80 MHz
256 点复数 FFT: 约 2000 个周期
执行时间: 2000/80M = 25 us
每 100ms 一次, CPU 占用: 0.025%
```

结论: MCU 完全能处理,不需要 FPGA 或硬件 FFT 加速器。省去了硬件复杂度和成本。

### 6.3 最终划分

| 功能 | 实现 | 理由 |
|------|------|------|
| ADC 采样 | 硬件(DMA) | 解放 CPU |
| FFT | 软件(CMSIS-DSP) | MCU 算力足够 |
| 特征检测 | 软件 | 需要灵活性 |
| BLE 协议栈 | 软件+硬件射频 | 标准 BLE |
| 电源管理 | 硬件+软件协作 | 联合优化 |

### 6.4 接口规范关键条款

- ADC 通过 DMA 传输到循环缓冲区,半满和全满中断通知软件
- FFT 每 100ms 执行一次,在 DMA 半满中断中触发
- BLE 在 FFT 完成后发送特征数据,最大 20 字节/包

## 7 团队协作

### 7.1 共享规范文档

所有 HW/SW 接口必须文档化,建议包含:

- 寄存器映射表
- 中断和 DMA 配置
- 时序图
- 错误处理流程
- 版本变更记录

### 7.2 定期集成检查点

```
Week 1: 接口规范评审
Week 3: 硬件原理图评审, 软件 HAL 框架评审
Week 5: 硬件 PCB 出图, 软件在评估板上跑通
Week 7: 硬件回板, 软件在真实硬件上调试
Week 8: 集成测试完成
```

### 7.3 HAL 作为合约

HAL 的价值:

1. 软件不直接依赖寄存器,硬件变更影响局部化
2. 软件可以在模拟器上开发(用 HAL 的 mock 实现)
3. 硬件调试时可以通过 HAL 注入测试向量

## 8 常见协同设计错误

### 8.1 硬件过度设计

没有软件输入就做硬件设计,结果:

- 加了不需要的外设,增加 BOM 成本
- 引脚分配不合理,软件布线困难
- 电源域划分不优,软件无法实现最佳低功耗策略

### 8.2 软件假设资源无限

软件团队没有参与硬件选型,结果:

- RAM 不够跑协议栈
- Flash 不够放 OTA 双镜像
- 没有 DMA 导致 CPU 全在搬运数据

### 8.3 接口变更未同步

硬件改了寄存器地址,没通知软件;或者软件改了 DMA 配置,没告诉硬件验证。结果:

- 调试时双方都说"我这边没问题"
- 浪费数天排查

### 8.4 没有 HAL 导致紧耦合

软件直接操作寄存器,硬件改版后:

- 所有软件都要改
- 容易遗漏导致隐性 bug
- 回归测试成本巨大

## 9 敏捷硬件开发

### 9.1 模块化设计

将硬件分为核心模块和外扩模块:

- 核心模块: MCU + 电源 + 调试接口(稳定不变)
- 外扩模块: 传感器接口、通信模块(可替换)

### 9.2 评估板先行

在自定义硬件完成前,用评估板:

- 软件团队提前开发
- 验证功能可行性
- 优化性能参数

### 9.3 FPGA 原型

对于有自定义数字逻辑的设计:

- 用 FPGA 先实现,软件提前验证
- 确认功能正确后再做 ASIC/定制 IC
- 避免昂贵的流片失败

## 总结

IoT 系统的软硬件协同设计不是可选项,而是必须:

1. 从需求阶段就让软硬件团队共同参与
2. 接口规范是并行开发的基础,必须详细且双方签署
3. HAL 是软硬件之间的合约,不可或缺
4. 用 SystemC/QEMU/评估板提前验证,不要等硬件回板
5. 避免常见错误:过度设计、假设资源无限、接口不同步

记住:集成阶段发现的问题,修复成本是设计阶段的 10-100 倍。前期多花时间在协同设计上,是最划算的投资。
