# 真随机数发生器TRNG硬件实现与评估
> **难度**：🔴 高级 | **领域**：密码学硬件 | **阅读时间**：约 22 分钟

## 引言

想象一下,你正在玩猜硬币游戏。如果硬币是公平的,每次正反面的概率各50%,对手永远无法预测结果。但如果硬币有偏,比如60%的概率出正面,对手观察足够多次后就能发现规律,从而获胜。密码学中的随机数就是这枚硬币——如果随机数可预测,再强大的加密算法也会形同虚设。

真随机数发生器(TRNG)利用物理现象产生不可预测的随机数,是整个密码学安全大厦的基石。没有可靠的TRNG,密钥可能是可预测的,随机数可能是有规律的,所有安全假设都会崩塌。

## 1. 随机数在安全中的核心地位

### 1.1 为什么随机数如此关键

密码学的几乎每个环节都依赖随机数的不可预测性:

| 应用场景 | 随机数用途 | 预测后果 |
|----------|-----------|---------|
| 密钥生成 | 密钥值来自随机数 | 攻击者可推算密钥 |
| Nonce | 防止重放攻击 | 可伪造消息 |
| 初始化向量(IV) | CBC/GCM模式的随机化 | 相同明文产生相同密文 |
| 会话令牌 | 身份认证凭证 | 可劫持会话 |
| 盐值(Salt) | 密码哈希的随机化 | 可预计算彩虹表 |
| 临时密钥 | 前向保密(DHE/ECDHE) | 可解密历史通信 |

### 1.2 随机数失败的著名案例

```
案例1: PlayStation 3 ECDSA密钥泄露(2010)
- 未使用真随机数作为k值
- 两次签名使用了相同的k值
- 攻击者从两个签名中直接推算出私钥
- 后果: PS3签名验证被完全绕过

案例2: Debian OpenSSL漏洞(2006-2008)
- 代码修改导致随机数熵源仅剩15位
- 可能的密钥数量从2^256降到2^15
- 后果: 所有Debian生成的SSH密钥和SSL证书可被穷举
```

## 2. PRNG vs TRNG

### 2.1 伪随机数发生器(PRNG)

PRNG(Pseudo-Random Number Generator)使用确定性算法从种子(seed)生成随机数序列:

```
PRNG工作原理:
Seed ---> [Deterministic Algorithm] ---> Random Sequence
              |
              v
         如果Seed已知,输出完全可预测
         如果Seed未知,输出看似随机
```

PRNG的特点:

- **确定性**:相同种子产生相同序列(这在测试中有用)
- **快速**:纯软件实现,每秒可生成数百MB
- **可重现**:给定种子可重现整个序列
- **安全性完全依赖种子**:种子可预测则输出可预测

### 2.2 真随机数发生器(TRNG)

TRNG(True Random Number Generator)利用物理熵源产生真正不可预测的随机数:

```
TRNG工作原理:
Physical Entropy Source ---> [Digitizer] ---> Raw Bits
                                          |
                                     [Health Test]
                                          |
                                   [Conditioning]
                                          |
                                    Random Output
```

TRNG的特点:

- **不可预测**:基于物理过程,无法在数学上预测
- **不可重现**:即使知道所有参数,也无法重现输出
- **速度有限**:受物理过程速度限制
- **可能偏差**:原始输出可能有统计偏差,需要后处理

### 2.3 PRNG与TRNG对比

| 维度 | PRNG | TRNG |
|------|------|------|
| 来源 | 数学算法 | 物理过程 |
| 可预测性 | 种子已知则可预测 | 不可预测 |
| 速度 | 极快(GB/s) | 有限(kB-MB/s) |
| 可重现性 | 可重现 | 不可重现 |
| 统计质量 | 极好(设计良好时) | 需要后处理 |
| 安全基础 | 种子的保密性 | 物理过程的不可预测性 |
| 硬件需求 | 无 | 专用电路 |

### 2.4 实际应用:TRNG种子DRBG

现代安全系统通常将两者结合使用:

```
[TRNG] ---> 高质量种子 ---> [DRBG (CSPRNG)] ---> 大量随机数
                             (NIST SP 800-90A)
```

DRBG(Deterministic Random Bit Generator)是由TRNG种子化的CSPRNG,兼具安全性和性能:

- **安全性**:种子来自TRNG,不可预测
- **性能**:DRBG算法快速,可满足大量需求
- **前向安全**:每次生成后更新内部状态,已知状态不能推算之前输出
- **后向安全**:定期从TRNG补充熵,已知状态不能推算未来输出

## 3. TRNG的熵源

### 3.1 热噪声(Thermal Noise)

最常见的TRNG熵源:

```
电阻热噪声:
  - 来源: 导体中电子的随机热运动
  - 幅度: 微伏级(需要放大)
  - 频谱: 白噪声(平坦频谱)
  - 优点: 真正随机,无处不在
  - 缺点: 幅度极小,需要高增益放大器

二极管击穿噪声:
  - 来源: 雪崩击穿或齐纳击穿中载流子的随机运动
  - 幅度: 毫伏级
  - 优点: 比电阻噪声幅度大
  - 缺点: 需要偏置电路
```

### 3.2 环形振荡器抖动(Ring Oscillator Jitter)

MCU中最常用的TRNG实现方式:

```
环形振荡器结构:
+---+   +---+   +---+   +---+
|NOT|-->|NOT|-->|NOT|-->|NOT|--+
+---+   +---+   +---+   +---+  |
   ^                              |
   +------------------------------+

抖动来源:
  - 热噪声导致门延迟随机变化
  - 电源噪声耦合到延迟
  - 相邻电路的串扰
  - 每个周期累积,多次翻转后抖动显著
```

抖动量化:
- 单个门延迟抖动:皮秒级
- N级环形振荡器累积:N倍增强
- 多个环形振荡器采样:进一步增大熵

### 3.3 亚稳态触发器(Metastable Flip-Flop)

```
亚稳态原理:
  - 触发器在建立/保持时间违例时进入亚稳态
  - 输出在0和1之间随机振荡
  - 最终稳定到0或1,但结果不可预测
  - 利用时钟抖动故意制造时序违例

实现方式:
  [Fast Clock] ---> [D Flip-Flop] <--- [Slow Clock with Jitter]
                       |
                   Random Bit
```

### 3.4 其他熵源

| 熵源 | 原理 | 应用场景 |
|------|------|---------|
| 放射性衰变 | 原子核衰变的随机性 | 高安全级实验室设备 |
| 量子效应 | 光子偏振/路径的量子随机性 | 量子随机数发生器(QRNG) |
| 光子散粒噪声 | 光电检测中光子的随机到达 | 光学TRNG |
| 大气噪声 | 无线电接收的大气噪声 | 早期硬件随机数源 |

## 4. TRNG架构

### 4.1 完整TRNG流水线

```
+----------+    +----------+    +------------+    +-------------+    +--------+
| Entropy  |    |Digitizer |    |  Health    |    |Conditioning |    |Output  |
| Source   |--->| Sampler  |--->|  Test      |--->|(Whitening)  |--->|Interface|
|(Analog)  |    |(ADC/Comp)|    |            |    |             |    |        |
+----------+    +----------+    +------------+    +-------------+    +--------+
     |               |               |                   |                |
  热噪声/         模拟转数字      统计检验          去偏差/增强        随机数
  环振抖动        0/1序列         检测故障           统计质量          输出
```

### 4.2 各阶段详解

**1. 熵源(Entropy Source)**

产生模拟随机信号:
- 热噪声:连续模拟信号
- 环振抖动:数字时钟抖动
- 亚稳态:数字0/1随机

**2. 数字化器(Digitizer)**

将模拟/半模拟信号转换为数字位:
- 比较器:模拟信号与阈值比较,输出0或1
- 采样器:在特定时刻采样环振输出
- 亚稳态采样:直接读取亚稳态触发器输出

**3. 健康测试(Health Test)**

实时监测熵源是否正常工作:
- 上电时:完整统计测试
- 运行时:轻量级连续测试
- 故障时:停止输出,报警

**4. 调节/白化(Conditioning)**

消除原始数据的统计偏差:
- 去偏差:消除0/1概率不等
- 增强:提高每位的熵
- 密码学调节:通过加密哈希进一步增强

## 5. 健康测试

### 5.1 重复计数测试(Repetition Count Test)

NIST SP 800-90B规定的必要测试:

```
目的: 检测熵源是否"卡住"(连续输出相同值)

方法:
1. 统计连续相同输出的次数
2. 如果连续相同次数超过阈值C,报告故障

阈值计算:
  C = 1 + ceil(-log(alpha) / H)
  其中 alpha = 2^(-20) (误报概率)
       H = min-entropy (估计的最小熵)

示例:
  如果 H = 0.9 (每bit), alpha = 2^(-20)
  C = 1 + ceil(20 / 0.9) = 1 + 23 = 24
  连续24个相同bit = 故障
```

### 5.2 自适应比例测试(Adaptive Proportion Test)

```
目的: 检测0和1的比例是否严重偏离50%

方法:
1. 在滑动窗口W内统计0和1的个数
2. 如果0或1的计数超过阈值,报告故障

窗口大小:
  - 连续模式: W = 512 (二值输出)
  - 离散模式: W = 1024

阈值:
  C = ceil(W/2) + K
  K由显著性水平alpha和窗口W计算
```

### 5.3 其他健康测试

| 测试 | 目的 | 标准参考 |
|------|------|---------|
| 重复计数 | 检测卡死 | NIST SP 800-90B |
| 自适应比例 | 检测偏差 | NIST SP 800-90B |
| 卡方测试 | 分布均匀性 | AIS 31 |
| 游程测试 | 连续0/1分布 | FIPS 140-1 |
| 自相关测试 | 输出独立性 | NIST SP 800-22 |

## 6. 调节算法(Conditioning)

### 6.1 冯诺依曼去偏差(Von Neumann Debiasing)

最简单的去偏差方法:

```
原理: 利用相邻bit对消除偏差

规则:
  00 -> 丢弃
  11 -> 丢弃
  01 -> 输出0
  10 -> 输出1

示例(原始序列有60%概率出1):
  原始: 1 1 0 1 0 1 0 0 1 0
  分组: (1,1)(0,1)(0,1)(0,0)(1,0)
  处理: 丢弃   0     0    丢弃   1
  输出: 0 0 1

优点: 不需要假设偏差值,简单有效
缺点: 输出速率约为输入的1/4
```

### 6.2 密码学调节

使用加密算法进行调节,安全性更高:

**AES-CBC-MAC调节:**

```
输入: n组原始数据 D1, D2, ..., Dn (每组128bit)
Key:  固定密钥K

Output = AES-CBC-MAC(K, D1 || D2 || ... || Dn)

特点:
  - 输出统计质量接近理想随机
  - 即使原始数据有一定偏差,输出仍然均匀
  - 需要AES硬件/软件支持
```

**SHA-256调节:**

```
输入: 原始数据 D (任意长度)
Output = SHA-256(D)

特点:
  - 简单,不需要密钥管理
  - 抗碰撞性保证输出不可逆
  - NIST SP 800-90B认可
```

### 6.3 NIST SP 800-90A DRBG

TRNG输出种子化DRBG,由DRBG提供最终随机数:

```
[TRNG] --熵--> [DRBG Instantiate] ---> 随机数输出
                    |
               [Reseed] <--- [TRNG] (定期补充熵)

DRBG算法选择:
1. CTR-DRBG (基于AES-CTR): 最常用,性能好
2. Hash-DRBG (基于SHA-2): 实现简单
3. HMAC-DRBG (基于HMAC-SHA-2): 安全证明完备
```

## 7. MCU内置TRNG

### 7.1 STM32 RNG外设

STM32多款MCU内置RNG外设,以环形振荡器为熵源:

```
STM32 RNG框图:
+---------------------------------------------+
| RNG Peripheral                               |
|  +------------------+    +----------------+  |
|  | Analog Noise    |--->| Post-Processing|  |
|  | Source           |    | (Conditioning) |  |
|  | (Ring Oscillators)|   |                |  |
|  +------------------+    +----------------+  |
|                           |                   |
|                      +--------+              |
|                      |RNG_DR  |              |
|                      |(32-bit)|              |
|                      +--------+              |
+---------------------------------------------+
       |
  AHB Bus (读取随机数)
```

关键特性:
- **熵源**:多个环形振荡器,频率互相异或采样
- **后处理**:LFSR(线性反馈移位寄存器)白化
- **输出**:32位随机数,每次AHB读取
- **时钟**:需要48MHz HSI48或PLL48
- **错误检测**:时钟错误检测(CES标志),种子错误检测(SECS标志)

### 7.2 STM32 HAL_RNG使用

```c
// STM32 HAL库使用RNG外设
#include "stm32l4xx_hal.h"

RNG_HandleTypeDef hrng;

// 初始化RNG
void RNG_Init(void) {
    hrng.Instance = RNG;
    hrng.Init.ClockErrorDetection = RNG_CED_ENABLE;
    HAL_RNG_Init(&hrng);
}

// 获取一个32位随机数
uint32_t get_random(void) {
    uint32_t random_value;
    if (HAL_RNG_GenerateRandomNumber(&hrng, &random_value) != HAL_OK) {
        // 错误处理: 熵源故障或时钟异常
        Error_Handler();
    }
    return random_value;
}

// 生成多个随机字节
void get_random_bytes(uint8_t *buf, size_t len) {
    uint32_t val;
    for (size_t i = 0; i < len; i += 4) {
        val = get_random();
        buf[i]   = (val >> 24) & 0xFF;
        buf[i+1] = (val >> 16) & 0xFF;
        buf[i+2] = (val >>  8) & 0xFF;
        buf[i+3] = (val)       & 0xFF;
    }
}
```

### 7.3 嵌入式Linux的/dev/random

嵌入式Linux系统中的熵池设计:

```
Linux内核熵池架构:
+----------+     +----------+     +----------+
| 硬件TRNG |---->|          |     |          |
+----------+     |  Input   |---->|  Output  |---> /dev/random
                 |  Pool    |     |  Pool    |     (阻塞,高熵)
+----------+     | (4096bit)|     | (2048bit)|---> /dev/urandom
| 键盘/鼠标|---->|          |     |          |     (非阻塞)
+----------+     +----------+     +----------+
                                          |
+----------+                              |
| 磁盘中断 |-----> 混入熵池               |
+----------+                              |
                                     [DRBG提取]

嵌入式Linux最佳实践:
1. 启用硬件TRNG: echo "hrng" > /dev/hwrng
2. 使用rngd守护进程: rngd -r /dev/hwrng
3. /dev/urandom足够安全(现代内核已修复)
4. 长期密钥生成仍建议从/dev/random读取
```

```
  6. 出具认证报告
```

### 8.3 最小熵估计(Min-Entropy Estimation)

最小熵是衡量TRNG质量的关键指标:

```
Shannon熵: H = -sum(p_i * log2(p_i))
  描述: 平均信息量
  用途: 通信编码

最小熵: H_min = -log2(max(p_i))
  描述: 最坏情况下的不可预测性
  用途: 密码学安全评估

示例:
  偏差币: 60%正面, 40%反面
  Shannon熵 = 0.971 bit
  最小熵 = -log2(0.6) = 0.737 bit

  密码学关心的是最小熵: 攻击者总会猜测最可能的结果
```

NIST SP 800-90B提供了多种最小熵估计方法:

- 最常见值估计
- 碰撞估计
- 马尔可夫估计
- 压缩估计
- 取多种方法的最小值作为最终估计

## 9. 常见实现错误

### 9.1 熵积累不足

```
错误做法:
  // MCU刚上电,立即生成密钥
  uint8_t key[32];
  for (int i = 0; i < 32; i++) {
      key[i] = read_rng();  // 熵源可能还未稳定
  }

正确做法:
  // 等待TRNG就绪
  while (!is_rng_ready()) {
      // 等待健康测试通过
  }
  // 确认足够熵后再生成密钥
  uint8_t key[32];
  for (int i = 0; i < 32; i++) {
      key[i] = read_rng();
  }
```

### 9.2 缺少健康监测

```
错误做法:
  // 只在初始化时测试一次
  rng_self_test();  // 上电自检
  // 之后永远不再检查

正确做法:
  // 运行时持续监测
  while (1) {
      if (!rng_health_test()) {
          // 熵源故障,停止使用
          rng_disable();
          alert_security_event();
      }
      // 正常使用
      random_value = read_rng();
  }
```

### 9.3 直接使用TRNG输出不经调节

```
错误做法:
  // 假设TRNG输出完美均匀
  key = trng_read();  // 可能有统计偏差

正确做法:
  // 通过DRBG或密码学哈希调节
  raw_entropy = trng_read();
  key = SHA256(raw_entropy);  // 密码学调节
  // 或
  drbg_reseed(raw_entropy);
  key = drbg_generate(32);
```

## 10. TRNG吞吐量与应用需求

### 10.1 不同应用的随机数需求

| 应用 | 需求量 | 频率 | 可接受方案 |
|------|--------|------|-----------|
| 密钥生成 | 32-64字节 | 一次性 | TRNG直接读取 |
| TLS握手 | 数百字节 | 每次连接 | TRNG种子DRBG |
| AES-GCM IV | 12字节 | 每次加密 | 计数器或TRNG |
| 高速加密 | 连续流 | 持续 | DRBG(定期重种子) |
| 批量密钥生成 | 数KB | 批量 | DRBG |

### 10.2 吞吐量匹配

```
典型TRNG吞吐量:
  MCU内置RNG:  ~1 Mbps
  独立TRNG芯片: ~10 Mbps
  PCIe HSM:   ~100 Mbps

应用需求匹配:
  密钥生成(32B): TRNG直接输出即可
  TLS(每连接1KB): TRNG + DRBG
  高速加密(>1MB/s): 必须用DRBG,TRNG仅种子
```

## 11. 攻击向量

### 11.1 温度攻击

```
原理: 改变温度影响热噪声和环振频率

攻击方法:
  - 加热: 热噪声幅度增大,但环振可能变慢
  - 冷却: 热噪声减小,TRNG可能输出固定值
  - 极端温度: 可能导致TRNG停振或卡死

防御:
  - 温度传感器监测
  - 健康测试检测输出异常
  - 超出工作温度范围时禁用TRNG
```

### 11.2 电压攻击

```
原理: 改变供电电压影响电路行为

攻击方法:
  - 降低电压: 环振变慢,可能进入确定状态
  - 电压毛刺: 可能使采样器捕获错误值
  - 电源纹波注入: 在特定频率注入,影响环振

防御:
  - 电压监测传感器
  - 独立供电(模拟/数字分离)
  - 电源滤波
```

### 11.3 电磁干扰(EMI)注入

```
原理: 注入特定频率的电磁信号影响TRNG行为

攻击方法:
  - 在环振频率附近注入信号: 可能"锁定"环振
  - 在采样时钟频率注入: 可能影响采样时机
  - 高功率EMI: 可能干扰数字化电路

防御:
  - 电磁屏蔽
  - 多源异或(多个独立熵源)
  - 调节算法消除外部影响
  - 健康测试检测异常模式
```

### 11.4 防御策略总结

```
多层防御:
  L1: 物理防护(屏蔽、封装)
  L2: 环境监测(温度、电压传感器)
  L3: 健康测试(重复计数、自适应比例)
  L4: 密码学调节(哈希、AES)
  L5: DRBG(前向/后向安全)
  L6: 审计(记录所有TRNG故障事件)
```

## 总结

TRNG是密码学安全的基础设施,核心要点:

1. **安全基石**:所有密码学操作都依赖随机数的不可预测性,TRNG提供真正的不可预测性
2. **物理熵源**:热噪声、环振抖动、亚稳态等物理过程是随机性的根本来源
3. **健康测试不可省略**:重复计数测试和自适应比例测试是运行时安全的保障
4. **调节必不可少**:原始TRNG输出几乎总有偏差,必须经过密码学调节
5. **TRNG + DRBG组合**:TRNG提供高质量种子,DRBG提供高性能随机数输出
6. **攻击面需要防护**:温度、电压、EMI都是潜在攻击向量,需要多层防御

对于IoT设备开发者,使用MCU内置TRNG时务必启用健康监测,并通过DRBG或密码学哈希调节输出,切勿直接使用原始TRNG数据。
