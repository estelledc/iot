# I2C与SPI总线选型决策指南

> **难度**：🟢 初级 | **领域**：总线选型 | **阅读时间**：约 15 分钟

## 引言

想象你要给一栋楼里的每个房间装门铃。方案 A：每个房间拉一根线到前台，前台按房间号区分是谁在按铃——这就是 SPI 的思路，每个设备一根专属"选择线"。方案 B：所有房间共用一根铃线，但每个房间有不同地址，前台收到信号后根据地址判断——这就是 I2C 的思路，用地址代替物理线。

两种方案都能完成任务，但各有取舍。SPI 用更多线换来了更快的速度和更简单的协议；I2C 用更少的线换来了更简洁的布线和更灵活的设备扩展。在 IoT 项目中，选错总线可能导致引脚不够用、采样率达不到、或者 PCB 布线变成噩梦。这篇文章帮你建立清晰的选型决策框架。

## 1. 两种总线的基本原理

### 1.1 I2C：地址寻址的两线总线

I2C (Inter-Integrated Circuit) 由 Philips（现 NXP）于 1982 年发明，最初是为了电视机内部芯片通信设计的。核心特征：

- **两根信号线**：SCL(时钟) + SDA(数据)
- **地址寻址**：每个从设备有唯一地址（通常 7-bit，也有 10-bit 扩展）
- **多主多从**：总线上可以有多个主设备
- **应答机制**：每传输一个字节，接收方返回 ACK/NACK

```
       VCC
        |
   +----+----+----+----+
   |    |    |    |    |
  RP   RP   RP   RP   RP  (上拉电阻)
   |    |    |    |    |
   +----+----+----+----+---- SCL ---- Master
   |    |    |    |    |
   +----+----+----+----+---- SDA ---- Master
        |         |        |
      Slave1   Slave2   Slave3
      (0x48)   (0x68)   (0x76)
```

### 1.2 SPI：片选寻址的四线总线

SPI (Serial Peripheral Interface) 由 Motorola 于 1980 年代中期发明，最初用于微处理器与外设之间的快速通信。核心特征：

- **四根信号线**：SCLK(时钟) + MOSI(主出从入) + MISO(主入从出) + CS(片选)
- **片选寻址**：每个从设备需要独立的 CS 线
- **单主多从**：通常只有一个主设备
- **无应答机制**：数据发出即认为传输完成，没有确认环节

```
                    +--------+
                    | Master |
                    +--------+
                   /   |  |   \
                SCLK MOSI MISO \
                /     |    |    \
          +----+  +----+  +----+  +----+
          |CS1 |  |CS2 |  |CS3 |  |CS4 |
          +----+  +----+  +----+  +----+
            |       |       |       |
          Slave1  Slave2  Slave3  Slave4
```

### 1.3 一句话区分

I2C 用"地址"找人，SPI 用"专线"找人。前者省线但需要协议开销，后者费线但直来直去。

## 2. 核心参数对比

### 2.1 总览对比表

| 参数 | I2C | SPI |
|------|-----|-----|
| 信号线数 | 2 (SCL, SDA) | 4+ (SCLK, MOSI, MISO, CS x N) |
| 寻址方式 | 软件地址(7/10-bit) | 硬件片选(CS引脚) |
| 最大速率 | 标准模式 100 kHz; 快速模式 400 kHz; 高速模式 3.4 MHz | 通常 1-50 MHz，取决于器件 |
| 双工模式 | 半双工 | 全双工 |
| 应答机制 | 有(ACK/NACK) | 无 |
| 多主支持 | 支持 | 不支持(需额外逻辑) |
| 协议开销 | 地址字节 + ACK | 无(直接数据) |
| 标准化 | 严格(NXP 规范) | 灵活(无统一标准) |
| 最大从设备数 | 127(7-bit) / 1023(10-bit) | 受可用 CS 引脚数限制 |

### 2.2 引脚消耗：N 台设备需要多少脚？

这是选型时最容易被低估的因素。假设 MCU 连接 N 个从设备：

| 从设备数量 | I2C 引脚 | SPI 引脚 | 差值 |
|-----------|---------|---------|------|
| 1 | 2 | 4 | +2 |
| 2 | 2 | 5 | +3 |
| 3 | 2 | 6 | +4 |
| 4 | 2 | 7 | +5 |
| 8 | 2 | 11 | +9 |
| 16 | 2 | 19 | +17 |

SPI 引脚数 = 3(固定) + N(CS 线)。当设备数量增多，SPI 的引脚压力急剧上升。

注意：SPI 可用 3-to-8 译码器(如 74HC138)将 3 根 GPIO 扩展为 8 路 CS，但这增加了额外器件和布线复杂度。

### 2.3 吞吐量实测对比

理论速率不等于实际有效吞吐量。I2C 有协议开销，SPI 也因 CS 切换有微小损耗：

| 操作 | I2C (400 kHz) | SPI (10 MHz) |
|------|--------------|--------------|
| 读 1 字节有效吞吐 | ~72 kHz (18% 效率) | ~5 MHz (50% 效率) |
| 读 16 字节有效吞吐 | ~280 kHz (70% 效率) | ~8 MHz (80% 效率) |
| 读 256 字节有效吞吐 | ~360 kHz (90% 效率) | ~9 MHz (90% 效率) |

I2C 每次transaction的开销包括：起始位(1) + 设备地址(7) + 读/写位(1) + ACK(1) = 10 bit。这使得短帧传输效率很低。但对于典型的传感器读数(2-6 字节)，400 kHz I2C 已足够。

### 2.4 功耗对比

| 场景 | I2C | SPI |
|------|-----|-----|
| 空闲态 | 上拉电阻持续消耗(VCC/Rp) | CS 拉高，几乎为零 |
| 通信态 | 与速率正相关 | 与速率正相关 |
| 低功耗优化 | 可关闭上拉(开漏+GPIO控制) | CS 拉高即可隔离 |

I2C 的上拉电阻是功耗"暗坑"。假设 VCC=3.3V，Rp=4.7k，每根线空闲电流约 0.7 mA，两根线共 1.4 mA。对电池供电设备不可忽略。解决方案包括：使用更大阻值(10k-47k)、GPIO 控制上拉使能、或切换到 SPI。

## 3. 选型关键考量

### 3.1 何时选择 I2C

以下场景优先考虑 I2C：

1. **多台低速传感器共享总线**：温湿度、气压、光照等传感器采样率通常在 1-100 Hz，400 kHz I2C 绑绑有余
2. **MCU 引脚紧张**：例如 ESP32-C3 (22 GPIO) 连接 5 个传感器，I2C 只需 2 脚，SPI 需要 8 脚
3. **板内短距离通信**：I2C 设计初衷就是芯片间(同一 PCB 上)通信，走线长度通常 < 30 cm
4. **设备地址可配置**：通过引脚或寄存器修改地址，同型号可挂多片
5. **需要热插拔**：I2C 的地址机制天然支持运行时动态添加设备

典型器件：BME280(温湿度气压)、MPU6050(IMU)、SSD1306(OLED 小屏，仅限低刷新率)

### 3.2 何时选择 SPI

以下场景优先考虑 SPI：

1. **高速 ADC/DAC**：例如 ADS1115(860 SPS 的 I2C 版) vs ADS8866(100 kSPS 的 SPI 版)，采样率差两个数量级
2. **显示屏驱动**：TFT 彩屏需持续刷新像素数据，SPI 的 10-80 MHz 时钟是刚需
3. **Flash 存储**：W25Q 系列 SPI Nor Flash，读取速度可达 50+ MHz
4. **实时数据流**：音频 ADC、高速数字麦克风(I2S 更专业，但 SPI 也能胜任中等速率场景)
5. **需要全双工通信**：同时读写，I2C 无法做到

典型器件：ILI9341(TFT 显示)、W25Q32(Flash)、ADS8866(ADC)、MAX31855(热电偶)

### 3.3 距离与信号完整性

| 距离 | I2C | SPI | 建议 |
|------|-----|-----|------|
| < 10 cm (同一 PCB) | 无问题 | 无问题 | 均可，按其他因素决定 |
| 10-30 cm (同一 PCB) | 可能需降低速率 | 无问题 | 优先 SPI |
| 30-100 cm (板间线缆) | 不推荐 | 仍可用，需注意阻抗 | SPI + 屏蔽线 |
| > 100 cm | 不适用 | 不适用 | 考虑 RS-485 / CAN |

I2C 的开漏输出加上拉电阻，导致上升沿由 RC 时间常数决定，距离越长上升越慢，信号越差。SPI 的推挽输出驱动能力强，能跑更远。

## 4. 混合总线设计

### 4.1 为什么需要混合设计

实际 IoT 项目中，几乎没有人只用一种总线。一个典型的环境监测节点可能同时需要：

- I2C 总线：BME280(温湿度)、BH1750(光照)、SCD40(CO2)
- SPI 总线：W25Q32(数据存储)、ILI9341(本地显示)
- 额外：1-Wire(DS18B20 备用温度)、UART(Debug)

### 4.2 混合设计原则

1. **按速率分流**：低速传感器走 I2C，高速外设走 SPI
2. **按引脚预算分配**：先统计可用 GPIO，再决定哪些设备走哪条总线
3. **避免总线竞争**：同一总线上不要混搭速率差异过大的设备(如 100 kHz 和 3.4 MHz 的 I2C 设备不能共存)
4. **注意电压域**：不同总线可能处于不同电压域，需要电平转换

### 4.3 引脚规划示例

以 STM32F103C8T6(48-pin, 37 GPIO)为例：

```
I2C1 (PB6=SCL, PB7=SDA):
  - BME280 (0x76)   温湿度气压
  - BH1750 (0x23)   光照
  - SCD40  (0x62)   CO2

SPI1 (PA5=SCK, PA6=MISO, PA7=MOSI):
  - CS1 (PA4): W25Q32  Flash存储
  - CS2 (PA3): ILI9341 TFT显示

UART1 (PA9/PA10): Debug输出
ADC1 (PA0): 电池电压采样
```

总计使用 12 个 GPIO，剩余 25 个留给扩展。如果全用 SPI，5 个设备需要 8 个 GPIO，且 CS 线管理更复杂。

## 5. 传感器生态与双接口器件

### 5.1 双接口器件

许多传感器同时提供 I2C 和 SPI 接口，通过引脚配置选择：

| 器件 | 类型 | I2C 速率 | SPI 速率 | 接口选择方式 |
|------|------|---------|---------|-------------|
| BME280 | 温湿度气压 | 400 kHz | 10 MHz | 寄存器配置 |
| LSM6DS3 | 6轴IMU | 400 kHz | 10 MHz | CSB 引脚电平 |
| ADS1115 / ADS1118 | 16-bit ADC | 400 kHz (1115) | 10 MHz (1118) | 不同型号 |
| MAX31855 / MAX31856 | 热电偶 | -- | 5 MHz | 仅SPI版本 |
| BMP388 | 气压 | 400 kHz | 10 MHz | CSB 引脚电平 |

注意：I2C 版本和 SPI 版本有时是不同型号(如 ADS1115 vs ADS1118)，购买时需确认。

### 5.2 选型建议

对于双接口器件，选择依据：

1. 总线上已有设备数量 -- I2C 设备多就继续用 I2C，避免引入新的 CS 线
2. 采样率需求 -- 传感器本身采样率 < 200 Hz 时，I2C 足够
3. 引脚余量 -- GPIO 紧张时优先 I2C

## 6. 决策流程图

以下流程图帮助你在项目初期快速做出选型判断：

```
开始选型
  |
  +--> 需要全双工通信? --是--> SPI
  |        |
  |        否
  |        |
  +--> 需要高速传输(>1Mbps)? --是--> SPI
  |        |
  |        否
  |        |
  +--> 需要连接多台设备(>4)且引脚紧张? --是--> I2C
  |        |
  |        否
  |        |
  +--> 传输距离 > 30cm? --是--> SPI (或考虑其他总线)
  |        |
  |        否
  |        |
  +--> 需要极低功耗(电池供电)? --是--> 评估I2C上拉功耗后决定
  |        |
  |        否
  |        |
  +--> 是否为Flash/Display/高速ADC? --是--> SPI
  |        |
  |        否
  |        |
  +--> 默认选择 I2C (引脚最省，布线最简)
```

简化版判断口诀：

- **高速大流量** --> SPI
- **低速多传感** --> I2C
- **屏和Flash** --> SPI
- **引脚不够用** --> I2C
- **拿不准** --> I2C 起步，性能不够再换 SPI

## 7. 实际项目案例

### 7.1 案例 1：气象站 (Weather Station)

需求：采集温度、湿度、气压、光照、风速，OLED 显示，数据存 SD 卡。

| 设备 | 总线选择 | 理由 |
|------|---------|------|
| BME280 (温湿度气压) | I2C | 低速传感器，400 kHz 绰绰有余 |
| BH1750 (光照) | I2C | 与 BME280 共享总线，省引脚 |
| 风速传感器 (脉冲量) | GPIO 中断 | 脉冲计数，非总线设备 |
| SSD1306 (OLED 128x64) | I2C | 小屏幕低刷新率，I2C 模式可用 |
| SD 卡 | SPI | 文件系统需要高速写入 |

引脚消耗：I2C(2) + SPI(SD卡: 4) = 6 GPIO。如果 OLED 也用 SPI，则需 8 GPIO，且两路 CS 管理。

设计要点：
- I2C 总线上挂 3 个设备(BME280 + BH1750 + SSD1306)，注意地址冲突排查
- SD 卡独占 SPI，避免与其他 SPI 设备共享 CS 时序冲突
- 如果后续加 CO2 传感器(SCD40)，直接挂 I2C 总线即可，零引脚增量

### 7.2 案例 2：工业振动监测仪 (Industrial Vibration Monitor)

需求：高频振动采样(10 kSPS+)、温度补偿、本地存储、Modbus 上报。

| 设备 | 总线选择 | 理由 |
|------|---------|------|
| ADS8866 (16-bit ADC) | SPI | 100 kSPS 采样率，I2C 完全无法满足 |
| ADXL355 (三轴加速度) | SPI | 高速输出数据速率(4 kHz ODR) |
| PT100 (温度) | SPI (通过 MAX31865) | RTD 测量需要 SPI 接口 |
| W25Q256 (32MB Flash) | SPI | 大量振动数据本地缓存 |
| RS-485 (Modbus) | UART | 工业标准通信接口 |

引脚消耗：SPI(3 固定 + 4 CS) + UART(2) = 9 GPIO。这个项目 I2C 没有出场机会——所有设备都需要高速或特定接口。

设计要点：
- 4 路 CS 线需合理分配 GPIO，避免与 UART 和调试口冲突
- SPI 总线速率需要取所有设备最低公共速率，或分时切换速率
- 振动数据量估算：3 轴 x 2 字节 x 10 kSPS = 60 KB/s，需确保 Flash 写入速度跟得上

### 7.3 案例对比总结

| 维度 | 气象站 | 工业振动监测仪 |
|------|--------|--------------|
| 总线类型 | I2C + SPI 混合 | 纯 SPI |
| I2C 设备数 | 3 | 0 |
| SPI 设备数 | 1 | 4 |
| 关键约束 | 引脚数量 | 采样速率 |
| 总线速率 | 400 kHz I2C / 10 MHz SPI | 20 MHz SPI |
| 设计难点 | 地址冲突排查 | CS 线管理 + 速率协调 |

## 8. 新兴替代方案：I3C

### 8.1 I3C 是什么

I3C (Improved Inter-Integrated Circuit) 由 MIPI 联盟于 2017 年发布，目标是"把 I2C 的简单和 SPI 的速度结合起来"。

### 8.2 I3C vs I2C vs SPI

| 特性 | I2C | SPI | I3C |
|------|-----|-----|-----|
| 信号线 | 2 | 4+ | 2 |
| 最大速率 | 3.4 MHz | 50+ MHz | 12.5 MHz (SDR) / 33 MHz (DDR) |
| 寻址方式 | 7/10-bit 地址 | CS 片选 | 动态地址分配 |
| 功耗 | 中(上拉电阻) | 低 | 低(推挽 + 上拉混合) |
| 向后兼容 | -- | -- | 兼容 I2C 从设备 |
| 热插拔 | 有限 | 不支持 | 原生支持 |
| 中断 | 无(需轮询) | 需额外线 | 带内中断(IBI) |
| 生态成熟度 | 极高 | 极高 | 中等(2024) |

### 8.3 I3C 的适用场景

I3C 目前主要用于高端手机和可穿戴设备中传感器总线的整合：

- 手机中替换 I2C 总线，一条 I3C 总线替代多条 I2C 总线
- 需要带内中断(IBI)的场景，省去额外的中断引脚
- 传感器数量多且需要动态配置的系统

### 8.4 IoT 项目是否需要考虑 I3C

对大多数初级 IoT 项目，I3C 暂时不是优先选择：

1. MCU 支持有限：主流 MCU(STM32/ESP32)的 I3C 外设还在普及阶段
2. 调试工具少：逻辑分析仪对 I3C 的协议解析支持不如 I2C/SPI 完善
3. 传感器生态：支持 I3C 的传感器型号还不多

建议：了解 I3C 的存在和发展趋势即可，当前项目继续以 I2C/SPI 为主。

## 9. 常见误区

### 9.1 "I2C 一定比 SPI 慢"

错误认知：I2C 最高只有 3.4 MHz，所以比 SPI 慢得多。

正确理解：在传感器场景中，速率瓶颈往往在传感器本身的输出数据率(ODR)，而不在总线速率。BME280 最大 ODR 约 83 Hz，即使 100 kHz 的标准模式 I2C 也完全够用。速率差异只在高速 ADC、显示屏等场景才有实际影响。

### 9.2 "I2C 只能接 127 个设备"

错误认知：7-bit 地址意味着总线最多挂 127 个设备。

正确理解：实际限制更小。首先，很多传感器的地址只有 2 种可选(如 BME280 是 0x76 或 0x77)，同型号挂 2 片就有地址冲突。其次，I2C 总线电容限制(标准模式 400 pF)会先于地址空间成为瓶颈。一个实际的设计中，同一条 I2C 总线上挂 10-15 个设备已属上限。

### 9.3 "SPI 不需要上拉电阻所以功耗更低"

错误认知：I2C 有上拉电阻耗电，SPI 没有所以更省电。

正确理解：I2C 上拉电阻确实有静态功耗，但(1)可以通过增大阻值(10k-47k)降低到微安级；(2)SPI 在高速运行时的动态功耗可能反而更高(推挽驱动充放电)；(3)真正低功耗场景下，两者都需要进入睡眠模式，此时差异更小。功耗优化应从系统层面考虑，而非单一总线。

## 10. PCB 布线考量

### 10.1 I2C 布线要点

1. **上拉电阻位置**：尽量靠近主设备端，避免长线导致的 RC 延迟
2. **走线长度**：同一总线上所有设备尽量靠近，总走线长度 < 30 cm(400 kHz)
3. **并联电容**：每个器件的输入电容(通常 5-10 pF)会累加，注意不超过 400 pF 总限制
4. **过孔数量**：尽量减少，每个过孔约增加 0.5 pF 寄生电容

### 10.2 SPI 布线要点

1. **等长匹配**：SCLK 和数据线尽量等长，尤其是高速(> 10 MHz)场景
2. **CS 线布线**：每根 CS 线到对应从设备的路径要短且直
3. **串行时钟质量**：SCLK 边沿要干净，避免振铃导致从设备误采样
4. **地平面**：SPI 高速信号需要完整地平面作为回流路径

### 10.3 布线复杂度对比

| 布线维度 | I2C | SPI |
|---------|-----|-----|
| 信号层数 | 1 层(2 根线简单扇出) | 1-2 层(多 CS 需规划) |
| 阻抗控制 | 通常不需要 | 高速时需要(50 欧姆) |
| DRC 检查重点 | 上拉位置、总线电容 | CS 连接、等长、串扰 |
| 适合 PCB 层数 | 2 层板即可 | 2 层板(低速) / 4 层板(高速) |

## 总结

I2C 与 SPI 的选型不是"哪个更好"的问题，而是"哪个更适合当前场景"的问题。核心决策逻辑可以归纳为三步：

1. **看速率需求**：需要 > 1 Mbps 吞吐量的设备(显示、Flash、高速 ADC)用 SPI；低速传感器用 I2C
2. **数引脚预算**：设备多、引脚少用 I2C；设备少、引脚充裕用 SPI
3. **算系统成本**：包括引脚成本、布线成本、功耗成本、调试成本

对初级 IoT 开发者，建议从 I2C 起步(引脚省、接线少、调试简单)，遇到速率瓶颈时再引入 SPI。大多数传感器开发板同时提供两种接口，切换成本不高。

最后记住：I2C 和 SPI 不是互斥的。在同一个项目中混合使用两种总线，是成熟设计的标志，而不是妥协。
