# 机会路由在IoT不稳定链路中的应用
> **难度**: 高级 | **领域**: 路由协议 | **阅读时间**: 约 22 分钟

## 引言

想象你要给远方的朋友寄一封信,但邮递员不太靠谱 -- 有时候他会把信弄丢。传统的做法是: 指定一个邮递员,如果他弄丢了就让他再送一次。机会路由的做法完全不同: 你同时把信交给所有路过的邮递员,谁先到就算谁的,其他人就不用送了。

这就是机会路由(Opportunistic Routing)的核心思想。在IoT无线网络中,链路天生就是不稳定的 -- 一条链路的成功接收率可能只有50%甚至更低。传统路由每次只选一个下一跳,丢了就重传,能量白白浪费。机会路由让发送者广播数据包,谁收到了谁就有机会转发,把"链路不稳定"从缺点变成了优势。

## 1. IoT中链路不稳定的现实

### 1.1 无线链路的本质

无线通信不像有线连接那样可靠。数据包的接收受到多种因素影响:

- 多径衰落: 信号经过不同路径到达接收端,可能互相增强也可能互相抵消
- 阴影效应: 障碍物(墙壁、树木、人体)阻挡信号
- 干扰: 其他无线设备(WiFi、蓝牙、微波炉)占用同一频段
- 环境变化: 温度、湿度、风都会影响信号传播

### 1.2 链路质量的度量

数据包接收率(PRR, Packet Reception Rate)是最直接的链路质量指标。在典型的IoT部署中:

- 近距离(通信半径30%以内): PRR通常在90%以上
- 中等距离(通信半径30-70%): PRR在50-90%之间波动
- 远距离(通信半径70-100%): PRR在10-50%之间,极不稳定

一个关键观察是: 大量IoT链路处于"中间地带"(PRR 30-80%),既不是完全可靠也不是完全不可用。传统路由忽略了这些链路的价值,机会路由则充分利用它们。

### 1.3 传统路由在不稳定链路上的问题

传统路由协议(如RPL)为每个目的地选择一条确定的路径。当路径上某条链路的PRR只有50%时:

- 平均需要2次传输才能成功送达一跳
- 一条5跳路径,期望传输总次数 = 2 x 5 = 10次
- 每次失败的重传都浪费了发送端和接收端的能量
- 如果多条链路都不稳定,重传次数指数级增长

## 2. 机会路由的核心概念

### 2.1 从单播到广播利用

机会路由的核心转变是: 发送者不再只给一个指定的下一跳发包,而是广播出去。由于无线通信的广播特性,周围多个邻居都可能收到这个包。

传统路由把"其他邻居也收到了"看作浪费,机会路由把它看作机会: 只要有一个邻居收到并且比发送者离目的地更近,这次传输就是成功的。

### 2.2 接收概率的聚合效应

这是机会路由最关键的数学优势。假设一个节点有3个邻居,每条链路的PRR都是50%:

- 传统路由(选一个下一跳): 一次传输成功概率 = 50%
- 机会路由(任一邻居收到即可): 一次传输成功概率 = 1 - (1-0.5)^3 = 87.5%

PRR从50%提升到87.5%,这就是链路多样性(Link Diversity)的威力。邻居越多、链路独立性越强,聚合效应越显著。

### 2.3 转发者选择

广播后,可能有多个邻居收到了数据包。需要决定: 谁来转发?

基本原则是: 让"进展最大"的接收者转发。"进展"可以是:

- 地理进展: 离目的地最近的
- 拓扑进展: 到目的地跳数最少的
- ETX进展: 到目的地期望传输次数最少的

只有最佳接收者转发,其他接收者抑制,避免重复。

## 3. ExOR协议详解

### 3.1 ExOR的工作流程

ExOR(Extremely Opportunistic Routing)是机会路由的开创性协议,其工作流程如下:

**第一步: 发送者准备**。发送者维护一个转发者列表(Forwarder List),按照到目的地的优先级排序。列表包含在数据包头部。

**第二步: 广播发送**。发送者广播一批数据包。所有在转发者列表中的邻居都监听。

**第三步: 优先级协调**。收到数据包的邻居根据自己在列表中的优先级决定是否转发:

```
对于每个收到数据包的转发者 F:
  如果F是列表中优先级最高的成功接收者:
    F负责转发这个包
  否则:
    F抑制(不转发)
```

**第四步: 重复**。选中的转发者成为新的发送者,用自己的转发者列表重复这个过程。

### 3.2 ExOR的性能

在有损无线网络上的实验表明,ExOR相比传统最短路径路由:

- 吞吐量提升2-3倍
- 在链路质量差的场景中改善更明显
- 在高质量链路上,与传统路由性能相当(不会更差)

### 3.3 ExOR的局限

ExOR最初是为WiFi Mesh网络设计的,直接应用于IoT传感器网络有困难:

- 协调开销大: 需要在转发者之间传递控制信息
- 批处理延迟: ExOR以批为单位处理,不适合实时应用
- 内存需求: 需要缓存多个数据包
- 不适合占空比工作: IoT节点大部分时间在睡眠

## 4. 转发者选择策略

### 4.1 基于地理位置的选择

最简单的策略是利用地理位置: 任何比发送者更接近目的地的接收者都是合格的转发者,越接近的优先级越高。

优点: 不需要预先知道网络拓扑,与地理路由(如GPSR)自然结合。

### 4.2 基于ETX的选择

ETX(Expected Transmission Count)衡量的是从节点到目的地的期望传输次数。选择ETX最小的接收者作为转发者。

优点: 考虑了链路质量,不仅是距离。
缺点: 需要维护ETX表,有状态开销。

### 4.3 基于能量的选择

在IoT中,能量是关键资源。可以在选择转发者时考虑节点的剩余能量:

- 优先选择能量充足的节点
- 避免让低电量节点承担转发任务
- 延长整个网络的寿命

## 5. 重复抑制机制

### 5.1 为什么重复是严重问题

如果多个接收者都认为自己应该转发,同一个数据包会被多次转发,产生:

- 带宽浪费: 多份相同数据占用信道
- 能量浪费: 多个节点无谓地转发
- 广播风暴: 重复包被再次广播,指数级增长

对于能量受限的IoT设备,重复抑制是生死攸关的问题。

### 5.2 基于优先级定时器的抑制

这是最常用的方法,原理如下:

```
接收者根据自己的优先级设定一个转发定时器:
  优先级越高 -> 定时器越短 -> 越先转发

当定时器到期时:
  如果还没有听到其他节点转发同一个包:
    转发
  如果已经听到更高优先级的节点转发了:
    抑制(取消转发)
```

具体实现:

```python
def on_receive(packet, my_priority, max_priority):
    # 定时器与优先级成正比
    # 优先级1(最高)等待最短,优先级max等待最长
    delay = my_priority * SLOT_TIME
    start_timer(delay)

def on_timer_expire(packet):
    if not heard_forwarding(packet):
        forward(packet)
    # 否则不做任何事(抑制)

def on_overhear_forwarding(packet):
    cancel_timer(packet)
```

### 5.3 基于ACK的抑制

转发者在转发后发送ACK,其他接收者听到ACK后抑制。这种方法更可靠但开销更大,因为ACK本身也是一次传输。

### 5.4 混合方案

实际系统通常结合多种策略: 定时器提供基本的协调,ACK作为确认机制,加上序列号检测防止迟到的重复。

## 6. 期望传输次数分析

### 6.1 传统路由的ETX

对于单条链路,PRR为p时: ETX = 1/p

一条多跳路径的ETX是每一跳ETX的总和。例如,3跳路径,每跳PRR分别为0.8、0.6、0.7:

ETX = 1/0.8 + 1/0.6 + 1/0.7 = 1.25 + 1.67 + 1.43 = 4.35

### 6.2 机会路由的有效ETX

机会路由利用多个潜在接收者,有效ETX显著降低。

假设一跳有3个候选转发者,PRR分别为p1、p2、p3。至少一个收到的概率:

P_success = 1 - (1-p1)(1-p2)(1-p3)

有效ETX = 1 / P_success

以p1=p2=p3=0.5为例:

P_success = 1 - 0.5^3 = 0.875
ETX_opp = 1/0.875 = 1.14

对比单链路ETX = 1/0.5 = 2.0,机会路由节省了43%的传输次数。

### 6.3 链路质量越差,优势越大

| 每条链路PRR | 传统ETX(单链路) | 机会ETX(3条链路) | 节省比例 |
|------------|-----------------|-----------------|---------|
| 90% | 1.11 | 1.00 | 10% |
| 70% | 1.43 | 1.04 | 27% |
| 50% | 2.00 | 1.14 | 43% |
| 30% | 3.33 | 1.46 | 56% |

可以看到,链路质量越差,机会路由的优势越明显。这正是IoT场景的特点: 链路常常处于中等到较差的水平。

## 7. 面向传感器网络的机会路由

### 7.1 ORW协议

ORW(Opportunistic Routing for Wireless sensor networks)是专门为资源受限IoT设计的机会路由协议。

ORW的关键设计:

**EDC指标**: Expected Duty Cycled wakeups(期望占空比唤醒次数)。IoT节点大部分时间在睡眠,通信需要等待邻居醒来。EDC衡量的是从当前节点到汇聚节点需要等待多少次邻居唤醒。

**被动协调**: 不需要显式的转发者列表或控制消息。每个接收者根据自己的EDC值独立决定是否转发,EDC越小(离汇聚节点越近)的越先转发。

**与占空比MAC兼容**: ORW设计时考虑了节点的睡眠-唤醒周期,发送者在邻居醒来时广播,醒来的邻居中EDC最小的负责转发。

### 7.2 ORW的性能

在实际传感器网络测试中,ORW相比标准CTP(Collection Tree Protocol):

- 数据到达率提高了20-50%,特别是在链路质量差的区域
- 延迟降低: 不需要等待特定下一跳唤醒
- 能量效率: 与CTP相当或更好,因为减少了重传

## 8. 能量收益分析

### 8.1 单跳能量模型

假设发送一个数据包消耗能量E_tx,接收消耗E_rx。传统路由(单链路PRR=p):

- 期望发送能量: E_tx / p
- 接收端总能量: E_rx(成功时) + 接收失败的包也消耗能量

机会路由(3个候选,每个PRR=p):

- 一次广播消耗E_tx(无论多少邻居收到)
- 成功概率: 1-(1-p)^3,远高于p
- 期望发送能量: E_tx / [1-(1-p)^3]
- 额外开销: 多个接收者都消耗E_rx,但只有一个转发

### 8.2 具体数字

以CC2420无线芯片为例(许多IoT传感器使用):

- E_tx = 0.06 mJ/包
- E_rx = 0.05 mJ/包

PRR = 50%时,一跳传输:

**传统路由**:
- 发送: 0.06/0.5 = 0.12 mJ
- 总计: 约0.12 mJ(一对一)

**机会路由**(3个候选接收者):
- 发送: 0.06/0.875 = 0.069 mJ
- 3个接收者: 3 x 0.05 = 0.15 mJ(但只有一个转发)
- 总计: 约0.22 mJ

看起来机会路由在单跳上并没有省能量? 实际上,机会路由的优势在多跳路径上体现: 因为每跳成功率更高,需要的跳数更少,总能量更低。

### 8.3 多跳路径的整体收益

考虑一条端到端路径,传统路由5跳,每跳PRR=50%:

- 传统: 每跳ETX=2,总传输10次,总能量 = 10 x E_tx = 0.6 mJ
- 机会路由: 每跳ETX=1.14,可能只需4跳(更好的接收者跳过中间节点),总传输4.56次,总能量 = 4.56 x E_tx = 0.27 mJ

端到端能量节省55%。

## 9. IoT中机会路由的挑战

### 9.1 协调开销

机会路由的最大挑战是: 如何在多个接收者之间协调,确保只有一个转发。

在WiFi Mesh中(ExOR),节点一直开着,可以持续监听。但在IoT中:

- 节点大部分时间在睡眠
- 唤醒要消耗能量
- 监听邻居是否转发了需要保持接收器开着

这个协调开销可能抵消机会路由带来的收益。

### 9.2 内存限制

存储转发者列表、缓存未确认的数据包,都需要内存。8KB RAM的传感器节点可能无法支撑复杂的机会路由协议。

### 9.3 实现复杂度

机会路由的协议栈比传统路由复杂:

- 需要修改MAC层(支持广播+监听)
- 需要定时器管理(转发优先级)
- 需要重复检测(序列号管理)
- 需要状态维护(哪些包已被确认转发)

### 9.4 何时不该用机会路由

- 链路质量好(PRR > 90%): 传统路由已经够好,机会路由的开销不值得
- 节点极度受限(<4KB RAM): 协议开销太大
- 实时性要求高: 定时器等待引入延迟
- 网络密度低(邻居少于3个): 链路多样性不足,机会路由退化为传统路由

## 10. 地理机会路由

### 10.1 概念

将地理路由和机会路由结合: 广播数据包,任何比发送者更接近目的地的接收者都可以转发。不需要预先建立转发者列表,只需要知道目的地的位置。

### 10.2 协调机制

每个接收者根据自己与目的地的距离设置转发定时器:

```
timer_delay = (1 - progress / max_progress) * MAX_DELAY
```

其中progress是接收者相对于发送者的前进距离。离目的地越近,progress越大,延迟越短,越先转发。

这种方式的优点: 完全分布式,不需要任何预先协调,每个节点只需要知道自己的位置和目的地的位置。

### 10.3 与GPSR的比较

| 特性 | GPSR | 地理机会路由 |
|------|------|------------|
| 转发方式 | 单播给最近邻居 | 广播,多个候选 |
| 链路利用 | 只用最好的一条 | 利用所有可能的链路 |
| 失败处理 | 重传或周边模式 | 自然由其他接收者接替 |
| 空洞处理 | 周边模式 | 可结合周边模式 |
| 开销 | 低(单播) | 中(广播+协调) |
| 适用链路质量 | 好到中等 | 中等到差 |

## 11. 实际案例: 桥梁结构健康监测

### 11.1 场景描述

一座钢结构桥梁上部署了100个振动传感器,用于监测桥梁的结构健康状况。传感器嵌入桥梁结构中,一旦部署就无法更换电池,必须依靠初始电池运行数年。

### 11.2 通信环境

钢结构桥梁是极其恶劣的无线通信环境:

- 金属结构导致严重的多径衰落
- 信号反射和散射造成快速变化的信道
- 平均链路PRR只有60%
- 链路质量随时间波动大(车辆通过时更差)

### 11.3 传统方案的问题

使用RPL路由协议:

- 平均每跳需要约1.67次传输(1/0.6)
- 从最远的传感器到网关需要8跳
- 端到端平均传输次数: 13.3次
- 网络寿命: 约6个月
- 6个月后需要维护人员爬到桥上更换电池,成本高昂

### 11.4 机会路由方案

采用地理机会路由,每个节点平均有3-4个可用邻居:

**每跳成功率计算**: 3个邻居,每个PRR=60%
- P_success = 1 - (1-0.6)^3 = 1 - 0.064 = 93.6%
- 有效ETX = 1/0.936 = 1.07

**端到端改善**: 
- 平均路径长度: 7跳(机会路由可以"跳过"中间节点)
- 端到端平均传输次数: 7 x 1.07 = 7.49次
- 对比传统: 13.3次,减少44%

### 11.5 性能对比

| 指标 | 传统RPL | 地理机会路由 | 改善 |
|------|---------|------------|------|
| 每跳期望传输 | 1.67 | 1.07 | 36% |
| 端到端传输次数 | 13.3 | 7.49 | 44% |
| 网络寿命 | 6个月 | 18个月 | 3倍 |
| 数据到达率 | 85% | 97% | +12% |

网络寿命从6个月延长到18个月,意味着维护周期从半年一次延长到一年半一次,显著降低了运维成本。对于嵌入桥梁结构中无法轻易更换电池的传感器来说,这是决定系统可行性的关键指标。

## 总结

机会路由代表了一种思维转变: 不再把无线链路的不稳定性视为需要克服的缺陷,而是把多条不稳定链路的聚合效应视为可利用的资源。在IoT场景中,无线链路质量普遍不高,机会路由通过广播和多接收者协调,将单链路50%的成功率提升到87.5%甚至更高。虽然协调开销、内存限制和实现复杂度是需要面对的挑战,但在链路质量差且电池不可更换的部署场景中,机会路由带来的能量节省和寿命延长是传统路由无法企及的。
