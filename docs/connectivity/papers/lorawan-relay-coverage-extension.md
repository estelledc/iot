# LoRaWAN中继节点覆盖扩展方案
> **难度**：🟡 中级 | **领域**：LoRaWAN网络扩展 | **阅读时间**：约 20 分钟

## 引言

想象一个山谷里的村庄, 手机信号基站建在山顶, 山顶信号满格, 但山谷深处完全没有信号。解决方案之一是在半山腰装一个信号放大器 -- 它接收山顶基站的信号, 再转发给山谷里的手机。这个"半山腰的放大器"就是中继(Relay)的概念。

在LoRaWAN网络中, 有些设备因为距离网关太远、被建筑物或地形遮挡, 无法直接与网关通信。LoRaWAN中继节点就是解决这个问题的标准方案 -- 它是一种特殊的终端设备, 能够接收附近设备的信号并转发给网关, 无需部署完整的网关基础设施。

## 1. 覆盖盲区问题

### 1.1 盲区产生的原因

即使精心规划了网关部署, 仍然会存在覆盖盲区:

| 盲区类型 | 典型场景 | 原因 |
|---------|---------|------|
| 距离型 | 偏远农田、森林深处 | 超出网关最大覆盖范围 |
| 遮挡型 | 地下室、隧道、山谷 | 建筑/地形阻挡信号传播 |
| 穿透型 | 深层建筑内部 | 多层墙壁衰减过大 |
| 临时型 | 施工现场、活动场地 | 无法预先部署基础设施 |

### 1.2 传统解决方案的局限

增加网关是最直接的解决方案, 但并非总是可行:

- **回程链路缺失**: 偏远地区可能没有以太网或蜂窝网络覆盖
- **供电困难**: 没有市电接入点
- **成本过高**: 单个网关500-2000美元, 加上安装和维护
- **审批障碍**: 某些位置不允许安装大型设备
- **临时需求**: 短期使用不值得部署永久基础设施

### 1.3 中继的核心价值

中继的本质是用一个低成本、低功耗的设备, 在不需要IP回程的情况下扩展网络覆盖:

```
传统方案(需要回程):
终端设备 ---(LoRa)---> 网关 ---(IP网络)---> 网络服务器

中继方案(无需额外回程):
终端设备 ---(LoRa)---> 中继 ---(LoRa)---> 网关 ---(IP)---> 网络服务器
```

## 2. LoRaWAN中继规范(TS011)

### 2.1 标准化背景

2022年, LoRa Alliance发布了TS011规范, 正式将中继功能标准化:

- **文档**: LoRaWAN Relay Specification TS011-1.0.0
- **目标**: 定义标准的中继协议, 确保不同厂商设备互通
- **原则**: 对网络服务器尽可能透明, 最小化对现有协议的改动
- **兼容性**: 中继功能向后兼容, 不影响非中继设备

### 2.2 中继设备的身份

在TS011规范中, 中继设备本身是一个特殊的LoRaWAN终端设备(Class A):

- 拥有自己的DevEUI和入网凭证
- 正常入网(OTAA/ABP)获得DevAddr
- 除了转发功能外, 也可以发送自己的应用数据
- 由网络服务器通过MAC命令控制其中继行为

### 2.3 核心协议机制

中继通信使用专用的WOR(Wake-on-Radio)通道:

```
时序图:

终端设备                    中继设备                    网关
    |                          |                        |
    |---WOR前导码(唤醒)------->|                        |
    |                          |(被唤醒, 打开接收窗口)   |
    |---上行数据包------------>|                        |
    |                          |---转发上行数据包------->|
    |                          |                        |
    |                          |<--下行数据包(含转发)----|
    |<--转发下行数据包---------|                        |
    |                          |                        |
```

## 3. 中继架构详解

### 3.1 网络拓扑

```
                    +----------+
                    | 网络服务器 |
                    +----+-----+
                         |
                    +----+-----+
                    |   网关    |
                    +----+-----+
                         |
              +----------+----------+
              |                     |
         +----+-----+         +----+-----+
         |   中继    |         | 直连终端  |
         +----+-----+         +----------+
              |
    +---------+---------+
    |         |         |
+---+---+ +--+----+ +--+----+
| 终端1  | | 终端2  | | 终端3  |
+-------+ +-------+ +-------+
```

### 3.2 数据包转发流程

上行转发(终端到网络服务器):

1. 终端设备在WOR通道发送唤醒前导码
2. 中继设备被唤醒, 打开接收窗口
3. 终端发送上行数据包(加密的, 中继无法解密)
4. 中继在标准LoRaWAN通道上转发给网关
5. 中继添加转发元数据(原始RSSI、SNR等)
6. 网络服务器识别并处理转发包

下行转发(网络服务器到终端):

1. 网络服务器将下行数据包发送给中继
2. 中继存储该下行包
3. 当目标终端下次发送上行时, 中继转发下行包

### 3.3 端到端安全性

中继只做透明转发, 不破坏端到端安全: 终端与网络服务器之间的加密(AppSKey)不受影响, 中继无法读取应用数据内容, 网络服务器验证MIC仍然有效。

## 4. Wake-on-Radio(WOR)机制

### 4.1 WOR的必要性

中继设备通常是电池供电, 不能一直接收(持续接收约10-15mA)。WOR机制让中继大部分时间休眠:

```
中继设备的工作状态:

时间 ------>
[休眠][监听][休眠][休眠][监听][休眠][休眠][监听]...
 ~1uA  短暂  ~1uA              短暂
      检查WOR                 检查WOR

如果检测到WOR前导码 -> 保持接收, 等待数据包
如果未检测到 -> 立即返回休眠
```

### 4.2 前导码与唤醒

```
终端发送的WOR帧结构:

|<--- WOR前导码(长) --->|<--- WOR帧 --->|<--- 实际数据包 --->|
|  持续时间 >= 中继     |  目标中继信息  |  标准LoRaWAN帧    |
|  的监听间隔           |  和调度信息    |                   |

前导码长度确保中继在任何时刻醒来都能检测到
```

### 4.3 功耗分析

| 工作模式 | 电流 | 时间占比 | 平均功耗贡献 |
|---------|------|---------|------------|
| 深度休眠 | 1-2uA | 约99% | 极低 |
| WOR监听 | 5-10mA | 约0.5-1% | 低 |
| 接收数据 | 10-15mA | 按需 | 取决于流量 |
| 转发数据 | 20-40mA | 按需 | 取决于流量 |

电池寿命估算: 2000mAh电池, 低流量场景下可达1-3年。

## 5. 中继与网关的对比

| 特性 | 中继设备 | 网关 |
|------|---------|------|
| 硬件成本 | 50-100美元 | 500-2000美元 |
| IP回程 | 不需要 | 必须 |
| 供电 | 电池可行 | 需持续供电 |
| 容量 | 10-50设备 | 1000-5000设备 |
| 延迟 | 增加(双跳) | 最小 |
| 安装复杂度 | 低 | 高(需布线) |
| 远程管理 | 有限 | 完整 |
| 固件升级 | 困难 | 容易(通过IP) |

选择中继: 盲区设备少于50个、无IP回程、快速临时覆盖、预算有限。

选择网关: 设备超过100个、需可靠双向通信、有回程和电力、长期部署。

## 6. 中继容量限制

### 6.1 占空比约束

```
假设1%占空比限制(欧盟):
- 每小时可用发射时间: 36秒
- SF7(56ms/包): 最多转发约640包/小时
- SF10(370ms/包): 最多转发约97包/小时
- SF12(1319ms/包): 最多转发约27包/小时
```

中继转发使信道占用翻倍(原始上行+转发上行各占一次空口)。

### 6.2 实际容量估算

| 设备上报频率 | 单中继可服务设备数(SF7) | 单中继可服务设备数(SF12) |
|------------|----------------------|----------------------|
| 每15分钟 | 约40-50个 | 约8-10个 |
| 每小时 | 约150-200个 | 约25-30个 |
| 每天1次 | 理论不限 | 约500-600个 |

如果中继使用电池供电, 转发越多电池消耗越快, 需平衡设备数量和电池寿命。

## 7. 中继应用场景

### 7.1 地下停车场

```
+--楼顶--+   网关(楼顶安装)
|        |
+--地面--+
+---B1---+   中继(B1层楼梯间, 市电供电)
+---B2---+              |
+---B3---+   停车位传感器(20个)
```

中继在B1层向上连通网关, 向下覆盖B2/B3。

### 7.2 山谷农业监测

```
       网关(*)  <--- 山顶, 有4G回程
      / 山坡 \
     /         \
    /    中继(*) \  <--- 半山腰, 太阳能供电
   /              \
  /   传感器群     \
 /___山谷果园______\
```

中继太阳能供电, 50个传感器每小时上报, 中继位置使其同时"看到"山顶网关和谷底传感器。

### 7.3 临时活动与深层建筑

- 临时活动: 沿线路部署中继链, 活动后回收
- 深层建筑: 外墙附近放置中继覆盖内部50-100米

## 8. 部署实施要点

### 8.1 中继位置优化

1. 必须同时"看到"网关和终端设备(到网关RSSI > -120dBm)
2. 选择"高处"或"转角"位置(走廊拐角、楼层间、山坡)
3. 避免信号边缘位置(到网关链路应有10dB以上余量)

### 8.2 链路验证与设备选型

部署前现场测试: 在候选位置确认能连通网关, 在盲区确认能连通候选位置。

设备要求: 支持TS011固件、适当发射功率、合适供电方式(电池/太阳能/市电)、室外需IP65以上防护。

## 9. 替代方案对比

### 9.1 LoRa Mesh网络

```
TS011中继: 设备 -> 中继 -> 网关  (仅一跳)
Mesh方案:  设备 -> 节点A -> 节点B -> 网关  (多跳)
```

Mesh代表: Wirepas Mesh(商用)、Meshtastic(开源)。优点是覆盖扩展更强, 缺点是延迟增加、容量下降、非标准。

### 9.2 决策矩阵

| 考虑因素 | 中继 | Mesh | 增加网关 |
|---------|------|------|---------|
| 盲区设备数 | < 50 | 50-500 | > 100 |
| 可用回程 | 不需要 | 不需要 | 必须 |
| 标准兼容 | TS011 | 非标准 | 标准 |
| 部署复杂度 | 最低 | 中等 | 最高 |
| 总成本 | 最低 | 中等 | 最高 |

## 10. 实际案例: 智慧停车场

某商业综合体地下3层停车场, 每层200个车位, 楼顶有网关但B2/B3无信号。

```
楼顶: 网关(已有)
B1层: 中继1(楼梯间, 市电) -- 覆盖B1+部分B2
B2层: 中继2(电梯厅, 市电) -- 覆盖B2+部分B3

注意: TS011只支持单跳, 每个中继必须能直连网关。
B3层如中继无法直达楼顶, 需通过漏缆延伸天线或增加网关。
中继方案更适合1-2层地下场景。
```

## 总结

LoRaWAN中继节点(TS011规范)为覆盖扩展提供了标准化低成本方案。核心优势: 无需IP回程, 终端级硬件成本, 电池供电可行。核心限制: 容量有限(不超过50设备), 仅单跳转发。

对于小规模覆盖补盲(盲区设备少、无回程、无电力), 中继是性价比最高的首选。设备多或需高容量时应选择增加网关。
