# Wireless M-Bus无线抄表在智能水表中的应用
> **难度**：🟡 中级 | **领域**：无线计量 | **阅读时间**：约 20 分钟

## 引言

想象你是一个供水公司的抄表员。过去你每个月要开着小货车,穿街走巷,挨家挨户去抄水表。有些水表在院子的角落里,有些在地下井盖下面,冬天冻手夏天晒背。一天最多抄 200 户,整个城区 10 万户水表至少需要一支庞大的抄表队伍忙碌整个月。

Wireless M-Bus(无线仪表总线)就像给每个水表装了一个"小广播站"。水表定时把自己的读数广播出去,你开车从街上经过,车上的接收器就自动把附近所有水表的数据收了。更进一步,在路灯杆上装几个固定接收器,水表的数据就能 24 小时不间断地实时传回控制中心。

## 1. Wireless M-Bus标准概述

### 1.1 有线M-Bus的无线延伸

Wireless M-Bus(简称 wM-Bus)是有线 M-Bus 标准 EN 13757 的无线扩展部分,定义在 EN 13757-4 中。它继承了有线 M-Bus 的应用层数据格式(DIF/VIF 编码),但用 Sub-GHz 无线电替代了双芯电缆。

这意味着有线和无线 M-Bus 在应用层是兼容的: 一个水表无论通过有线还是无线方式传输,它发出的数据格式是一样的。接收端的软件可以统一处理两种方式的数据。

### 1.2 为什么选择Sub-GHz

Wireless M-Bus 主要工作在 868MHz(欧洲)和 169MHz(长距离)频段,而不是常见的 2.4GHz。选择 Sub-GHz 频段有几个重要原因:

- 穿透力更强: 低频无线电波更容易穿透墙壁、地板和地下管井
- 传输距离更远: 同等功率下,868MHz 的覆盖范围是 2.4GHz 的 2-3 倍
- 干扰更少: 2.4GHz 频段挤满了 WiFi 和蓝牙设备,Sub-GHz 频段相对清净
- 功耗更低: Sub-GHz 射频前端的功耗通常低于 2.4GHz

### 1.3 在智能计量中的地位

Wireless M-Bus 是欧洲智能计量领域的主导标准。在德国、法国、意大利、北欧等地区,大量的水表、气表、热表都采用 Wireless M-Bus 进行无线抄表。据估计,欧洲已有超过数千万只 Wireless M-Bus 计量表投入使用。

## 2. 频段与通信模式

### 2.1 两个主要频段

Wireless M-Bus 定义了两个主要工作频段:

868MHz 频段是最常用的。这是欧洲的 ISM(工业/科学/医疗)免许可频段,发射功率通常限制在 25mW。在城市环境中,典型通信距离为 100-500 米。

169MHz 频段专为长距离应用设计。虽然数据速率较低,但信号的穿透力和传输距离远优于 868MHz。在开阔环境中可以达到数公里的通信距离。这个频段在某些国家需要许可。

### 2.2 通信模式详解

Wireless M-Bus 定义了多种通信模式,每种针对不同的应用场景:

S 模式(Stationary,固定式): 计量表每隔几秒发送一次数据包,主要用于步行抄表(walk-by)或车载抄表(drive-by)场景。抄表员经过时,手持或车载接收器捕获数据。S 模式使用 868MHz 频段,数据速率较低,传输距离中等。

T 模式(Frequent Transmit,频繁发送): 计量表更频繁地发送数据,适用于固定网络(fixed network)场景。T 模式的特点是发送间隔短(通常几秒到十几秒一次),便于固定接收器持续接收。

C 模式(Compact,紧凑): 是 S 和 T 模式的改进版本,采用更高效的编码方式,在相同功率下获得更好的通信性能。C 模式是较新的标准,在新部署中越来越常用。

N 模式(Narrowband,窄带): 工作在 169MHz 频段,采用窄带调制。N 模式的最大优势是超长距离和超强穿透力,非常适合城市环境中水表埋在地下的场景。它是固定网络方案的首选。

### 2.3 模式对比

| 特性 | S模式 | T模式 | C模式 | N模式 |
|------|-------|-------|-------|-------|
| 频段 | 868MHz | 868MHz | 868MHz | 169MHz |
| 典型用途 | walk-by | 固定网络 | 固定网络 | 远距固定网络 |
| 数据速率 | 32.768 kbps | 100 kbps | 100 kbps | 2.4/4.8 kbps |
| 穿透力 | 中 | 中 | 中 | 强 |
| 典型距离 | 100-300m | 100-500m | 100-500m | 500m-数km |

## 3. 通信模式: 单向与双向

### 3.1 单向通信

大多数 Wireless M-Bus 部署采用单向通信模式。计量表周期性地广播自己的数据,接收器被动接收。这种模式的最大优势是极低的功耗:

- 计量表的无线模块大部分时间处于睡眠状态
- 只在发送时短暂唤醒(几毫秒到几十毫秒)
- 不需要监听接收,省去了接收电路的功耗
- 电池寿命可以达到 15 年以上

### 3.2 双向通信

在某些高级应用中,需要双向通信:

- 远程配置: 修改计量表的发送间隔、数据格式等参数
- 按需读取: 在非标准时间点读取即时数据
- 阀门控制: 远程关闭水阀或气阀(如欠费停供)
- 固件更新: 远程更新计量表的软件

双向通信要求计量表定期打开接收窗口监听来自基站的下行指令,这会增加功耗,缩短电池寿命。因此通常采用"主要单向、偶尔双向"的策略: 计量表在每次发送数据后短暂打开接收窗口(如1-2秒),检查是否有下行指令。

## 4. 协议栈结构

### 4.1 物理层

Wireless M-Bus 的物理层采用 FSK(频移键控)或 GFSK(高斯频移键控)调制方式。不同模式的具体参数有所不同:

```
868MHz 模式 (S/T/C):
- 调制: GFSK
- 频偏: 50kHz (典型)
- 发射功率: 10-25mW
- 接收灵敏度: -105 到 -110 dBm

169MHz N模式:
- 调制: GFSK 或 4-GFSK
- 频偏: 2.5kHz (窄带)
- 发射功率: 10-500mW (视频段监管要求)
- 接收灵敏度: -120 dBm 以下
```

### 4.2 数据链路层

无线报文(Telegram)的结构如下:

```
[前导码] [同步字] [长度] [C字段] [M字段] [A字段] [CI] [数据] [CRC]
  |        |       |      |       |       |      |     |      |
  |        |       |    控制字段  制造商ID  地址  控制  有效载荷 校验
  |        |     字节数
  |     同步标识
  载波检测
```

M 字段包含制造商 ID(每个制造商有唯一标识),A 字段包含设备的序列号和版本信息。这两个字段合在一起可以唯一标识全球范围内的每一个计量表。

### 4.3 应用层

应用层沿用有线 M-Bus 的 DIF/VIF 数据格式(EN 13757-3)。这意味着无论计量数据通过有线还是无线传输,上层应用软件处理的数据结构是相同的。这种兼容性极大地简化了混合部署的系统集成。

## 5. 智能水表应用

### 5.1 超声波水表与Wireless M-Bus

现代智能水表通常采用超声波流量测量技术,配合 Wireless M-Bus 无线通信模块。超声波水表没有活动机械部件,使用寿命更长,计量精度更高。

一个典型的 Wireless M-Bus 智能水表的内部架构:

```
[超声波换能器] --> [信号处理MCU] --> [流量累计存储]
                                          |
                                   [Wireless M-Bus 射频模块]
                                          |
                                   [天线] )))  --> 无线信号
                                          |
                                   [锂电池 3.6V]
```

### 5.2 数据存储与发送

智能水表在本地存储丰富的数据,包括:

- 当前累计水量(主读数)
- 过去数天/数周/数月的历史读数(消费曲线)
- 最大/最小流量记录
- 报警标志(如回流检测、泄漏疑似、电池低电量)
- 表计运行小时数

这些数据通过 Wireless M-Bus 定期发送。发送间隔取决于工作模式: T 模式下通常每 16 秒发送一次,S 模式下可能每隔数小时发送一次。

### 5.3 电池寿命

电池寿命是智能水表最关键的设计指标之一。典型参数:

| 参数 | 数值 |
|------|------|
| 电池类型 | 锂亚硫酰氯(Li-SOCl2) 3.6V |
| 电池容量 | 2-4 Ah |
| 发射功率 | 10-25 mW |
| 单次发送时长 | 5-50 ms |
| 发送间隔 | 16秒(T模式)到数小时(S模式) |
| 睡眠电流 | 小于 1 uA |
| 设计寿命 | 10-15 年 |

电池寿命计算的关键在于占空比。以 T 模式为例: 每 16 秒发送一次,每次持续约 10ms,占空比仅为 0.0625%。绝大部分时间射频模块处于深度睡眠状态,电流消耗不到 1 微安。

## 6. 数据采集网络架构

### 6.1 三种采集方式

Wireless M-Bus 数据采集有三种方式,从简单到复杂:

步行抄表(Walk-by): 抄表员手持接收器步行经过计量表安装位置,自动接收范围内所有水表的数据。优点是设备投资最小,缺点是仍需人工到现场且数据频率低(通常月度)。

车载抄表(Drive-by): 接收器安装在车辆上,抄表员开车沿街道行驶即可采集沿途所有水表数据。效率比步行高很多,一辆车一天可以覆盖数千只表。

固定网络(Fixed Network): 在建筑物或路灯杆上安装永久性接收器,形成覆盖全城的接收网络。水表数据 24 小时实时采集,通过回传链路(蜂窝/光纤)传回数据中心。初始投资最高,但运营成本最低,且能实现实时监控。

### 6.2 固定网络架构

固定网络是 IoT 时代 Wireless M-Bus 的主流部署方式:

```
[水表] )))  --> [接收器/集中器] -- 蜂窝/光纤 --> [前端系统] --> [云平台]
[水表] )))  -->    (路灯杆上)                   (数据中心)    (分析/账单)
[水表] )))  -->

覆盖范围: 一个接收器覆盖 100-500m 半径
城市部署: 每 300-500m 一个接收器
单个接收器可管理: 数百到数千只表
```

### 6.3 169MHz N模式的优势

在城市固定网络中,169MHz N 模式越来越受欢迎:

- 穿透力: 信号可以穿透地下水表井盖、厚墙和建筑楼层
- 覆盖范围: 一个接收器可以覆盖更大区域,减少所需接收器数量
- 抗干扰: 169MHz 频段的干扰源远少于 868MHz
- 电池寿命: 虽然单次发送功耗略高,但发送频率可以更低

## 7. 安全机制

### 7.1 加密标准

Wireless M-Bus 的安全机制定义在 EN 13757-7 中,采用 AES-128 加密算法。每个计量表拥有唯一的加密密钥,在出厂时或通过安全通道配置。

### 7.2 安全威胁与防护

Wireless M-Bus 面临的安全威胁和对应防护措施:

窃听(Eavesdropping): 攻击者可以用接收器捕获无线报文,获取用户的用水习惯(如判断是否在家)。AES-128 加密使得攻击者即使截获报文也无法解读内容。

表计伪造(Meter Spoofing): 攻击者伪造水表发送虚假读数,企图少缴水费。每个表的唯一密钥和消息认证码(MAC)可以验证数据来源的真实性。

重放攻击(Replay Attack): 攻击者录制有效报文并重新播放。报文中的帧计数器(Frame Counter)可以检测和拒绝重复报文。

### 7.3 密钥管理

密钥管理是 Wireless M-Bus 安全体系中最具挑战性的部分。在大规模部署(数万到数十万只表)中,需要安全地生成、存储、分发和更新每只表的密钥:

- 出厂预置: 制造商在出厂时为每只表烧录唯一密钥,密钥通过安全渠道交付给水务公司
- 现场配置: 安装时通过近场通信(NFC)或光学接口配置密钥
- 远程更新: 通过双向通信通道安全地更新密钥(需要表支持双向模式)

## 8. 与其他智能计量技术的对比

### 8.1 主流技术对比

| 特性 | Wireless M-Bus | NB-IoT | LoRaWAN | PLC(电力线) |
|------|---------------|--------|---------|-------------|
| 频段 | 868/169MHz | 运营商授权频段 | Sub-GHz ISM | 电力线 |
| 网络 | 私有 | 公共蜂窝 | 私有/公共 | 电力网 |
| 标准 | EN 13757-4 | 3GPP | LoRa Alliance | 多种 |
| 覆盖 | 100m-数km | 数km | 数km | 沿电力线 |
| 电池寿命 | 10-15年 | 5-10年 | 5-10年 | 不需要(有源) |
| 适用区域 | 欧洲主导 | 全球 | 全球 | 主要用于电表 |
| 数据费用 | 无(私有网络) | 运营商资费 | 取决于部署 | 无 |
| 成熟度 | 20年以上 | 较新 | 较新 | 成熟 |

### 8.2 选型考量

Wireless M-Bus 的核心优势是成熟度和生态: 在欧洲拥有最庞大的安装基数,标准稳定,供应商众多。对于欧洲地区的水务、燃气、供暖公司来说,Wireless M-Bus 仍然是最稳妥的选择。

NB-IoT 的优势是利用已有的蜂窝网络基础设施,不需要自建接收器网络,适合分散的大面积部署。但需要支付运营商数据费用,且覆盖依赖运营商网络的完善程度。

LoRaWAN 在灵活性方面有优势,可以自建网络也可以使用公共网络,但在计量领域的生态不如 Wireless M-Bus 成熟。

## 9. 实际案例: 城市供水公司的智能水表系统

### 9.1 项目背景

某欧洲城市供水公司管理 10 万只水表。传统的人工抄表每月需要大量人力,且无法及时发现漏水。公司决定分阶段部署 Wireless M-Bus 智能水表系统。

### 9.2 技术方案

系统采用 169MHz N 模式和固定网络架构:

| 组件 | 数量 | 说明 |
|------|------|------|
| 智能水表 | 100000 只 | 超声波水表,169MHz N模式 |
| 固定接收器 | 200 个 | 安装在路灯杆上 |
| 回传链路 | 200 条 | 4G 蜂窝回传 |
| 前端系统 | 1 套 | 数据中心部署 |
| 云分析平台 | 1 套 | 公有云部署 |

选择 169MHz N 模式的原因: 城市中大量水表安装在地下或半地下管井中,169MHz 的穿透力可以确保信号可靠传输; 每个接收器的覆盖半径可达 500 米以上,200 个接收器即可覆盖全城。

### 9.3 数据采集策略

每只水表每小时发送一次数据报文。报文内容包括: 当前累计读数、过去 24 小时的逐时消耗量、最大瞬时流量、报警标志。

200 个接收器分散在全城,每个接收器负责接收附近 500 只左右水表的数据。接收器通过 4G 网络将数据实时回传到前端系统。

### 9.4 云平台分析能力

云平台对采集到的数据进行多维度分析:

泄漏检测: 系统在凌晨 1 点到 5 点分析每个计量点的流量。如果在这个时段持续检测到流量(排除合理用水),系统会生成疑似泄漏告警并推送到运维团队的工单系统。

异常消耗识别: 如果某个水表的日消耗量突然变为平时的 3 倍以上,系统标记为异常并通知用户核实(可能是管道破裂或表计故障)。

表计健康监测: 监控每只表的通信成功率和电池电压。通信成功率持续低于 90% 的表需要检查安装环境; 电池电压低于阈值的表需要提前规划更换。

### 9.5 实施成果

系统全面投运后取得了显著成效:

- 抄表成本降低了 90% 以上,原来的抄表队伍转型为维护和客服团队
- 漏水检测时间从"下个月抄表时才发现"缩短到"1小时内告警"
- 非收益水(Non-Revenue Water)损失率从 18% 降至约 15%
- 用户可以通过手机 App 查看自己的用水数据,投诉减少了约 40%
- 自动计费消除了人工抄表的误差,水费纠纷几乎为零

## 总结

Wireless M-Bus 是欧洲智能计量领域的核心技术。它通过 Sub-GHz 无线通信实现了对水表、气表、热表的远程自动抄表,从步行抄表到车载抄表再到固定网络,逐步实现了从人工到完全自动化的演进。

169MHz N 模式的出现让 Wireless M-Bus 在城市固定网络场景中的优势更加突出: 超强的地下穿透力、更远的覆盖距离、更少的基础设施投入。配合 AES-128 加密和完善的安全机制,Wireless M-Bus 可以满足公用事业计量对数据安全和隐私保护的严格要求。

对于学习者而言,Wireless M-Bus 是理解"电池供电无线传感器如何在极低功耗下工作 10 年以上"的典型案例。它的占空比设计(99.9% 以上的时间在睡眠)、Sub-GHz 频段选择(穿透力与距离的权衡)、以及安全机制(AES-128加密与密钥管理)都是 IoT 无线通信设计中的核心话题。
