# 卫星IoT LEO低轨星座连接方案
> **难度**：🔴 高级 | **领域**：卫星IoT通信 | **阅读时间**：约 22 分钟

## 引言

想象你在一片茫茫大海上航行, 手机完全没有信号。岸上的基站再密集, 也覆盖不到这里。地球表面大约只有 20% 被蜂窝网络覆盖, 剩下的 80%(海洋、沙漠、森林、极地)是通信的盲区。但如果天上有一群卫星在不停地绕地球飞, 你的设备随时能抬头找到一颗, 把数据发上去呢? 这就是卫星 IoT 的核心思路: 用太空中的基础设施, 为地面网络无法触及的地方提供连接。

本文聚焦低轨(LEO)卫星星座在 IoT 场景中的应用, 从轨道选择、主流星座、链路预算到终端设计, 逐步展开。

## 1 为什么需要卫星IoT

### 1.1 地面网络的覆盖极限

蜂窝网络(4G/5G)和 LPWAN(LoRa/NB-IoT)的基站都建在地面, 覆盖半径有限。一旦离开人口密集区, 信号就会消失:

- 远洋船舶: 离岸 50km 后失去蜂窝信号
- 沙漠油田: 方圆数百公里无基站
- 极地科考: 基础设施极度匮乏
- 热带雨林: 地形复杂, 基站部署成本极高

### 1.2 卫星覆盖的天然优势

卫星从太空俯瞰地球, 一颗卫星的波束覆盖面积可达数百万平方公里。如果把足够多的卫星组成星座, 就能实现全球无死角覆盖, 无论设备在哪里, 头顶总有卫星飞过。

### 1.3 IoT场景的特殊需求

IoT 设备通常只需要发送很小的数据包(几十到几百字节), 对延迟要求不高(分钟级可接受), 但对成本和功耗极度敏感。这与传统卫星通信(语音/宽带)的需求截然不同, 催生了专门的卫星 IoT 方案。

## 2 LEO vs GEO: 轨道选择

### 2.1 GEO地球同步轨道

GEO 卫星位于赤道上方约 36000km 高度, 相对地面静止不动:

| 参数 | GEO 典型值 |
|------|-----------|
| 轨道高度 | 36000km |
| 单程延迟 | 约 270ms(往返 600ms 以上) |
| 覆盖范围 | 单颗覆盖地球 1/3 |
| 终端成本 | 高(需大天线/高功率) |
| 适合场景 | 广播、宽带回传 |

GEO 的高延迟和高终端成本使其不太适合低成本 IoT 应用。

### 2.2 LEO低轨道

LEO 卫星运行在 500-2000km 高度, 绕地球飞行一圈约 90-120 分钟:

| 参数 | LEO 典型值 |
|------|-----------|
| 轨道高度 | 500-2000km |
| 单程延迟 | 约 3-13ms(往返 20-50ms) |
| 单颗覆盖 | 小(需星座实现连续覆盖) |
| 终端成本 | 低(小天线/低功率即可) |
| 适合场景 | IoT、窄带通信 |

### 2.3 IoT选择LEO的理由

```
LEO 优势:
- 距离近 -> 路径损耗小 -> 终端可用小天线加低功率
- 延迟低 -> 虽然 IoT 不要求实时, 但低延迟降低协议复杂度
- 终端成本低 -> 符合 IoT 大规模部署的经济要求
- 发射成本降低 -> SpaceX 等降低了 LEO 卫星部署门槛

LEO 代价:
- 需要数十到数百颗卫星才能实现连续覆盖
- 卫星高速移动, 需要处理多普勒频移和切换
- 卫星寿命较短(5-10年), 需要持续补充
```

## 3 主流LEO IoT星座

### 3.1 Iridium(铱星)

铱星是最早的全球 LEO 通信星座, 1998 年开始运营:

- 66 颗工作卫星加备份星, 轨道高度约 780km
- 六个轨道面, 每面 11 颗, 实现全球连续覆盖
- 支持语音和数据, IoT 服务品牌为 Iridium SBD(Short Burst Data)
- 单条消息最大 340 字节, 延迟秒级
- 终端模块(如 Iridium 9603)约 100-200 美元

### 3.2 Orbcomm

专注 IoT/M2M 消息的先驱:

- 第一代 35 颗(1997), 第二代 18 颗(2014)
- 轨道高度约 750km, VHF 频段(137-150MHz)
- 面向资产追踪、集装箱监控、车队管理

### 3.3 Swarm/SpaceBee

Swarm Technologies(已被 SpaceX 收购)代表了超低成本卫星 IoT 的方向:

- 约 150 颗 0.25U 超小卫星, 轨道高度 450-550km
- VHF 137MHz 上下行
- 单条消息最大 192 字节
- 存储转发模式, 延迟从分钟到数小时不等
- 设备模块成本约 5 美元, 月服务费约 5 美元
- 2024 年已并入 SpaceX, 服务逐步整合

### 3.4 Lacuna Space

将 LoRaWAN 搬上太空的创新方案:

- LEO 卫星搭载 LoRaWAN 网关
- 标准 LoRa 终端设备经过固件更新后即可与卫星通信
- 卫星充当天空中的 LoRaWAN 网关
- 实现地面 LoRa 网络与卫星覆盖的无缝混合

### 3.5 Kineis

法国 IoT 卫星星座:

- 基于 ARGOS 系统的下一代, 25 颗纳卫星
- 轨道高度约 650km
- 面向环境监测、资产追踪、海洋观测
- 消息大小: 数十字节, 延迟: 分钟级

### 3.6 星座对比

| 星座 | 卫星数量 | 轨道高度 | 频段 | 消息大小 | 延迟 | 设备成本 |
|------|---------|---------|------|---------|------|---------|
| Iridium | 66 | 780km | L-band | 340B | 秒级 | 100-200美元 |
| Orbcomm | 18 | 750km | VHF | 数百B | 分钟级 | 50-100美元 |
| Swarm | 150 | 450-550km | VHF | 192B | 分钟-小时 | 5美元 |
| Lacuna | 数颗 | LEO | LoRa ISM | LoRa帧 | 分钟-小时 | 10-20美元 |
| Kineis | 25 | 650km | UHF | 数十B | 分钟级 | 20-50美元 |

## 4 链路预算挑战

### 4.1 自由空间路径损耗

卫星链路的核心挑战是距离带来的信号衰减。自由空间路径损耗(FSPL)公式:

```
FSPL(dB) = 20*log10(d) + 20*log10(f) - 147.55

其中:
  d = 距离(米)
  f = 频率(Hz)

示例计算(LEO 600km, 频率 900MHz):
  FSPL = 20*log10(600000) + 20*log10(900000000) - 147.55
       = 115.56 + 179.08 - 147.55
       = 147.09 dB

对比地面 LoRa 10km 链路:
  FSPL = 20*log10(10000) + 20*log10(900000000) - 147.55
       = 80 + 179.08 - 147.55
       = 111.53 dB

差距: 约 35dB, 相当于信号强度差 3000 多倍
```

### 4.2 IoT终端的约束

IoT 设备受到严格限制:

- 天线: 小型全向天线, 增益 0-3dBi(不能用大锅)
- 发射功率: 0.5-2W(27-33dBm), 受法规和功耗限制
- 电池: 需要运行数年, 不能持续大功率发射

### 4.3 弥补链路预算的手段

| 策略 | 增益来源 |
|------|---------|
| 卫星天线高增益 | 30-35dBi(相控阵或抛物面) |
| 扩频增益 | 10-20dB(LoRa 类扩频调制) |
| 低速率/长传输时间 | 处理增益, 信噪比要求降低 |
| 重传机制 | 多次发送提高成功概率 |
| 窄带信号 | 降低噪声带宽 |

## 5 终端设计

### 5.1 天线设计

卫星 IoT 终端通常使用简单的全向天线:

- 贴片天线(patch): 平面结构, 适合固定安装
- 鞭状天线(whip): 适合移动设备和资产追踪器
- 增益通常 0-3dBi, 指向天顶方向优化

### 5.2 功率管理

```python
# 卫星 IoT 终端功率管理伪代码
class SatelliteIoTDevice:
    def __init__(self):
        self.tx_power_dbm = 30        # 1W 发射功率
        self.sleep_current_ua = 5     # 休眠电流 5uA
        self.tx_current_ma = 500      # 发射电流 500mA
        self.pass_interval_min = 90   # 卫星过顶间隔约 90 分钟

    def run_cycle(self):
        # 1. 深度休眠, 等待卫星过顶窗口
        self.deep_sleep(self.pass_interval_min)

        # 2. 唤醒, 准备数据
        sensor_data = self.read_sensors()
        message = self.pack_message(sensor_data)  # 192 字节以内

        # 3. 在卫星过顶窗口内发射
        self.transmit(message)

        # 4. 可选: 等待下行确认
        ack = self.listen_for_ack(timeout_sec=10)

        # 5. 回到深度休眠
        self.deep_sleep(self.pass_interval_min)
```

### 5.3 GPS定时

许多卫星 IoT 方案要求终端具备 GPS 能力:

- 确定自身位置, 用于计算卫星过顶时间
- 预补偿多普勒频移(知道自身位置加卫星轨道参数)
- 为数据添加位置标签(资产追踪的核心功能)
- GPS 模块功耗: 冷启动约 30-40mA, 热启动约 20mA

## 6 存储转发机制

### 6.1 工作原理

LEO 卫星不是一直在头顶, 对于非密集星座, 两次过顶之间可能间隔数十分钟到数小时:

```
时间线:

设备        [采集数据]---[存储]----------[发射]---[休眠]----
卫星过顶    .............|................|^可见窗口|........
地面站      ..................................[接收转发]......

流程:
1. 设备采集传感器数据, 存入本地缓冲区
2. 设备等待卫星进入可见范围(仰角大于 10-15 度)
3. 在卫星过顶窗口(约 5-10 分钟)内发射消息
4. 卫星接收并存储消息
5. 卫星飞到地面站上方时, 下传消息
6. 地面站将消息送入互联网/云平台
```

### 6.2 延迟特性

存储转发的延迟取决于星座密度:

- 密集星座(Iridium 66 颗): 几乎实时, 秒级延迟
- 中等星座(Kineis 25 颗): 分钟到十分钟级
- 稀疏星座(初期 Swarm): 可能数小时

对于大多数 IoT 场景(环境监测、资产追踪), 分钟到小时级延迟完全可以接受。

## 7 应用场景

### 7.1 海洋运输

远洋船舶和集装箱是卫星 IoT 的典型用户:

- 船舶追踪: AIS 只在沿海有效, 远洋需要卫星
- 集装箱监控: 温度、湿度、开关门状态
- 冷链物流: 食品药品的全程温度记录
- 每条船可能携带数百个 IoT 传感器

### 7.2 远程农业

偏远农场和牧场远离蜂窝覆盖:

- 土壤湿度和温度传感器
- 灌溉系统远程控制
- 牲畜追踪(GPS 项圈)
- 气象站数据上传

### 7.3 野生动物追踪

```
典型野生动物追踪器配置:
- GPS 加卫星 IoT 模块
- 重量: 20-50g(鸟类)到 500g(大型哺乳动物)
- 电池: 锂电池, 寿命 1-3 年
- 数据: 每天 4-12 个位置点
- 消息大小: 20-50 字节/条
- 场景: 候鸟迁徙路线, 濒危物种保护
```

### 7.4 基础设施监控

偏远地区的管道、电力线、水利设施:

- 石油/天然气管道泄漏检测
- 输电线路状态监测
- 水坝水位和结构健康监测
- 每个监测点每天上传几条数据即可

### 7.5 灾难应急

地面网络被破坏时, 卫星成为最后的通信手段:

- 地震/洪水后基站瘫痪, 应急定位信标
- 灾区传感器数据回传, 救援队伍位置追踪

## 8 成本分析

### 8.1 设备成本

```
卫星 IoT 模块价格范围:

传统方案(Iridium SBD):
  模块: 100-200 美元, 适合高可靠性商业应用

新一代方案(Swarm/Kineis):
  模块: 5-50 美元, 适合大规模低成本部署

LoRa-from-Space(Lacuna):
  模块: 标准 LoRa 模块加固件更新
  额外成本: 几乎为零(已有 LoRa 设备)
```

### 8.2 服务费用

| 方案 | 月费/设备 | 每条消息成本 | 数据量限制 |
|------|----------|-------------|-----------|
| Iridium SBD | 15-50美元 | 约0.05-0.10美元 | 按消息计费 |
| Swarm | 5美元 | 包含在月费中 | 750条/月 |
| Kineis | 1-5美元 | 包含在月费中 | 按套餐 |
| Lacuna | 按量 | 低 | 按数据包 |

### 8.3 与传统卫星对比

传统卫星通信(如海事 VSAT)月费可达数百到数千美元, 终端价格数千美元。新一代卫星 IoT 将成本降低了 1-2 个数量级, 使得百万级设备部署成为经济可行的方案。

## 9 混合地面-卫星连接

### 9.1 设计理念

最优方案往往是地面和卫星的混合:

```
设备决策逻辑:

if 蜂窝信号可用:
    使用 LTE/NB-IoT (低成本, 低延迟)
elif LoRa 网关可达:
    使用 LoRaWAN (免许可频段, 低功耗)
else:
    回退到卫星 (全球覆盖, 较高成本)
```

### 9.2 实现方式

- 多模模块: 集成蜂窝加卫星的芯片组
- 协议适配: 统一上层协议, 底层自动选择承载
- 云平台透明: 后端不关心数据经由哪条路径到达

### 9.3 典型架构

```
[IoT 设备] --LoRa/LTE--> [地面网关/基站] --> [云平台]
    |                                            ^
    +------卫星链路------> [LEO 卫星] --> [地面站] --+
```

设备首先尝试地面连接, 失败后自动切换到卫星。对云平台来说, 数据来源是透明的。

## 10 未来趋势

### 10.1 直连设备(Direct-to-Device)

最令人兴奋的趋势是让标准手机和 IoT 芯片直接连接卫星:

- 3GPP NTN(非地面网络)标准化: Release 17 开始
- Starlink Direct-to-Cell: 让普通手机连接 Starlink 卫星
- AST SpaceMobile: 大型天线卫星提供手机直连

### 10.2 星地融合

未来的通信网络将不再区分地面和卫星:

- 统一的网络架构和协议栈
- 无缝漫游: 设备在地面和卫星网络间自动切换
- 统一计费: 用户不感知底层承载变化

### 10.3 成本持续下降

- SpaceX 等降低发射成本(每公斤入轨成本从数万美元降至数千美元)
- 卫星制造标准化(立方星/纳卫星批量生产)
- 芯片集成度提高(卫星 IoT 功能集成到标准 SoC)

## 总结

卫星 IoT 正在从昂贵的小众技术变成普惠的基础设施。LEO 星座凭借低延迟、低成本终端和日益降低的部署门槛, 成为连接地球上剩余 80% 盲区的关键力量。Swarm 的 5 美元模块、Lacuna 的 LoRa-from-Space、以及 3GPP NTN 标准化, 都在推动卫星 IoT 走向大规模商用。

对于 IoT 系统设计者来说, 关键是理解卫星链路的特点(存储转发延迟、有限带宽、链路预算约束), 并设计出能够智能切换地面和卫星连接的混合方案。未来, 天上有没有信号将不再是一个问题。
