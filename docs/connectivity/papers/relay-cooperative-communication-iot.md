# 中继协作通信在IoT覆盖扩展中的应用
> **难度**: 高级 | **领域**: 协作通信 | **阅读时间**: 约 22 分钟

## 引言

想象一个村庄里的传话游戏。村长站在村子一端要把消息传到另一端, 但村子太长喊破嗓子对面也听不见。解决办法很简单: 在中间站几个人, 一个接一个传过去。每个人只需喊给最近的下一个人听, 不需要超远距离的吼叫。

中继通信的原理与此完全一致。当IoT设备的发射功率太低、无法直接送达网关时, 让中间节点帮忙转发。一次长距离传输变成两次短距离传输, 所需的总能量远低于直传。更进一步的"协作通信"则像多人同时喊同一句话, 对面把几个人的声音合在一起就能听清, 这就是协作分集: 多个简单设备合作, 达到单个复杂设备的效果。

## 1. 为什么需要中继通信

### 1.1 IoT设备的先天限制

```
典型IoT设备:
  发射功率: 0-14dBm (1-25mW)
  天线增益: 0-2dBi (小型PCB天线)
  电池容量: 200-2000mAh

对比蜂窝手机:
  发射功率: 23dBm (200mW)
  天线增益: 0dBi
  电池容量: 3000-5000mAh
```

IoT设备的有效通信距离远小于手机, 而电池寿命要求却更严格(通常需要数年不换电池)。

### 1.2 两跳短传vs一跳长传的能量对比

无线信号功率以距离的n次方衰减(自由空间n=2, 室内n=3-4):

```
一跳直传(距离d): P_direct 正比于 d^n
两跳中继(每跳d/2): 2 * (d/2)^n = d^n / 2^(n-1)

能量节省: 2^(n-1)倍
  n=2 (自由空间): 节省2倍
  n=3 (城市环境): 节省4倍
  n=4 (室内环境): 节省8倍
```

室内环境中使用一个中继节点可将总传输能耗降至直传的八分之一。

## 2. 中继转发类型

### 2.1 放大转发(AF)

中继收到信号后直接放大并转发, 不做解码:

```
时隙1: S发送 -> R收到 y_r = h_sr * x + n_r
时隙2: R放大转发 -> D收到 y_d = h_rd * G * y_r + n_d

优点: 实现极其简单, 硬件成本最低
缺点: 噪声被一起放大, 多跳后信噪比持续恶化
适合: 短距离、少跳数场景
```

### 2.2 解码转发(DF)

中继完整解码收到的信号, 重新编码后发送:

```
时隙1: S发送 -> R解码得到 x_hat
时隙2: R重新编码发送 -> D收到干净信号

优点: 噪声不跨跳累积, 适合多跳
缺点: 需要完整编解码能力, 增加复杂度和延迟
风险: 解码错误会被传播
```

### 2.3 压缩转发(CF)

中继不完整解码, 将接收信号压缩后转发。目的节点结合直接收到的信号和中继压缩信号联合解码。介于AF和DF之间, 适合中继信道质量不足以可靠解码的场景。

### 2.4 三种类型对比

| 特征 | AF | DF | CF |
|------|-----|-----|-----|
| 中继复杂度 | 最低 | 最高 | 中等 |
| 噪声处理 | 放大 | 消除 | 压缩 |
| 处理延迟 | 最小 | 最大 | 中等 |
| 多跳适用性 | 差 | 好 | 中等 |
| IoT适用场景 | 简单中继 | 可靠多跳 | 协作解码 |

## 3. 协作分集

### 3.1 虚拟MIMO

传统MIMO需要多根天线, 但IoT设备通常只有一根。协作通信让多个单天线设备合作形成"虚拟MIMO":

```
传统MIMO: 一个设备多根天线 -> 接收机
虚拟MIMO: 多个设备各一根天线 -> 网关

效果: 网关收到来自不同空间位置的多个信号副本
      任一设备的信道衰落不会导致通信失败
```

### 3.2 协作分集增益

源节点发送信号, 中继也向目的节点转发副本。目的节点收到经历独立衰落的两个副本:

```
时隙1: S -> D (直接路径h_sd), S -> R (信道h_sr)
时隙2: R -> D (信道h_rd)

D收到两个独立副本:
  分集阶数 = 2
  错误概率与 1/SNR^2 成正比 (无分集时为 1/SNR)
```

分集阶数为2意味着误码率下降速度是无分集时的平方, 在衰落信道中是巨大的可靠性提升。

## 4. 协作通信协议

### 4.1 固定中继

中继始终转发, 不管直接路径质量。简单但浪费: 直接路径好时中继转发多余, 50%时隙被占用。

### 4.2 选择性解码转发

中继只在成功解码时才转发, 避免传播错误。解码失败时源节点重传或中继保持静默。

### 4.3 增量中继

最节省资源的协议。目的节点能正确解码时发ACK, 不需中继(节省一个时隙); 不能解码时发NACK, 中继才转发。平均频谱效率高于固定中继。

### 4.4 机会中继

多个候选中继时选最佳的一个。选择标准: 选瓶颈链路(源到中继和中继到目的中较弱者)质量最好的中继。分布式实现用计时器法: 信道越好计时器越短, 最先超时的中继发送。

## 5. 中继选择算法

### 5.1 最佳中继选择

```
对每个候选中继Ri:
  gamma_1i = |h_s_Ri|^2 * Ps / N0  (源到中继SNR)
  gamma_2i = |h_Ri_d|^2 * Pr / N0  (中继到目的SNR)
  gamma_i = min(gamma_1i, gamma_2i) (瓶颈SNR)

选择: R_best = argmax{gamma_i}
```

### 5.2 能量感知选择

IoT网络还需考虑中继剩余电量:

```
综合指标: score_i = alpha * gamma_i + (1-alpha) * E_remain_i / E_max

alpha=0: 只考虑能量(选电量多的)
alpha=1: 只考虑信道(选信号好的)
动态调整: 能量充足时alpha偏大, 紧张时alpha偏小
```

## 6. LoRaWAN中的中继

### 6.1 覆盖挑战

LoRaWAN原生星形拓扑, 所有设备直连网关。虽然LoRa链路预算高(可达157dB), 但地下室(混凝土楼板衰减15-25dB/层)、室内深处、山区和远郊仍有覆盖盲区。

### 6.2 LoRa Alliance中继规范

2022年发布的官方中继规范:

```
中继设备:
  - 有自己的DevEUI和会话密钥
  - 在专用信道/SF上监听终端
  - 缓存上行帧后在自己的窗口转发给网关

终端设备:
  - 检测到无法直达网关时切换到中继模式
  - 在中继专用信道发送
  - 透明转发, 不需知道中继存在

网络服务器:
  - 识别中继转发的帧并管理路由
```

### 6.3 中继功耗问题

```
中继需持续监听: 约5-10mA(LoRa接收模式)
100%监听: 5mA * 24h = 120mAh/天
2000mAh电池: 仅约17天

解决方案:
  1. 市电供电(最常见)
  2. 定义监听时间窗口减少占空比
  3. 终端先发唤醒信号
```

## 7. 网状中继与星形中继

### 7.1 对比

星形中继(单跳到中继再到网关)拓扑简单、延迟可控, LoRaWAN Relay采用此模式。网状中继(多跳链式转发)覆盖范围大幅扩展但拓扑复杂, Thread、ZigBee、WirelessHART采用此模式。

### 7.2 不同协议的策略

| 协议 | 中继类型 | 典型跳数 | 适用场景 |
|------|---------|---------|---------|
| LoRaWAN Relay | 星形 | 1跳 | LPWAN覆盖扩展 |
| Thread | 网状 | 5-10跳 | 智能家居 |
| WirelessHART | 网状 | 3-5跳 | 工业监控 |
| ZigBee | 网状 | 最多30跳 | 楼宇自动化 |
| BLE Mesh | 泛洪 | 无限制 | 照明控制 |

## 8. 能量考量

### 8.1 中继能量模型

```
E_relay = E_rx + E_process + E_tx
  E_rx: 接收能耗
  E_process: 处理能耗(DF > CF > AF)
  E_tx: 发射能耗
  E_idle: 空闲监听能耗(往往是主要来源)
```

### 8.2 中继位置优化

```
源S在(0,0), 目的D在(d,0), 中继R在(x,0)
总能耗: E = x^n + (d-x)^n
最优位置: x = d/2 (中点)

直觉: 中继放在源和目的正中间, 两跳等距, 总能耗最低
实际: 信道不对称时最优位置偏向较差的一侧
```

### 8.3 专用vs机会中继

专用中继是专门部署的市电设备, 持续可用但需额外成本。机会中继利用空闲的普通设备临时转发, 不需额外部署但可用性不确定。两者的权衡是可靠性与成本之间的取舍。

## 9. 传感器网络中的协作通信

### 9.1 协作MIMO

多个单天线传感器协作实现分布式波束成形:

```
N个节点同时发送同一数据, 调整相位使信号在基站相干叠加
阵列增益: 10*log10(N) dB
  4节点: 6dB增益
  10节点: 10dB增益
```

### 9.2 实现挑战

多节点需要精确的时间和频率同步才能相干叠加, 每个节点需要到目的的信道信息来计算相位调整。这些需求增加了复杂度和信令开销, 目前仍主要处于研究阶段。

## 10. 多跳性能考量

### 10.1 延迟与可靠性

每一跳增加传输和处理延迟。更关键的是端到端可靠性是各跳可靠性的乘积:

```
端到端可靠性 = R_1 * R_2 * ... * R_K

每跳99%: 1跳99%, 3跳97.03%, 5跳95.10%, 10跳90.44%
每跳95%: 1跳95%, 3跳85.74%, 5跳77.38%, 10跳59.87%
```

多跳中继必须确保每跳高可靠性, 否则端到端质量快速下降。

### 10.2 跳数限制原则

一般原则是尽量减少跳数, 必要时增加网关密度。LoRa Relay限1跳保持简单性, Thread允许5-10跳覆盖家庭面积, WirelessHART限3-5跳满足工业实时性。

## 11. 实际案例: 地下停车场监测

### 11.1 场景描述

三层地下停车场部署CO浓度传感器, LoRaWAN网关在地面建筑屋顶。

### 11.2 覆盖分析

```
混凝土楼板衰减: 约20dB/层
LoRa链路预算(SF12, 125kHz): 最大路径损耗151dB

屋顶到B1: 约100dB (OK, 余量51dB)
屋顶到B2: 约120dB (OK, 余量31dB)
屋顶到B3: 约140dB (仅11dB余量)

B3考虑车辆遮挡和多径衰落后大部分区域无法可靠通信
```

### 11.3 中继部署方案

```
B1层中继: 楼梯间, 直连屋顶网关(1层楼板, 余量充足)
B2层中继: 楼梯间, 转发给B1中继
路径: B3传感器 -> B2中继 -> B1中继 -> 屋顶网关

每跳穿越1层楼板: 约100dB, 余量51dB(非常充裕)
中继供电: 接入停车场照明电路(市电)
```

### 11.4 成本对比

```
方案A(每层网关): 3台网关约6000元 + 3条回程链路
方案B(中继): 2台中继约800元 + 1台网关约2000元 = 2800元
节省: 超过50%, 且无需额外回程链路
```

中继方案以不到一半的成本实现了全覆盖, 每跳链路余量充裕保证可靠性。

## 总结

中继协作通信通过中间节点转发解决了IoT设备通信距离不足的问题。两跳短传比一跳长传在室内环境可节省8倍能量。三种基本转发类型(AF/DF/CF)各有适用场景, 协作分集通过虚拟MIMO提升可靠性。LoRaWAN等LPWAN协议已标准化中继功能, 在地下空间和覆盖边缘场景中, 中继以远低于增加网关的成本实现了有效的覆盖扩展。
