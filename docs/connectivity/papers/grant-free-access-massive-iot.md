# 免授权接入在大规模IoT连接中的方案
> **难度**: 高级 | **领域**: 大规模接入 | **阅读时间**: 约 22 分钟

## 引言

想象一个大型超市的结账流程。传统方式是排队结账——每位顾客先排到收银台(请求),收银员说"下一位"(授权),然后才能结账(传输数据)。1000位顾客10个收银台,排队等待时间很长。另一种方式: 每位顾客直接把购物清单投进一个大箱子,后台系统自动识别和处理所有清单。这就是免授权接入(Grant-free Access)的核心思想。

大规模IoT中,数万乃至数十万设备注册在同一基站下,但任意时刻只有少数需要发送数据(稀疏活跃)。每个设备都经历"请求-授权"流程,信令开销会吞噬大量系统资源。免授权接入允许设备需要时直接发送,由基站负责检测并解码。

## 1. 基于授权的接入: 问题所在

### 1.1 传统4步调度流程

```
步骤1: 设备 --[调度请求(SR)]--> 基站
  设备告知基站: "我有数据要发"
步骤2: 基站 --[调度授权(Grant)]--> 设备
  基站分配资源: "在这个时隙/频率发送"
步骤3: 设备 --[数据传输]--> 基站
  设备在指定资源上发送数据
步骤4: 基站 --[ACK/NACK]--> 设备
  基站确认: "收到/请重传"

时间开销: 约10-15ms
信令开销: 3条控制消息(SR + Grant + ACK)
```

### 1.2 IoT效率灾难

```
IoT数据包: 50字节(温度/湿度/状态)
调度信令(SR+Grant头): 约80字节
信令占比: 80/(80+50) = 62%

类比: 写50字短信, 先填80字"申请写信表"
超过60%的资源用于"申请"而非"通信"
```

### 1.3 大规模突发场景

10000个传感器每小时汇报,平均每秒约3个活跃。但整点"闹钟效应"可能100+设备在1秒内同时请求调度。PRACH竞争导致碰撞退避重试,越拥塞越多重试越拥塞,触发雪崩效应。100设备产生200条信令,而实际数据只有100个小包。

## 2. 免授权接入概念

### 2.1 核心思想

设备无需请求许可,直接发送导频加数据:

```
传统: 设备->[请求]->基站->[授权]->设备->[数据]->基站  (4步)
免授权: 设备->[导频+数据]->基站                       (1步)
```

### 2.2 三个关键要素

**预配置资源**: 基站预先划出一块时频资源,所有设备都知道可以在这里发送,无需每次请求。

**竞争式接入**: 多个设备可能同时使用同一资源,碰撞不可避免,需要接收端技术解决。

**活跃设备检测**: 基站通过导频/特征码识别哪些设备在发送,联合完成检测加解码。

### 2.3 与传统RACH的关键区别

传统RACH仍需4步(前导到RAR到连接请求到竞争解决),数据在第3步才发送,RACH只是"获取调度"的前置步骤。免授权是1步完成全部过程,数据在第1步就直接发送。

## 3. 为什么免授权适合大规模IoT

### 3.1 稀疏活跃模式

```
注册设备: N = 10000 - 1000000
任意时刻活跃: K = 10 - 100
活跃比例: K/N < 1% (极度稀疏)

含义:
- 为每个设备预留专用资源: 极其浪费
- 调度每个活跃设备的开销远超数据本身
- 免授权: 只有活跃设备使用资源, 其他不占用
```

### 3.2 效率与延迟对比

```
传输效率:
基于授权: 50B数据 + 80B开销 = 38%效率
免授权:   50B数据 + 10B导频 = 83%效率

接入延迟:
基于授权: 10-15ms (SR到数据传输完成)
免授权:   <1ms    (直接发送)
```

### 3.3 电池寿命影响

基于授权每次传输涉及5次收发操作: 监听下行控制信道、发送SR、等待Grant(保持收发器开启)、发送数据、等待ACK。免授权仅1-2次: 发送导频加数据,可选等待ACK。射频活跃时间减少约60%,电池寿命提升约30%。

## 4. 5G NR中的配置授权

### 4.1 Configured Grant

3GPP 5G NR引入配置授权: RRC一次性配置周期性上行资源,设备在配置资源上直接发送:

```
时间: [CG][  ][CG][  ][CG][  ][CG]
       ^有数据   ^有数据      ^无数据(空)
       直接发    直接发       不发
```

**Type 1**(纯RRC配置): 无需DCI激活,设备立即可用,最简单,适合固定周期IoT。

**Type 2**(RRC加DCI激活): 更灵活但略复杂,适合半固定流量模式。

### 4.2 关键参数示例

```python
def configured_grant_params():
    """5G NR配置授权的典型参数"""
    return {
        'periodicity': '10ms',       # 授权周期
        'time_offset': '2ms',        # 时间偏移
        'mcs': 2,                    # 调制编码方案
        'repetitions': 4,            # 重复次数(覆盖增强)
        'num_rbs': 4                 # 分配的资源块数
    }
```

### 4.3 延迟改善

动态调度: SR处理加调度决策加Grant传输加处理时间,总计约10-15ms。配置授权: 直接在预配置资源发送,延迟小于1ms。延迟降低10-15倍,满足URLLC和周期性IoT需求。

## 5. 2步RACH

3GPP Release 16将传统4步RACH简化为2步:

```
传统4步: Msg1(前导) -> Msg2(RAR) -> Msg3(数据) -> Msg4(解决)
2步RACH: MsgA(前导+数据合并) -> MsgB(解决+ACK)

延迟: 从15-20ms降到5-8ms (减少约60%)
信令: 从4条降到2条
```

MsgA将前导和数据包打包一起发送。前导用于定时估计和设备区分,有效载荷携带身份信息和应用数据。特别适合IoT: 小数据和前导一起发,少两次收发操作,对事件触发上报和周期小数据场景理想。

设备不知道精确定时,需使用保护间隔;功率控制不精确,需用重复提升可靠性。

## 6. 碰撞解决技术

免授权中多设备可能在同一资源同时发送,需要接收端技术解决碰撞。

### 6.1 NOMA/SIC碰撞解决

利用功率差异: 多设备信号叠加后,基站先解码最强信号,减去,再解码次强,重复。即使三个设备"碰撞"在同一资源上,只要功率差异足够,SIC可以全部成功解码。

### 6.2 扩频码碰撞解决

每设备预分配唯一扩频码,接收端对每个码做相关检测。出现相关峰表示该设备活跃,从相关输出解码数据。即使功率相同也可分离,但码字数量有限,超载需要高级检测器。

### 6.3 联合活跃检测与数据解码

最先进的方法: 从N个注册设备中找出K个活跃设备(K远小于N),利用活跃模式的稀疏性,通过压缩感知或贝叶斯推断一次性完成检测加解码。

## 7. 压缩感知用于活跃检测

### 7.1 稀疏性利用

```
注册设备N=10000, 活跃K=50
活跃向量: [0,0,1,0,...,0,1,0,...,0] 只有K个1
稀疏度: 0.5%

传统方法: 需要N=10000个导频维度
压缩感知: 需要约K*log(N/K) = 50*log(200) = ~380个导频维度
从380维观测恢复10000维稀疏信号
```

### 7.2 常用算法

| 算法 | 复杂度 | 性能 | 适用场景 |
|------|--------|------|---------|
| OMP(正交匹配追踪) | 低 | 中 | 实时性要求高 |
| LASSO(L1正则化) | 中 | 好 | 一般场景 |
| AMP(近似消息传递) | 中 | 好 | 大规模系统 |
| SBL(稀疏贝叶斯学习) | 高 | 最好 | 精度要求高 |

### 7.3 联合处理框架

实际系统需要同时完成三件事: 谁在发送(活跃检测)、信道如何(信道估计)、发了什么(数据解码)。导频阶段检测活跃设备并估计信道,数据阶段利用信道估计解码数据,可迭代优化相互提升精度。

## 8. 导频序列设计

### 8.1 导频的三重功能

免授权中导频同时承担: 设备身份标识(唯一导频识别设备)、信道估计(已知导频加接收信号估计信道)、定时同步(确定设备时间对齐)。设计目标是在有限长度内同时满足三个功能。

### 8.2 正交 vs 非正交导频

正交导频长度L需大于等于设备数N,N=10000时导频太长不可行。非正交导频L远小于N,存在导频间干扰,但利用活跃稀疏性(K远小于N)仍可检测。实际使用ZC序列(良好相关性,LTE/NR采用)或高斯随机序列(理论最优)。

### 8.3 导频长度权衡

```
短导频(L=50): 开销小但检测概率低, 适合K<20
长导频(L=200): 检测可靠但开销大, 适合K=50-100
典型选择: L = 4K 到 8K
```

## 9. LoRaWAN的免授权机制

LoRaWAN从设计之初就是免授权接入:

```
LoRaWAN Class A上行:
1. 设备有数据
2. 直接选随机信道+数据率
3. 发送数据包
4. 不需要任何预先请求

接入方式: 纯ALOHA (随机时刻, 随机信道, 直接发送)
```

优势: 零接入延迟、设备端极简单、不需下行控制信道。局限: 无物理层碰撞解决,纯ALOHA理论最大吞吐18.4%,高负载碰撞严重。改进: 网关侧SIC、定时ALOHA减少碰撞、多网关分集(多个网关可能只部分碰撞)、捕获效应(功率差大于6dB时强包可解码)。

## 10. Sigfox的免授权方案

Sigfox是免授权的极简实现:

```
设备选随机频率(100Hz微信道), 发3次重复(不同频率)
可用微信道: 约1920个(192kHz / 100Hz)
单次碰撞概率p, 3次全碰概率: p^3
若p=10%, 3次全碰仅0.1%

无上行调度 / 无下行控制 / 无信道估计
无功率控制 / 无定时同步

设备: 生成数据 -> 随机选频 -> 发送 -> 完成
基站: 所有微信道持续监听 -> 检测解码 -> 去重 -> 上传
```

这是"无协调"接入的极端形式,以频率分集换取传输可靠性。

## 11. 挑战与混合方案

### 11.1 碰撞与可靠性

核心矛盾: 免授权等于无协调,碰撞不可避免。解决方向: 物理层SIC/NOMA分离碰撞信号,MAC层退避重传,系统层多网关分集,算法层先进的活跃检测加解码。一般IoT 99%可达,工业IoT 99.99%需要混合方案,URLLC 99.999%需要免授权加保底机制。

### 11.2 检测误差

两类错误: 漏检(活跃设备未检测到,数据丢失,需重传)和虚警(非活跃被误判活跃,浪费解码资源)。降低漏检阈值则虚警增加,反之亦然。需根据应用需求平衡。

### 11.3 可扩展性

设备数N增加: 导频长度需增加,检测复杂度增加,碰撞概率增加。解决: 分时/分频将设备分组,层次化先粗检测再精检测。

### 11.4 混合接入策略

实际系统通常采用混合方案:

```
第一次尝试: 免授权(乐观, 最低延迟)
  设备直接发送
  如果成功 => 完成

如果失败: 回退到基于授权(稳妥, 保证送达)
  发送调度请求, 等待调度

整体效果: 低延迟 + 高可靠性
```

## 12. 实际应用案例: 智慧农业

### 12.1 场景描述

```
区域: 大面积农业产区
传感器: 50000个土壤传感器
汇报频率: 每小时一次
数据包: 土壤湿度/温度/pH (40字节)
网络: NB-IoT
```

### 12.2 免授权方案设计

```
NB-IoT配置授权(Configured Grant):
每10ms一个CG时隙, 每时隙6设备
每秒容量: 600设备

正常(14/s): 利用率2.3%, 全部一步成功, 延迟<10ms
突发(500/s): 利用率83%, 大部分成功, 少量碰撞下时隙重传
```

### 12.3 基站活跃检测

基站用压缩感知检测活跃设备: 每设备预分配ZC导频,对所有可能导频做相关检测,超过阈值判定活跃。漏检率小于1%,虚警率小于0.1%,平均解码成功率大于99%。

### 12.4 效益分析

```
信令减少: 调度信令归零, 总信令减少约80%
延迟改善: 30-50%
射频活跃时间减少: 约60%
电池寿命提升: 约30%
50000传感器稳定接入, 电池可用10年以上
传感器端零额外复杂度
```

## 总结

免授权接入是解决大规模IoT连接效率的核心技术。通过消除"请求-授权"流程,实现零调度延迟和最小信令开销,契合IoT"大量设备、小数据包、稀疏活跃"的特点。

5G NR的配置授权和2步RACH是标准化方案,LoRaWAN和Sigfox则从设计之初就采用免授权架构。压缩感知使基站能高效检测稀疏活跃设备。混合接入(免授权为主、授权兜底)兼顾低延迟与可靠性。所有复杂度集中在基站侧,IoT终端保持最简设计。
