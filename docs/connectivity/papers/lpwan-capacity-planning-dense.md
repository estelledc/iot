# LPWAN密集部署容量规划与干扰分析
> **难度**: 高级 | **领域**: 网络规划 | **阅读时间**: 约 22 分钟

## 引言

想象一个城市广场上有10万人同时在喊话传递消息。如果大家都随机喊(像LoRaWAN的ALOHA机制),声音会互相淹没,大部分消息都收不到;如果有一个指挥官按顺序安排谁什么时候喊(像NB-IoT的调度机制),效率会高很多但需要额外的协调成本。对于智慧城市这样需要同时连接数十万设备的场景,理解每种LPWAN技术的容量极限至关重要--否则系统在部署到一半时就"堵死了"。

## 1. LPWAN容量的基本概念

### 1.1 什么是网络容量

网络容量指在满足服务质量(QoS)前提下系统能同时服务的最大设备数量。对LPWAN来说,QoS通常定义为: 数据包到达率(PDR)>95%,延迟允许数秒到数分钟,连续N次传输至少1次成功。

### 1.2 影响容量的关键因素

```
容量 = f(频谱资源, 时间资源, 接入机制, 干扰管理)

频谱: 可用带宽、信道数量、是否共享
时间: 每包空中传输时间(ToA)、占空比限制
接入机制: 随机接入(ALOHA碰撞) vs 调度接入(Scheduled无碰撞)
干扰: 同网干扰、异网干扰、近远效应
```

## 2. LoRaWAN容量分析

### 2.1 Pure ALOHA碰撞模型

LoRaWAN上行采用Pure ALOHA随机接入--设备有数据就发,不检测信道是否空闲:

```
吞吐量公式: S = G * e^(-2G)
- S: 归一化吞吐量(成功比例)
- G: 归一化信道负载
- 最大吞吐量: S_max = 1/(2e) = 18.4% (当G=0.5)
含义: 即使最优负载,信道利用率也只有18.4%,超过81%被碰撞浪费
```

### 2.2 扩频因子(SF)正交性

不同SF(7-12)之间近似正交,SF7信号不会与SF12碰撞:

```
但实际容量不是简单6倍:
- SF7 ToA: ~51ms (125kHz, 12字节)
- SF12 ToA: ~1647ms (125kHz, 12字节)
- SF12传输时间是SF7的32倍!

SF分布对容量影响:
| 策略 | 相对容量 |
|------|----------|
| 均匀分布(各1/6) | 1x基准 |
| 距离自适应 | ~2.5x |
| ADR优化 | ~3-4x |
```

### 2.3 单网关容量计算

```
公式: N_max = (T_period * DC) / (ToA * (1 + retry_factor))

示例(SF7, 1%占空比, 51ms ToA, 0.3重传系数):
N_packets/hour = (3600 * 0.01) / (0.051 * 1.3) = 543 包/小时/信道

3信道 x 6SF(加权) = 约4000-6000 包/小时/网关
若每设备每小时发1包: 支持约4000-6000设备/网关
```

### 2.4 实际容量衰减

| 因素 | 容量影响 | 原因 |
|------|----------|------|
| ALOHA碰撞 | -50%~-70% | 随机接入固有损耗 |
| 确认帧开销 | -10%~-30% | 下行ACK占用接收窗口 |
| 近远效应 | -10%~-20% | 远距离设备被近距离淹没 |
| 异网干扰 | -5%~-30% | 同频段其他LoRaWAN网络 |

**实际经验值**: 单网关城市环境稳定支撑约1000-3000设备(每小时1包,PDR>95%)。

## 3. 占空比对容量的影响

### 3.1 欧洲占空比限制

```
EU868频段:
| 子频段 | 频率 | 占空比 | 功率 |
|--------|------|--------|------|
| g(默认) | 868.0-868.6 | 1% | 14 dBm |
| g1 | 868.7-869.2 | 0.1% | 14 dBm |
| g2 | 869.4-869.65 | 10% | 27 dBm |

1%占空比 = 每小时最多发送36秒
SF7(51ms): 每小时最多706包
SF12(1647ms): 每小时最多21包!
```

### 3.2 各SF占空比约束下的容量

```
1%占空比, 单设备每小时可发包数:
| SF | ToA(12B) | 包数/小时 |
|----|----------|-----------|
| 7  | 51ms     | 706       |
| 8  | 92ms     | 391       |
| 9  | 165ms    | 218       |
| 10 | 329ms    | 109       |
| 11 | 659ms    | 54        |
| 12 | 1647ms   | 21        |

关键洞察: SF12是瓶颈,ADR推设备到低SF对容量帮助巨大
```

### 3.3 美国vs欧洲

美国FCC无硬性占空比限制,频段902-928MHz(26MHz宽,64信道),功率30dBm。欧洲ETSI有占空比,频段863-870MHz(7MHz,3-8信道),功率14dBm。结论: 美国LoRaWAN容量约为欧洲的3-5倍。

## 4. Sigfox容量分析

### 4.1 超窄带(UNB)优势

```
Sigfox: 单消息仅占100Hz带宽
- 192kHz可用带宽 / 100Hz = 1920条并行消息(理论)
- 加上时间(消息~2秒): 每秒峰值~960条

对比LoRaWAN(SF7): 单消息占满整个125kHz信道
Sigfox在频率维度并行度远高于LoRaWAN
```

### 4.2 Sigfox容量限制与冗余

```
服务限制: 上行140条/天, 下行4条/天, Payload 12字节
基站容量: 理论100万+设备, 实际~50万设备/基站

抗碰撞策略: 每条消息发送3次(时间+频率分集)
| 基站设备数 | 单次PDR | 3次冗余PDR |
|-----------|---------|-----------|
| 10,000    | 98%     | 99.99%    |
| 100,000   | 82%     | 99.4%     |
| 500,000   | 60%     | 93.6%     |
```

## 5. NB-IoT容量分析

### 5.1 调度接入的优势

NB-IoT使用LTE调度机制,基站分配资源,从根本上避免ALOHA碰撞:

```
LoRaWAN: 设备随机发 -> 碰撞丢包
NB-IoT: 基站调度[A][B][C] -> 无碰撞(RACH有少量碰撞但基站可检测重调度)
```

### 5.2 NB-IoT容量模型

```
资源: 180kHz带宽, 12子载波 x 1000子帧/秒 = 12000 RU/秒

| 覆盖条件 | 重复次数 | RU/包 | 设备数(1包/小时) |
|----------|----------|-------|-----------------|
| 良好(MCL<130dB) | 1 | 4 | ~10,800,000 |
| 普通(MCL<144dB) | 16 | 64 | ~675,000 |
| 极端(MCL<164dB) | 128 | 512 | ~84,000 |

实际值(含开销): 良好覆盖约50,000-100,000设备/小区
```

### 5.3 部署模式影响

Standalone(独立GSM频段)容量最大无干扰; In-band(LTE内一个PRB)受LTE调度影响; Guard-band(LTE保护带)需注意邻道干扰。

## 6. 网络间干扰分析

### 6.1 同频干扰场景

```
场景1: 两个LoRaWAN运营商共享EU868 -> 碰撞概率随密度线性增长
场景2: LoRaWAN vs Sigfox同频段 -> LoRaWAN受Sigfox窄带干扰
场景3: NB-IoT -> 授权频段,无同技术异网干扰(核心优势)
```

### 6.2 干扰量化

```python
import math

def collision_probability(n_devices_b, n_channels, n_sf, toa_avg, period):
    """两个共存LoRaWAN网络的碰撞概率"""
    traffic_b = (n_devices_b * toa_avg) / (period * n_channels * n_sf)
    p_collision = 1 - math.exp(-2 * traffic_b)
    return p_collision

# 示例: 对方网络5000设备, 3信道6SF, 100ms ToA, 每小时1包
p = collision_probability(5000, 3, 6, 0.1, 3600)
print(f"碰撞概率: {p:.4f}")  # 约1.5%, 乘以5000设备=每小时75包丢失
```

### 6.3 缓解策略

| 策略 | 效果 | 代价 |
|------|------|------|
| 频率规划 | 高 | 需运营商协调 |
| 监听前发送(LBT) | 中高 | 增加延迟功耗 |
| 时间同步 | 高 | 需GPS/网络同步 |
| 网关密度增加 | 高 | 基础设施成本 |

## 7. ADR对容量的影响

### 7.1 ADR原理

```
1. 设备发送数据
2. 网关测量SNR
3. 服务器分析最近20包SNR
4. 计算: SNR余量 = 实测SNR - 解调门限 - 安全余量(10dB)
5. 余量>0则降低SF/功率, <0则提高
目标: 每设备用满足通信的最低SF
```

### 7.2 ADR容量提升量化

```
1000设备, 覆盖半径5km:

无ADR(全SF12): ToA=1647ms, 碰撞~35%, 有效容量~650设备
有ADR(按距离分配):
| 距离 | SF | 设备占比 | ToA |
|------|-----|---------|------|
| 0-1km | SF7 | 4% | 51ms |
| 1-2km | SF8 | 12% | 92ms |
| 2-3km | SF9 | 20% | 165ms |
| 3-4km | SF10 | 28% | 329ms |
| 4-5km | SF11 | 20% | 659ms |
| 5km+ | SF12 | 16% | 1647ms |

加权平均ToA: ~420ms, 容量提升约3.9倍, 有效容量~2500设备
```

## 8. 网关密集化策略

### 8.1 多网关容量增益

```
单网关(中心): 半径5km, 边缘设备用SF11/12, 容量~3000设备
3网关(三角, 间距3km): 有效半径~2.5km, 多数设备用SF7-9
等效容量~12000设备(4倍而非3倍!)

额外增益来源:
1. SF降低效应: 近网关 -> 低SF -> 占用时间短
2. 空间分集: 同一包被多网关收到,抵消碰撞
3. 干扰分散: 每小区负载低,碰撞概率指数下降
```

### 8.2 网关选址优化

优化目标: 最小化网关数满足覆盖+容量。约束: 每设备至少被1网关覆盖,每网关负载<上限,95%设备PDR>目标值。常用算法: 贪心(逐步添加到负载最高区)、K-means聚类(网关放质心)、整数线性规划(精确最优)。

## 9. 实践案例: 智慧城市10万设备

### 9.1 设备清单

```
| 类型 | 数量 | 发送频率 | 包大小 |
|------|------|----------|--------|
| 智能水表 | 40,000 | 4次/天 | 20B |
| 智能路灯 | 15,000 | 12次/天 | 30B |
| 垃圾桶传感器 | 10,000 | 6次/天 | 15B |
| 空气质量 | 5,000 | 24次/天 | 40B |
| 共享单车 | 20,000 | 48次/天 | 25B |
| 物流追踪 | 10,000 | 24次/天 | 30B |
| 停车传感器 | 8,000 | 96次/天 | 10B |
总计: 108,000设备, 日消息~4,500,000条
```

### 9.2 LoRaWAN容量评估

```
日消息4.5M, 高峰系数2.5 -> 高峰小时~469,000包
单网关(8信道, ADR优化): ~4000包/小时(PDR>95%)
所需网关: 469,000/4,000 = ~117个, 安全余量后~160个
ADR优化(80%设备SF7-9): 所需网关降至~80个
```

### 9.3 NB-IoT容量评估

```
单基站(3PRB): >2,000,000包/小时
高峰需求469,000 -> 理论2-3个基站即可承载
但覆盖(城区3-5km半径)需15-20个基站
容量远非瓶颈,覆盖才是部署密度决定因素
```

### 9.4 混合部署建议

```
固定低频(水表、垃圾桶): NB-IoT -- 深度覆盖好,无需自建
固定中频(路灯、空气): LoRaWAN -- 灵活自建,80网关装路灯杆
移动设备(单车、物流): NB-IoT -- 运营商全城覆盖,移动性好
停车传感器: LoRaWAN -- 地下停车场需自建网关
```

## 总结

LPWAN密集部署容量规划需综合考虑物理层特性、MAC协议效率和环境因素。LoRaWAN基于ALOHA的随机接入使其容量受限于碰撞概率,需通过网关密集化、ADR优化和信道规划提升。NB-IoT依托蜂窝调度机制容量远高于LoRaWAN但成本模式不同。实际部署中混合方案往往最优--根据设备特性、部署位置和数据模式选择最合适的LPWAN技术。容量规划需从理论分析出发,结合仿真验证,通过实测迭代优化。
