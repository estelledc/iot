# 地理路由GPSR在IoT传感器网络中的应用
> **难度**: 高级 | **领域**: 路由协议 | **阅读时间**: 约 22 分钟

## 引言

想象你在一个陌生城市里要去某个地方,手里没有地图也没有导航软件,但你知道目的地在你的东北方向。最直觉的策略就是: 每到一个路口,选那条最朝东北方向走的路。这个"凭方向感走路"的策略,就是地理路由的核心思想。

传统网络路由就像查地铁线路图 -- 每个站都有完整的路线表,知道去任何目的地该坐哪条线。但对于大规模IoT传感器网络,成千上万个节点要维护路由表,内存和通信开销巨大。地理路由另辟蹊径: 只要每个节点知道自己的位置和邻居的位置,就能做出转发决策,不需要任何路由表。

GPSR(Greedy Perimeter Stateless Routing)是地理路由中最经典的协议,它用两种模式解决了"凭方向走路遇到死胡同"的问题,是理解地理路由绕不开的基石。

## 1. 地理路由的基本概念

### 1.1 什么是地理路由

地理路由的核心思想是: 用物理位置代替网络地址来做路由决策。每个节点需要知道三件事:

- 自己的地理位置(通过GPS或定位算法获得)
- 邻居节点的位置(通过周期性的beacon消息交换)
- 目的节点的位置(通过位置服务获得)

有了这些信息,转发决策就变成了一个简单的几何问题: 在所有邻居中,选离目的地最近的那个,把数据包交给它。

### 1.2 与传统路由的本质区别

传统路由(如AODV、RPL)依赖路由表,路由表记录了"要去目的地X,下一跳是节点Y"。这种方式的问题在于:

- 路由表占内存: 网络越大,表越大
- 路由发现有开销: 需要洪泛路由请求消息
- 拓扑变化需要更新: 节点移动或失效时,路由表要重新计算

地理路由则完全不同:

- 不需要路由表: 每个节点只存邻居列表
- 不需要路由发现: 转发是局部决策
- 天然适应拓扑变化: 邻居变了,下次beacon就更新了
- 可扩展性好: 每个节点的状态量是O(1),与网络规模无关

### 1.3 地理路由的适用场景

地理路由特别适合以下IoT场景:

- 大规模传感器网络: 节点数量上千,路由表存不下
- 移动节点: 拓扑频繁变化,传统路由更新跟不上
- 资源受限设备: 32KB RAM的MCU,没空间存大路由表
- 户外部署: GPS信号可用,位置信息容易获取

## 2. GPSR协议概述

### 2.1 两种转发模式

GPSR的全称是Greedy Perimeter Stateless Routing,它的精髓在于两种模式的切换:

**贪心模式(Greedy Mode)**: 正常情况下,选离目的地最近的邻居转发。这是默认模式,效率高,路径接近最短路。

**周边模式(Perimeter Mode)**: 当贪心模式遇到"局部最小值"(没有邻居比自己离目的地更近)时,切换到周边模式,沿着平面图的面边界绕行,直到找到比卡住的地方更接近目的地的节点,再切回贪心模式。

这两种模式的配合,就像走路时"能直走就直走,遇到墙就沿墙绕"。

### 2.2 协议状态

GPSR中"Stateless"意味着: 数据包的转发决策不依赖于节点存储的路由状态,只依赖于数据包中携带的信息(目的地位置)和节点本地已知的信息(邻居位置)。每个节点独立做决策,不需要与其他节点协调。

## 3. 贪心转发机制

### 3.1 基本算法

贪心转发的算法非常简单:

```
输入: 数据包(携带目的地D的位置), 当前节点S, 邻居列表N
输出: 下一跳节点

1. 对每个邻居 n in N:
   计算 dist(n, D)  // 邻居n到目的地D的欧几里得距离
2. 选择 next_hop = argmin(dist(n, D))  // 选最近的
3. 如果 dist(next_hop, D) < dist(S, D):
   转发给 next_hop  // 有进展,继续贪心
4. 否则:
   切换到周边模式  // 卡住了,需要绕行
```

### 3.2 贪心转发的优势

贪心转发之所以在大多数情况下表现良好,是因为:

- 计算简单: 只需要比较距离,O(N)复杂度(N为邻居数)
- 路径质量好: 每一跳都在向目的地靠近,路径长度接近最优
- 延迟低: 不需要等待路由发现,收到包立即转发
- 自适应: 邻居变化时自动适应,不需要重新计算路由

### 3.3 贪心转发的一个具体例子

假设一个农田监测网络,节点A在坐标(0,0),要把土壤湿度数据发给汇聚节点Sink在(100,80)。A的邻居有:

| 邻居 | 坐标 | 到Sink的距离 |
|------|------|-------------|
| B | (20,15) | 102.0 |
| C | (25,10) | 103.1 |
| D | (15,25) | 98.4 |

A到Sink的距离是128.1。邻居D到Sink最近(98.4),所以A选D作为下一跳。D收到后重复这个过程,直到数据到达Sink。

## 4. 局部最小值问题

### 4.1 什么是局部最小值

局部最小值(Local Minimum)是贪心转发的致命弱点。当一个节点发现: 自己的所有邻居都比自己离目的地更远,贪心转发就卡住了。

这种情况在以下场景中常见:

- 覆盖空洞(Void/Hole): 某个区域没有节点部署
- 物理障碍: 河流、建筑物等阻断了节点部署
- 稀疏部署: 节点密度不够,形成覆盖间隙

### 4.2 局部最小值的形象理解

想象你要往东走去目的地,但面前是一堵南北方向的长墙。你的所有邻居(墙这边的人)都比你离目的地更远,因为目的地在墙那边。这时候你不能继续"选离目的地最近的方向走",你需要"沿着墙走",绕到墙的尽头再继续往东。

GPSR的周边模式就是这个"沿墙走"的策略。

### 4.3 检测局部最小值

检测很简单: 如果当前节点是所有邻居中离目的地最近的(没有邻居能让数据包更接近目的地),就判定为局部最小值。这个检测是贪心转发算法的第4步。

## 5. 周边路由机制

### 5.1 图平面化

周边模式需要在平面图上运行右手规则(Right-Hand Rule)。但无线网络的连通图通常不是平面的(链路会交叉),所以第一步是平面化。

GPSR使用两种平面化算法:

**相对邻域图(RNG)**: 如果节点u和v之间存在某个节点w,使得max(dist(u,w), dist(v,w)) < dist(u,v),则删除边(u,v)。直觉上: 如果两个节点之间有"更近的中间人",就不需要直接相连。

**Gabriel图(GG)**: 如果以边(u,v)为直径的圆内存在其他节点,则删除边(u,v)。Gabriel图是RNG的子集,边更少。

```
平面化示例:

原始图:          RNG平面化后:
A---B            A---B
|\ /|            |   |
| X |     =>     |   |
|/ \|            |   |
C---D            C---D

交叉边AC和BD被删除
```

### 5.2 右手规则遍历

在平面化后的图上,周边模式使用右手规则遍历面的边界:

- 想象你站在一个迷宫里,右手一直摸着墙走
- 每到一个节点,选择"最右边"的那条边(相对于你来的方向)
- 这样就会沿着面的边界绕一圈

具体来说: 数据包从节点u转发到v,v需要选择下一跳。v计算从u到v的方向角,然后逆时针扫描,选第一个邻居作为下一跳。

### 5.3 退出条件

周边模式不会无限绕行。退出条件是: 当数据包到达一个节点,该节点比"进入周边模式时的节点"离目的地更近,就切回贪心模式。

数据包头部记录了进入周边模式时的位置(称为Lp),每到一个新节点就检查: 当前节点到目的地的距离是否小于Lp到目的地的距离。如果是,说明已经绕过了空洞,可以重新使用贪心转发。

## 6. 位置获取技术

### 6.1 GPS定位

GPS是最直接的位置获取方式,精度可达2-5米。但对IoT来说有几个问题:

- 功耗高: GPS模块的功耗在30-50mW,对电池供电的传感器是巨大负担
- 室内不可用: GPS信号无法穿透建筑物
- 成本: GPS模块增加硬件成本

### 6.2 无GPS定位替代方案

对于不能使用GPS的场景,有多种替代方案:

**锚点定位**: 少量节点配备GPS(锚点),其他节点根据与锚点的距离估算自己的位置。

**RSSI三角测量**: 根据接收信号强度估算距离,三个以上已知位置的参考点就能定位。精度较低,典型误差5-10米。

**DV-Hop**: 类似于距离向量路由,每个节点计算自己到锚点的跳数,再根据平均每跳距离估算物理距离。适合均匀部署的网络。

**混合方案**: 在IoT中常见的做法是: 部分节点(如网关、路由器)配备GPS,其余节点用DV-Hop估算。成本和精度之间取得平衡。

### 6.3 位置精度对路由的影响

位置估计误差会影响GPSR的性能。如果节点不知道自己或邻居的准确位置:

- 贪心转发可能选错邻居: 以为某邻居更近实际上更远
- 可能产生路由环路: 节点A选B,B又选A
- 周边模式的平面化可能出错: 该删的边没删,不该删的删了

好消息是,GPSR对中等程度的位置误差有较好的容忍性。研究表明,当位置误差小于通信半径的20%时,GPSR的性能下降不超过15%。

## 7. GPSR在IoT中的优势

### 7.1 内存效率

传统路由协议需要存储路由表,表的大小与网络节点数成正比。对于1000个节点的网络,一条路由表项通常需要10-20字节,整张表需要10-20KB。对于只有32KB RAM的传感器节点,路由表可能占用一半以上的内存。

GPSR只需要存储邻居列表。一个节点通常有10-20个邻居,每个邻居需要存储: 节点ID(2字节) + 坐标(4字节) + 最后更新时间(2字节) = 8字节。总共只需要80-160字节,是传统路由的百分之一。

### 7.2 通信开销

传统路由协议(如AODV)在建立路由时需要洪泛路由请求(RREQ)消息,网络中每个节点都会收到并转发。一次路由发现的消息数量是O(N),N为网络节点数。

GPSR没有路由发现过程,唯一的控制开销是周期性的beacon消息(每个节点广播自己的位置)。Beacon是本地广播,只有一跳邻居收到。消息数量是O(N)但每条消息只传一跳,总通信量远小于路由发现。

### 7.3 扩展性

GPSR的每个节点只做局部决策,状态量与网络规模无关。这意味着: 无论网络有100个节点还是10000个节点,每个节点的存储和计算开销都一样。

### 7.4 自适应性

当网络拓扑变化时(节点加入、离开、移动),GPSR自动适应。新节点加入后,发送beacon通知邻居,邻居更新邻居列表,下次转发就能考虑新节点。不需要任何网络范围的更新过程。

## 8. 位置误差的影响与缓解

### 8.1 误差来源

在真实IoT部署中,位置估计总有误差:

- GPS误差: 2-10米,受遮挡影响
- DV-Hop误差: 可达通信半径的30-50%
- RSSI误差: 受多径效应影响,室内可达10米以上
- 节点移动: 位置信息过时

### 8.2 误差对性能的影响

位置误差对GPSR两种模式的影响不同:

**对贪心模式**: 误差可能导致选错下一跳。在极端情况下,可能产生路由环路(A发给B,B以为A更近又发回A)。但大多数情况下,只是路径稍微偏离最优,不会完全失效。

**对周边模式**: 影响更大,因为平面化依赖准确的距离计算。错误的平面化可能导致必要的边被删除,使得周边遍历找不到绕行路径。

### 8.3 缓解策略

- 有界容错: 在贪心选择时加入容错范围,只有当邻居明显更近时才转发
- 周期性位置更新: 缩短beacon间隔,保持位置信息新鲜
- 多次探测取平均: 位置估计做多次取平均,减少随机误差
- 回退机制: 如果检测到环路,标记该路径避免再次选择

## 9. 地理路由的变种

### 9.1 GEAR: 地理与能量感知路由

GEAR在转发决策中同时考虑距离和剩余能量。选择下一跳时,不仅选离目的地近的,还考虑哪个邻居电量更充足。这样可以避免某些节点因为位置好而被反复使用导致提前耗尽电量。

### 9.2 GEM: 地理组播

GEM将地理路由扩展到组播场景。当数据需要发送给多个目的地时,GEM在必要的分叉点复制数据包,每个副本继续用地理路由到达各自的目的地。

### 9.3 3D地理路由

许多IoT部署是三维的: 多层建筑、水下传感器、无人机网络。传统GPSR设计用于二维平面,三维扩展面临的挑战包括: 三维空间中的平面化更复杂,空洞的形状更多样,绕行算法需要在体积中导航而不是沿面边界。

## 10. 地理路由与拓扑路由的对比

### 10.1 特性对比

| 特性 | 地理路由(GPSR) | 拓扑路由(AODV/RPL) |
|------|---------------|-------------------|
| 路由表 | 不需要 | 需要 |
| 位置信息 | 必须 | 不需要 |
| 扩展性 | 优秀(O(1)状态) | 受限(O(N)状态) |
| 空洞处理 | 复杂(周边模式) | 自然处理(路由发现绕过) |
| 移动性 | 天然支持 | 需要路由维护 |
| 室内适用 | 受限(需定位) | 适用 |

### 10.2 选择建议

**选地理路由**: 大规模网络(>500节点)、节点可能移动、户外部署有GPS、内存极度受限。

**选拓扑路由**: 小规模网络(<100节点)、室内部署无GPS、链路质量差(地理近不代表链路好)、网络有明确的层次结构。

## 11. 实际案例: 农业传感器网络

### 11.1 场景描述

某农场部署了2000个土壤传感器,覆盖10平方公里。每个传感器监测土壤湿度、温度和pH值,数据汇报到农场建筑处的汇聚节点(Sink)。

节点硬件配置: 32KB RAM、低成本GPS模块、通信半径100米。

### 11.2 为什么选GPSR

- 2000个节点,传统路由表需要约40KB,超过了RAM容量
- 户外部署,GPS信号良好
- 节点位置固定,位置误差小
- 新节点可能随时加入(季节性扩展)

### 11.3 运行效果

**正常转发**: 大部分数据包用贪心模式直达Sink,路径长度接近直线距离除以通信半径,平均12-15跳。

**遇到空洞**: 农场中间有一条河,没有部署传感器,形成覆盖空洞。靠近河边的节点在贪心模式下卡住,自动切换到周边模式,沿河岸绕行,绕过后切回贪心模式。

**动态适应**: 春天新增传感器时,新节点发送beacon,周围节点立即感知,下次转发就可能选择新节点作为中继。不需要任何网络重配置。

### 11.4 性能数据

与AODV路由协议相比:

- 内存使用: GPSR约160字节 vs AODV约40KB(减少99.6%)
- 路由发现开销: GPSR为零 vs AODV每次新路由需要洪泛
- 网络寿命: GPSR延长约25%,因为省去了路由发现消息的能量
- 扩展到2000节点: GPSR性能无明显下降,AODV路由表已超出内存
- 数据到达率: GPSR约92%(受空洞影响),AODV约96%(路由发现能找到路径)

## 总结

GPSR代表了一种与传统路由完全不同的思路: 用空间位置替代路由表,用局部几何计算替代全局路由发现。对于大规模IoT传感器网络,这种"无状态"的设计带来了极低的内存开销和出色的扩展性。贪心模式和周边模式的组合解决了大多数路由场景,虽然空洞处理的复杂性和对位置精度的依赖是其局限,但在GPS可用的户外IoT部署中,GPSR仍然是一个值得认真考虑的协议选择。
