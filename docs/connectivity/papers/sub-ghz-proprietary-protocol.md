# Sub-GHz私有协议设计与频段规划
> **难度**：🟡 中级 | **领域**：私有无线协议 | **阅读时间**：约 20 分钟

## 引言

想象你经营一家特殊的外卖店,服务的不是普通顾客,而是分散在整个农场里的几十个自动喂食器。每个喂食器每隔几个小时需要接收一次很短的指令——"倒出200克饲料"。你不需要美团外卖平台那种复杂的调度系统(标准协议),因为你的需求太简单了:指令很短、频率很低、但必须送达、而且送餐员的电动车电池要撑好几年才换一次。

于是你决定自己设计一套最精简的送餐流程(私有协议):精确定义信封格式(包结构)、选一条人少的路线(Sub-GHz频段)、制定最省油的驾驶规范(低功耗策略)。没有多余的功能,每一个bit都为你的场景服务。

## 1. 为什么选择私有协议

### 1.1 标准协议的局限

LoRaWAN、Zigbee、Z-Wave等标准协议已经能覆盖大量IoT场景,但总有一些需求它们无法完美满足:极致低功耗的场景下标准协议栈的入网和保活开销可能占掉大部分电量;某些工业控制需要毫秒级确定性延迟而标准协议的路由发现和冲突避免会引入不确定性;传感器数据只有4字节时标准协议20字节以上的帧头显得浪费;特殊行业可能需要国密算法而标准协议绑定了AES;百万级部署中认证费和芯片溢价也是显著成本。

### 1.2 私有协议的代价

选择私有协议意味着放弃标准协议的生态:没有现成设备可以接入、没有第三方工具链、安全性完全自己验证、维护升级成本全部自担、新员工需要学习非标准技术栈。做这个决定前要认真评估:你的需求是否真的超出了标准协议的能力?很多时候,在标准协议上做轻量定制比从零开始更经济。

## 2. Sub-GHz ISM频段规划

### 2.1 全球主要ISM频段

ISM(工业、科学和医疗)频段是免许可证使用的无线电频段。Sub-GHz范围内主要有:

| 地区 | 频段 | 主要法规 |
|------|------|---------|
| 欧洲 | 433MHz / 863-870MHz | ETSI EN 300 220 |
| 北美 | 315MHz / 902-928MHz | FCC Part 15 |
| 中国 | 470-510MHz / 779-787MHz | 工信部规定 |
| 日本 | 920-928MHz | ARIB STD-T108 |
| 澳大利亚 | 915-928MHz | ACMA标准 |

### 2.2 欧洲868MHz频段详细规划

欧洲868MHz频段最常用,但子频段规则复杂:

```
子频段    频率范围(MHz)       最大功率   占空比
g        863.0 - 868.0       25 mW     1%
g1       868.0 - 868.6       25 mW     1%
g2       868.7 - 869.2       25 mW     0.1%
g3       869.4 - 869.65      500 mW    10%
g4       869.7 - 870.0       25 mW     1%

1%占空比 = 每小时最多发射36秒
0.1%占空比 = 每小时最多发射3.6秒
g3子频段功率高且占空比宽松,但带宽窄(250kHz)
```

### 2.3 北美915MHz频段

美国FCC Part 15对902-928MHz频段的要求与欧洲不同:跳频扩频(FHSS)方式允许最大1W功率,需至少50个跳频信道;数字调制方式也允许最大1W,带宽至少500kHz;窄带非跳频方式限制在50mW。美国没有欧洲那样的占空比限制,但对跳频有最大驻留时间(dwell time)要求。

### 2.4 频率选择策略

选择工作频率要综合考虑:需要全球部署就考虑433MHz(多地区可用但天线较大);只面向单一市场就选当地主流频段(欧洲868MHz/北美915MHz);超远距离需求选较低频率;天线尺寸受限则选较高频率(868/915MHz的四分之一波长天线约8cm,433MHz则需约17cm)。

## 3. 调制方式选择

### 3.1 常用调制方式

Sub-GHz私有协议中常用的调制方式,从简单到复杂:

**OOK/ASK**: 最简单的调制,数据1发射载波、数据0不发射。优点是TX功耗最低、解调器最便宜;缺点是抗噪声差、灵敏度低。常用于遥控器和简单报警器。

**FSK**: 用两个不同频率表示0和1,频偏通常25-50kHz。抗噪声好,实现成熟,大多数Sub-GHz芯片都支持。是传感器网络最常用的调制方式。

**GFSK**: FSK的改进版,在调制前加高斯滤波器平滑频率跳变,占用带宽比FSK小,邻道干扰更低。蓝牙也使用GFSK。

### 3.2 调制参数与性能权衡

以FSK/GFSK为例,数据速率和灵敏度之间存在直接的权衡关系:

```
数据速率 vs 接收灵敏度(FSK典型值):
  1 kbps:   -120 dBm
  10 kbps:  -110 dBm
  50 kbps:  -105 dBm
  100 kbps: -100 dBm
  
每降低10dB灵敏度,通信距离大约减半
```

如果传感器每15分钟发一次10字节数据,用1kbps(传输80ms)和100kbps(传输0.8ms)对单次功耗影响不大,但1kbps的灵敏度优势可以将通信距离翻倍。对于数据量小但距离要求远的场景,低速率是更优选择。

## 4. 数据包结构设计

### 4.1 最小化帧结构

私有协议的一大优势是帧结构可以极致精简:

```
最小帧结构(约10字节额外开销):

+----------+----------+------+--------+---------+-----+
| Preamble | SyncWord | Len  |  Addr  | Payload | CRC |
| 4 bytes  | 2 bytes  | 1 B  |  1 B   | N bytes | 2 B |
+----------+----------+------+--------+---------+-----+

Preamble: 10101010... 让接收器同步时钟
SyncWord: 唯一标识符,区分自己的包和其他信号
Len: 有效载荷长度
Addr: 设备地址(0-255)
Payload: 应用数据
CRC: CRC-16错误检测
```

对比标准协议的帧开销:私有最简帧约10字节,LoRaWAN帧约13字节,Z-Wave帧约15字节,Zigbee帧约25字节。当有效载荷只有4字节时,私有协议的有效载荷率为28.6%,而Zigbee只有13.8%。

### 4.2 Preamble设计

Preamble(前导码)长度需要权衡:

```
Preamble长度选择:
  太短(1-2字节): 接收器来不及同步,丢包率高
  推荐(3-5字节): 大多数Sub-GHz收发器能可靠同步
  太长(8-16字节): 浪费发射时间和电量

特殊场景 - 低功耗前导码检测(LPD):
  接收器每N ms唤醒,用低功耗电路检查信道是否有前导码
  发送方发送足够长的前导码(>N ms)确保能被检测到
  检测到前导码后接收器唤醒主接收器完成后续接收
  此场景Preamble可能需要几十到几百字节
```

### 4.3 SyncWord选择

SyncWord(同步字)用于区分自己的数据包和其他Sub-GHz信号。选择原则:不用全0或全1,与Preamble(0xAA)有明显区别,自相关性好(不容易与噪声误匹配)。常用值如0x2D01或0xD391(2字节),3字节SyncWord如0x7E154A可进一步降低误匹配率。

### 4.4 帧开销对比分析

```
不同协议的帧开销对比(发送4字节有效载荷):

协议           帧头开销    总包长    有效载荷率
私有最简帧      10字节     14字节    28.6%
LoRaWAN        13字节     17字节    23.5%
Z-Wave         15字节     19字节    21.1%
Zigbee         25字节     29字节    13.8%

当有效载荷较大(如50字节)时差距缩小:
私有帧: 50/60 = 83.3%
Zigbee: 50/75 = 66.7%
```

对于每次只发几个字节传感器数据的场景,私有协议的帧效率优势非常显著,直接转化为更短的发射时间和更低的功耗。

## 5. 功耗优化策略

### 5.1 发射端优化

发射通常是最耗电的操作。TX功耗等于发射电流乘以发射时间,两个方向都可以优化:降低TX功率(例如从+14dBm降到+10dBm,电流从约30mA降到约20mA,但距离减少约37%);减少发射时间(最小化包长度、去除不必要字段)。

### 5.2 接收端优化

接收功耗常被低估。RX电流通常与TX在同一量级(10-15mA),如果需要长时间监听,总能耗可能超过TX。三种常用的接收策略:

持续监听适合市电设备,延迟最低但功耗最高。定时轮询适合对延迟不敏感的电池设备,每T秒唤醒监听W毫秒。前导码触发唤醒使用超低功耗检测电路(约0.5mA)持续监听前导码信号,检测到后唤醒主接收器,适合需要低延迟的电池设备。

### 5.3 电池寿命估算

```
场景: 温度传感器,每15分钟发送一次
电池: 2节AA电池,3000mAh

每次发送周期:
  唤醒+测温+组包: 5ms * 5mA  = 0.007 uAh
  发射:          10ms * 30mA = 0.083 uAh
  等待ACK:       15ms * 15mA = 0.063 uAh
  每次总计: 0.153 uAh

每天:
  发送96次: 96 * 0.153 = 14.7 uAh
  睡眠: 1.0uA * 24h = 24 uAh
  每天总计: 38.7 uAh

理论电池寿命: 3000000 / 38.7 / 365 = 约21年
实际受电池自放电限制, 寿命约5-10年
```

## 6. 链路预算与距离计算

### 6.1 链路预算公式

链路预算决定通信距离:链路预算(dB) = TX功率(dBm) + TX天线增益(dBi) + RX天线增益(dBi) - RX灵敏度(dBm)。

```
示例计算:
  TX功率: +14 dBm
  天线增益: 0 dBi (简单鞭状天线,收发各一)
  RX灵敏度: -110 dBm
  链路预算: 14 + 0 + 0 - (-110) = 124 dB
  减去衰落余量15dB: 可用路径损耗 = 109 dB
```

### 6.2 距离估算

使用自由空间路径损耗公式反推距离。以868MHz和109dB可用路径损耗为例:自由空间理论距离约7.8km,实际室外(地面反射和遮挡)约2-4km,室内(多面墙壁)约100-500m。

天线选择也直接影响距离:PCB走线天线增益-3到0dBi,尺寸最小成本最低;弹簧天线0到2dBi,通用性好;鞭状天线(四分之一波长)0到2dBi,868MHz长约8cm;偶极子天线2到3dBi,适合网关。

## 7. 常用射频芯片

### 7.1 主流选型

| 芯片 | 厂商 | 特点 |
|------|------|------|
| CC1310 | TI | Sub-GHz + ARM M3,超低功耗 |
| CC1352R | TI | 双频Sub-GHz+2.4GHz,ARM M4F |
| EFR32FG14 | Silicon Labs | 灵活射频引擎 + ARM M4 |
| SX1276 | Semtech | LoRa + FSK双模,灵敏度极高 |
| STM32WL | ST | 集成Sub-GHz射频的MCU |
| Si4463 | Silicon Labs | 纯射频收发器,需外部MCU |

### 7.2 选型决策

需要集成MCU就选CC1310/CC1352/EFR32FG/STM32WL,BOM更简单;成本极致就用Si4463纯收发器搭配便宜的外部MCU;需要LoRa兼容就选SX1276或STM32WL;需要同时支持Sub-GHz和BLE双频就选CC1352或EFR32FG。

### 7.3 代码示例

以TI CC1310为例的简化发送流程:

```c
// CC1310 私有协议发送 (伪代码)
#include "rf_driver.h"

// 射频配置: 868MHz, GFSK, 50kbps, +14dBm
RF_Config rfConfig = {
    .frequency = 868,
    .modulation = GFSK,
    .dataRate = 50000,
    .txPower = 14
};

// 组装数据包
uint8_t packet[16];
packet[0] = DEVICE_ADDR;
packet[1] = sizeof(sensorData);
memcpy(&packet[2], &sensorData, sizeof(sensorData));

// 发射并等待完成
RF_transmit(rfHandle, packet, 2 + sizeof(sensorData));
```

## 8. 安全设计

### 8.1 安全威胁与对策

私有协议面临四类主要威胁:窃听(监听无线信号获取数据)、重放攻击(录制合法包稍后重发)、伪造(冒充合法设备)、干扰(发射噪声阻断通信)。

针对前三类威胁,大多数现代Sub-GHz芯片内置了AES-128硬件加速器,可以实现高效的加密和认证:

```
AES-128-CCM加密方案:
  明文: [Addr][Len][Payload]
  加密后: [Addr][Len][密文Payload][MIC-4字节]
  
  Payload用AES-128-CTR模式加密(保密性)
  MIC用AES-128-CBC-MAC计算(完整性+认证)
```

### 8.2 防重放攻击

两种常用方案:递增计数器方案在每个包中附加4字节计数器,接收方只接受大于上次值的包,简单有效;滚动码方案让收发双方维护同步的伪随机序列,每次使用下一个值,常用于车钥匙和车库门遥控器。

## 9. 实际设计案例

### 9.1 农业传感器网络

```
场景: 50个大棚温湿度传感器,500m范围,星型拓扑

协议参数:
  频率: 868MHz, 调制: GFSK 50kbps
  TX功率: +14dBm, 上报间隔: 15分钟

数据包格式(16字节):
  Preamble   4B (0xAAAAAAAA)
  SyncWord   2B (0xD391)
  DeviceID   2B (最多65535个设备)
  SeqNum     2B (防重放计数器)
  Temp       2B (有符号,0.01度精度)
  Humidity   2B (0.01%精度)
  Battery    1B (电压*50)
  CRC        2B (CRC-16)

安全: AES-128-CCM,预共享密钥
预计电池寿命: 2节AA电池超过5年
```

### 9.2 信道接入策略

50个节点不能同时发送,需要协调。对于这种低密度场景,简单ALOHA方案最实用:每个节点在自己的15分钟周期内加一个随机延迟(0-10秒)后发送。每次发射仅占用约8ms信道时间,50个节点每15分钟各发一次,碰撞概率远低于1%。实现最简单,不需要时间同步,适合低密度低频率的传感器场景。

如果节点密度更高或发送频率更高,可以升级到TDMA(时分多址):网关定期广播信标包含时间信息和时隙分配,每个节点在分配的时隙内发送。需要网关和节点之间保持时间同步。

## 总结

Sub-GHz私有协议设计需要在多个维度做权衡。频段选择要同时考虑法规合规和物理特性,不同地区的频率规划差异很大,产品设计初期就要锁定目标市场。调制方式选择本质上是距离和速率的权衡,对于数据量小的IoT传感器场景,低速率换高灵敏度通常是更优选择。帧结构要尽可能精简,去除不需要的字段,这是私有协议相比标准协议的核心优势。安全方面AES-128加密和防重放机制是必须的底线,好在现代芯片都有硬件支持。

私有协议适合需求明确、场景固定、对功耗和成本有极致要求的项目。对于大多数通用IoT场景,标准协议(LoRaWAN/Zigbee/Z-Wave)仍然是更稳妥的选择。
