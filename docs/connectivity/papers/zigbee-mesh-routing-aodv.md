# Zigbee Mesh路由AODV算法分析
> **难度**：🔴 高级 | **领域**：Zigbee网络层 | **阅读时间**：约 22 分钟

## 引言

假设你在一座大型办公楼里, 需要把一份文件从一楼前台送到十二楼的某个办公室。你不知道该怎么走, 于是站在大厅里喊了一声: "谁知道怎么去十二楼1205室?" 这个喊声被周围的人一个接一个传递出去。终于, 十二楼有人听到了, 沿着传递链路回了一句: "从这条路走。" 于是你就获得了一条到达目的地的路线。

这就是AODV(Ad-hoc On-demand Distance Vector)路由算法的核心思想: 不预先维护到所有目的地的路由, 而是在需要时才"喊一声"去发现路径。对于Zigbee这类资源受限的物联网mesh网络来说, 这种"按需路由"的策略非常合适 -- 大多数时间设备并不需要和网络中所有节点通信, 预先维护完整路由表是对有限内存和计算资源的浪费。

本文将深入分析Zigbee网络中AODV路由算法的工作原理、Zigbee对标准AODV的修改、路由维护机制, 以及在实际部署中的性能考量。

## 1. Zigbee路由概述

### 1.1 为什么需要路由

在Zigbee mesh网络中, 并非所有设备都在彼此的通信范围内。一个传感器可能距离协调器(网关)很远, 中间需要经过多个路由器节点才能将数据送达。路由的任务就是找到从源节点到目标节点的多跳路径。

```
直接通信 vs 多跳路由:

  直接通信(范围内):
  [传感器A] -----无线直达-----> [协调器]

  多跳路由(超出范围):
  [传感器B] --> [路由器1] --> [路由器2] --> [协调器]
  (B无法直达协调器, 需要通过路由器中继)
```

### 1.2 路由策略选择

Zigbee网络提供两种路由策略, 各有适用场景:

**AODV网格路由**: 通过路由发现过程找到最优路径。路径质量好, 但需要额外的控制开销。适用于路由器之间的通信。

**树路由(Tree Routing)**: 沿着网络的树形层级结构逐跳转发。不需要路由发现, 路径可能不是最优的, 但实现简单且无额外开销。当路由表满或AODV发现失败时作为后备方案。

```
同一网络中两种路由的对比:

  网络拓扑:
          [C] (协调器)
         / | \
       [R1][R2][R3]
       /    |    \
     [R4]  [R5]  [R6]
      |
     [R7]

  R7 -> R2 的路由:
    树路由:  R7 -> R4 -> R1 -> C -> R2  (5跳, 经过协调器)
    AODV:    R7 -> R4 -> R1 -> R2       (3跳, 跳过协调器)
             或 R7 -> R4 -> R5 -> R2    (可能发现更优路径)
```

从上例可以看出, AODV能找到更短或更优的路径, 而树路由则机械地沿层级上下传递。

## 2. 标准AODV算法基础

### 2.1 核心思想

AODV是一种反应式(Reactive)路由协议, 其核心原则:

- **按需发现**: 只在需要发送数据时才寻找路由, 不主动维护到所有节点的路径
- **路由请求广播**: 源节点广播RREQ(Route Request)报文来寻找目标
- **路由回复单播**: 目标节点(或有路由信息的中间节点)通过RREP(Route Reply)单播回复
- **路由维护**: 路由建立后持续使用, 直到超时或链路断裂

### 2.2 与主动式路由的对比

```
主动式路由(如OLSR):
  - 所有节点持续交换路由信息
  - 路由表始终包含到所有节点的路径
  - 发送数据时立即可用, 无发现延迟
  - 代价: 大量控制消息, 高内存占用

反应式路由(AODV):
  - 平时不交换路由信息
  - 需要时才发现路由, 有首包延迟
  - 内存占用低, 控制开销少
  - 适合: 低流量、资源受限的IoT网络
```

对于Zigbee网络来说, 大多数传感器只是周期性(如每分钟)上报一次数据, 网络流量稀疏。反应式路由的"按需"特性恰好匹配这种流量模式, 避免了不必要的控制开销。

## 3. Zigbee AODV的关键修改

Zigbee并未原封不动地采用标准AODV, 而是针对IoT网络特点做了重要修改。

### 3.1 链路开销替代跳数

标准AODV使用跳数(Hop Count)作为路径度量: 跳数最少的路径就是"最优"路径。但在无线网络中, 跳数最少不一定最好 -- 一条两跳路径的两个链路可能都很不稳定(丢包率高), 而一条三跳路径的每个链路可能都很可靠。

Zigbee AODV引入了链路开销(Link Cost)的概念:

```
跳数 vs 链路开销:

  路径A (2跳):
  [S] ====弱信号====> [R1] ====弱信号====> [D]
  跳数 = 2 (标准AODV会选择这条)
  链路开销 = 7 + 7 = 14 (Zigbee AODV不选这条)

  路径B (3跳):
  [S] --强信号--> [R2] --强信号--> [R3] --强信号--> [D]
  跳数 = 3 (标准AODV不选这条)
  链路开销 = 1 + 1 + 1 = 3 (Zigbee AODV选择这条)
```

**路径开销 = 各链路开销之和**, Zigbee AODV选择路径开销最小的路由。

### 3.2 链路开销的计算

链路开销基于LQI(Link Quality Indicator)计算。LQI是IEEE 802.15.4 PHY层提供的链路质量指示值, 范围0-255, 值越大表示信号质量越好。

```
LQI到链路开销的映射:

  LQI范围        链路质量      链路开销
  200-255        优秀          1
  150-199        良好          2-3
  100-149        一般          4-5
  50-99          较差          6
  0-49           极差          7

  开销值越低 = 链路越好 = 优先选择
```

也有实现使用数据包传输成功率(Packet Delivery Ratio, PDR)来计算链路开销, 原理类似: 传输成功率越高, 链路开销越低。

### 3.3 设计动机

使用链路开销而非跳数的核心原因在于IoT场景的特殊需求。可靠性优先: 数据的可靠传输比传输速度更重要, 选择可靠链路可以减少重传次数, 反而提高整体效率。能耗优化: 不可靠链路导致频繁重传, 每次重传都消耗电能, 选择可靠路径实际上降低了整体能耗。适应性: 无线环境动态变化(人员走动、设备移动、干扰变化), 基于LQI的开销能反映实时链路质量。

## 4. 路由发现过程

### 4.1 RREQ广播

当源节点需要向某个目标发送数据, 且路由表中没有该目标的路由时, 触发路由发现。RREQ报文包含RREQ ID(唯一标识本次请求)、源地址、目标地址、路径开销(累积值)和半径(最大传播跳数)。

发现过程逐步展开:

```
步骤1: 源节点S广播RREQ (路径开销=0)
  [S] --RREQ--> [R1] (收到, 记录反向路由到S)
  [S] --RREQ--> [R2] (收到, 记录反向路由到S)
  [S] --RREQ--> [R3] (收到, 记录反向路由到S)

步骤2: R1、R2、R3各自增加链路开销后继续广播
  [R1] --RREQ--> [R4] (路径开销 = S到R1的开销)
  [R2] --RREQ--> [R5] (路径开销 = S到R2的开销)
  [R3] --RREQ--> [R6] (路径开销 = S到R3的开销)

步骤3: RREQ继续扩散, 直到到达目标D
  可能经过不同路径到达D, D收到多个RREQ
```

每个转发RREQ的中间节点都会: 记录RREQ的来源(建立反向路由), 将自己到上一跳的链路开销加到路径开销中, 然后重新广播修改后的RREQ。

### 4.2 重复RREQ的处理

由于RREQ是广播的, 一个节点可能通过不同路径收到同一个RREQ的多个副本。节点通过RREQ ID和源地址的组合来识别重复。如果是重复的RREQ且路径开销更高则丢弃; 如果路径开销更低则更新反向路由并重新广播。

### 4.3 RREP回复

目标节点收到RREQ后, 不会立即回复, 而是等待一小段时间(通常几十毫秒), 收集通过不同路径到达的RREQ。然后选择路径开销最低的那条路径, 沿反向路由发送RREP:

```
RREP回传过程:

  [D] --RREP--> [R5] --RREP--> [R2] --RREP--> [S]

  每个中间节点收到RREP后:
  1. 记录正向路由(到目标D的下一跳)
  2. 转发RREP给反向路由的下一跳

  最终, S到D路径上的每个节点都知道:
  - 正向: 发往D的数据, 下一跳是谁
  - 反向: 发往S的数据, 下一跳是谁
```

路由建立完成后, 源节点即可开始发送数据。

## 5. 路由维护

路由不是永久的。无线环境变化、设备移动或故障都可能导致路由失效。

### 5.1 路由超时

每条路由表项都有一个过期时间。如果在过期时间内该路由没有被使用(没有数据通过), 路由表项将被标记为过期并最终删除。数据通过时过期计时器重置, 确保路由表中只保留实际在用的路由, 节省宝贵的内存。

### 5.2 链路故障检测与修复

当中间节点尝试将数据转发给下一跳时, 如果MAC层连续多次未收到ACK确认(通常3次重试), 就判定该链路故障。

```
链路故障处理流程:

  [S] --> [R1] --> [R2] --X--> [R3] --> [D]
                   (R2到R3链路断裂)

  1. R2检测到转发失败(MAC ACK超时)
  2. R2生成RERR(Route Error)报文
  3. RERR沿反向路径传回源节点S:
     [R2] --RERR--> [R1] --RERR--> [S]
  4. 沿途每个节点标记相关路由为无效
  5. S收到RERR后, 如仍有数据要发, 启动新的路由发现
```

面对链路故障, Zigbee支持两种修复策略。源修复: RERR传回源节点, 由源节点重新发起路由发现, 路径全局最优但修复延迟较长。本地修复: 检测到故障的中间节点自行发起局部路由发现, 尝试绕过故障链路, 修复速度快但路径可能不是全局最优。

## 6. Many-to-One路由

### 6.1 IoT的典型流量模式

在大多数IoT网络中, 流量模式是"多对一"的: 大量传感器都向同一个协调器(网关)发送数据。如果每个传感器都独立发起AODV路由发现, 网络会被大量RREQ广播淹没。

### 6.2 Many-to-One路由发现

Zigbee引入了Many-to-One路由发现来解决这个问题。由协调器主动发起一次路由发现, 一次性建立从网络中所有路由器到协调器的路由:

```
Many-to-One路由发现:

  协调器广播Many-to-One RREQ:
  [C] --RREQ--> [R1] --> [R2] --> [R3]
  [C] --RREQ--> [R4] --> [R5]
  [C] --RREQ--> [R6] --> [R7] --> [R8]

  每个路由器收到后:
  1. 记录"到协调器的下一跳"(反向路由)
  2. 继续广播RREQ

  结果: 一次广播, 所有路由器都知道如何到达协调器
```

### 6.3 路由记录(Route Record)

Many-to-One解决了上行路由(设备到协调器)的问题。但协调器还需要知道如何向各设备发送数据(下行路由)。这通过路由记录机制实现: 当设备向协调器发送数据时, 数据包中携带一个路由记录字段, 沿途每个路由器将自己的地址追加到该字段中。协调器收到数据包后, 就知道了到达该设备的完整路径, 可以使用源路由(Source Routing)方式发送下行数据。

```
路由记录示例:

  [E1] 发送数据到 [C]:  E1 -> R3 -> R1 -> C

  数据包中的路由记录:
  发出时: [E1]
  经过R3: [E1, R3]
  经过R1: [E1, R3, R1]

  C收到后知道: 到E1的路径是 C -> R1 -> R3 -> E1
```

## 7. 树路由与网格路由的协同

在实际Zigbee网络中, AODV和树路由协同工作。树路由利用网络的树形层级结构, 规则简单: 如果目标是后代节点则向下转发, 否则向上转发给父节点。地址范围的计算基于网络深度和每层最大子节点数。

```
路由选择逻辑:

  要发送数据到目标D:
    1. 查找路由表, 有到D的AODV路由?
       -> 有: 使用AODV路由(通常是最优路径)
       -> 无: 继续下一步
    2. 路由表有空间发起路由发现?
       -> 有: 发起AODV路由发现, 等待RREP
       -> 无: 继续下一步
    3. 使用树路由
       -> 沿树形结构转发(可能不是最优, 但保证能到达)
```

这种分层策略确保了: 有资源时追求最优路径, 资源不足时仍能保证连通性。

## 8. 路由表管理

### 8.1 受限的路由表

Zigbee设备的内存非常有限, 路由表大小通常只有几十条到几百条。每条路由表项包含: 目标地址(16bit)、状态(活跃/发现中/失效/过期)、下一跳地址(16bit)和路径开销。

### 8.2 淘汰与集中化策略

当路由表已满需要为新路由腾出空间时, 通常按优先级淘汰: 首先删除已过期的路由, 然后删除最久未使用的路由, 最后删除路径开销最大的路由。路由表满也是树路由作为后备方案的重要原因。

在Zigbee网络中, 只有路由器(Router)和协调器(Coordinator)维护路由表。终端设备(End Device)不参与路由, 它们的所有数据都通过父节点转发。这种设计将路由的内存和计算开销集中在有持续供电的路由器上, 让电池供电的终端设备保持最低功耗。

## 9. 可扩展性与广播风暴

AODV路由发现的核心机制是广播RREQ, 这在大规模网络中会带来问题。当网络规模超过200个节点时, RREQ广播风暴可能严重影响网络正常数据传输 -- 每个RREQ广播占用无线信道约1-2ms, 所有邻居节点都必须接收和处理。

Zigbee采用多种策略来缓解广播风暴。限制RREQ半径: 设置RREQ的最大传播跳数, 源节点可以先使用较小半径尝试, 失败后再增大。Many-to-One路由: 协调器主动发起一次路由发现, 替代所有设备各自发起的单独发现。路由缓存: 中间节点如果已知到目标的路由, 可以直接回复RREP。分层路由: 将大型网络划分为多个子网, 限制RREQ的扩散范围。

## 10. 实际部署案例

### 10.1 智能楼宇场景

考虑一栋3层办公楼, 每层部署约50个Zigbee设备(温湿度传感器、照明控制器、人体传感器), 共约150个设备。协调器(网关)位于2楼机房。

```
楼宇网络部署:

  3楼: [E31]...[E50] --> [R31] [R32] [R33] [R34]
                              \    |    /
  ====================== 楼层间链路 ======================
                              \   |   /
  2楼: [E21]...[E50] --> [R21] [R22] [C] [R23] [R24]
                              \    |    /
  ====================== 楼层间链路 ======================
                              \   |   /
  1楼: [E11]...[E50] --> [R11] [R12] [R13] [R14]

  C = 协调器(2楼机房)
  R = 路由器(每层4-5个, 持续供电)
  E = 终端设备(传感器, 电池供电)
```

### 10.2 路由行为与优化

在这个部署中, 上行数据(传感器到协调器)使用Many-to-One路由, 协调器周期性(如每30分钟)发起一次路由发现; 下行控制(协调器到设备)使用源路由; 同层设备间通信(如开关控制灯泡)使用AODV路由发现, RREQ半径设置为3-4跳即可。

针对这个场景的优化建议: 路由器布局上确保每层路由器之间可以直接通信形成可靠的mesh骨干网, 路由器之间的LQI应在150以上; Many-to-One周期根据拓扑变化频率调整, 稳定环境下30-60分钟一次; 信道选择上避开WiFi常用的2.4GHz信道; 路由表容量上协调器建议40-50条, 中间楼层路由器30-40条, 边缘楼层20-30条。

## 总结

Zigbee网络中的AODV路由算法通过按需发现路由, 在资源受限的IoT设备上实现了高效的多跳mesh通信。与标准AODV相比, Zigbee的关键修改是使用基于LQI的链路开销替代简单跳数, 优先选择可靠路径而非最短路径。Many-to-One路由机制解决了IoT网络中"多对一"流量模式的效率问题。树路由作为AODV的后备方案, 在路由表满或发现失败时保证网络连通性。在实际部署中, 路由器布局、路由表大小、Many-to-One周期和信道选择是影响路由性能的关键因素。对于超过200个节点的大规模网络, 需要通过限制RREQ半径、分层路由等策略来缓解广播风暴问题。
