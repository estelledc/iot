# 唤醒接收器在IoT MAC层中的超低功耗设计
> **难度**：🔴 高级 | **领域**：超低功耗通信 | **阅读时间**：约 22 分钟

## 引言

在物联网 MAC 层设计中，最大的能量浪费来源不是数据发送，而是**空闲监听（Idle Listening）**——射频前端保持开启等待可能永远不会到来的数据包。占空比循环（Duty Cycling）能缓解但无法根治：低占空比省电却延迟高，高占空比响应快却仍浪费能量。

唤醒接收器（Wake-up Receiver, WuRx）提供了突破性思路：用一个功耗极低的辅助接收器始终监听触发信号，主射频仅在确实需要通信时才被唤醒。这种"双射频"架构正在重新定义事件驱动型 IoT 网络的 MAC 层设计。

---

## 1. 空闲监听问题

### 1.1 能量浪费的根源

典型 IoT 射频接收功耗 10-100 mW，但事件频率可能仅每小时甚至每天几次。结果是 99% 以上的接收能量消耗在空等上。

以一个每小时采集一次数据的温度传感器为例：每次通信持续 100ms，一小时中有效工作时间不到万分之三，但接收器如果持续开启就要消耗 36-360 焦耳/小时的能量。

### 1.2 占空比的固有矛盾

占空比方案让设备周期性短暂开启接收器检查信道：

- **低占空比（1%）**：省电但延迟高达数百毫秒，可能错过紧急事件
- **高占空比（10%）**：延迟低但空唤醒仍浪费 90% 的接收能量
- **空唤醒问题**：即使信道为空也要消耗唤醒能量（射频启动过渡态功耗）
- **不可预测流量**：事件驱动型应用无法优化睡眠周期

理想状态：射频完全关闭直到**确定**需要通信时才启动。

---

## 2. 唤醒接收器基本概念

### 2.1 双射频架构

WuRx 系统在每个设备上部署两套射频前端：

**主射频（Main Radio）**：高性能数据通信收发器，支持完整调制解调和信道编码，功耗高（发送 20-100 mW，接收 10-60 mW），负责实际数据交换。

**唤醒接收器（WuRx）**：超低功耗简单接收器，始终处于开启状态，功耗极低（1-100 微瓦），仅负责检测特定的唤醒信号并产生硬件中断。

### 2.2 工作流程

1. 正常状态：主射频关闭，MCU 深度睡眠，仅 WuRx 开启
2. 发送方需要通信：在 WuRx 频率上发送唤醒信号
3. WuRx 检测到唤醒信号：产生硬件中断唤醒 MCU
4. MCU 启动主射频：进入正常通信流程
5. 通信完成：主射频关闭，MCU 回到睡眠，WuRx 继续监听

### 2.3 功耗极低的原因

WuRx 不做传统接收器的大部分工作：不需要本振（LO）和混频器（无超外差结构）、不需要 ADC 采样和数字信号处理、不需要解调解码、不需要时钟同步和频率校准。本质上它只做一件事：判断"有信号"还是"没信号"。

## 3. WuRx 硬件架构

### 3.1 信号链路

```
天线 → 匹配网络 → 包络检测器 → 比较器/放大器 → 地址匹配逻辑 → 唤醒控制器 → MCU中断
```

### 3.2 核心模块

**包络检测器（Envelope Detector）**：WuRx 的核心。使用肖特基二极管或 MOSFET 进行射频整流，提取信号幅度包络。作为无源器件接近零功耗。输出是射频幅度的低频表示。

**比较器（Comparator）**：将包络输出与预设阈值比较，超过阈值输出高电平表示检测到唤醒信号。功耗亚微瓦级，如 TI TLV3691 仅 75nW。

**地址匹配逻辑**：可选模块，使用移位寄存器相关器逐位比较接收序列与预存设备地址，全部匹配才产生唤醒中断。减少误唤醒但增加 1-10 μW 功耗。

**唤醒控制器**：产生 MCU 中断信号，可能包含去抖动逻辑，功耗可忽略。

---

## 4. 功耗分解分析

### 4.1 WuRx 模块功耗

| 模块 | 典型功耗 | 备注 |
|------|----------|------|
| 包络检测器 | ~0（无源） | 肖特基二极管整流 |
| 比较器 | 50-500 nW | 超低功耗比较器 |
| 地址匹配逻辑 | 1-10 μW | 视地址长度和实现而定 |
| 偏置电路 | 0.5-5 μW | 为有源器件提供工作点 |
| **总计** | **1-50 μW** | 典型学术原型 |

### 4.2 与主流射频对比

| 技术 | 接收功耗 | WuRx 功耗比 |
|------|----------|-------------|
| BLE（接收） | 5-10 mW | WuRx 低 100-10000× |
| LoRa（接收） | 10 mW | WuRx 低 200-10000× |
| WiFi（接收） | 60-80 mW | WuRx 低 1200-80000× |
| Zigbee（接收） | 20-25 mW | WuRx 低 400-25000× |

### 4.3 系统级功耗示例

事件驱动传感器（每小时 1 次事件，每次通信 100ms，主射频 20mW）：

- **占空比方案（1%）**：平均接收功耗 = 20mW × 1% = 200 μW + 空唤醒开销 ≈ 250 μW
- **WuRx 方案**：持续监听 10 μW + 主射频摊薄 20mW × 100ms/3600s ≈ 0.56 μW → 总计 ≈ 10.6 μW
- **节省约 24 倍能量**

## 5. 唤醒信号设计

### 5.1 调制方式

通常使用 **OOK（On-Off Keying）**：有载波 = 逻辑"1"，无载波 = 逻辑"0"。这是最简单的数字调制，完美匹配包络检测器——有能量/无能量的二值判断。

### 5.2 信号结构

```
[前导码 4-8bit] [同步字 8bit] [设备地址 8-16bit] [校验位 4-8bit]
```

- 前导码：让 WuRx 电路稳定并检测信号存在
- 同步字：标识唤醒帧起始
- 设备地址：目标设备唯一标识
- 校验位：CRC 或奇偶校验防止误唤醒

### 5.3 数据速率

典型 1-100 kbps。较低速率灵敏度更好（积分时间更长），较高速率延迟更低。常见选择约 10 kbps，32 位帧传输仅需 3.2ms。

### 5.4 频率选择

**同频方案**：WuRx 与主射频相同频段，复用天线。简化硬件但增加干扰风险。

**异频方案**：WuRx 使用专用频段（如 Sub-GHz 唤醒 + 2.4GHz 数据）。隔离性好但需额外天线。

---

## 6. 地址机制

**广播唤醒**：任何唤醒信号唤醒所有设备。最简单，WuRx 无需地址逻辑，但误唤醒多。适合紧急警报等所有设备都需响应的场景。

**地址唤醒**：信号包含目标设备 ID，只有匹配设备唤醒。实现方式为移位寄存器逐位比较。地址长度权衡——8 位（低功耗，支持 256 设备，误唤醒率 1/256）vs 16 位（支持 65536 设备，误唤醒率 1/65536，但功耗增加 0.5-1 μW/位）。

**组播唤醒**：地址分为组 ID + 设备 ID。组播时只匹配组 ID 唤醒一组设备，单播时匹配完整地址。适合分区管理的传感器网络。

---

## 7. 灵敏度与功耗权衡

### 7.1 核心限制

WuRx 灵敏度（能检测的最小信号功率）决定唤醒距离。提高灵敏度需要放大→放大需要功耗。

| WuRx 功耗 | 典型灵敏度 | 估算唤醒距离（868MHz） |
|-----------|-----------|----------------------|
| 1 μW | -50 dBm | 10-30m |
| 10 μW | -60 dBm | 30-100m |
| 50 μW | -70 dBm | 100-300m |
| 100 μW | -80 dBm | 300-1000m |

### 7.2 与主射频的差距

主射频灵敏度 -95 到 -137 dBm，WuRx 典型 -50 到 -70 dBm，差距 25-87 dB。意味着唤醒距离远小于数据通信距离。

解决途径：更高功率的唤醒信号（增加发送端功耗）、接受更高 WuRx 功耗换灵敏度、网络拓扑补偿（链式唤醒）、接受有限的唤醒范围（部分场景可接受）。

## 8. 基于 WuRx 的 MAC 协议

### 8.1 基本交互流程

```
发送方                              接收方
  |--- [WuRx信道] 唤醒信号 -------->|  ← WuRx 检测到信号
  |                                 |  ← MCU 唤醒 + 主射频启动（5-10ms）
  |<-- [主信道] Ready 信号 ---------|
  |--- [主信道] 数据帧 ------------>|
  |<-- [主信道] ACK ----------------|
  |                                 |  ← 主射频关闭，MCU 睡眠
```

总额外延迟（相比始终在线）：5-20 ms，对大多数 IoT 应用完全可接受。

### 8.2 多跳网络中的 WuRx

中继节点大部分时间完全休眠。上游节点先发唤醒信号 → 中继醒来 → 转发数据 → 唤醒下一跳 → 自己休眠。逐跳唤醒（Sequential Wake-up）避免全网同时活跃，极大节省总网络能耗。

### 8.3 异步与同步模式

**异步**：WuRx 始终监听，发送方任何时候可发唤醒信号。延迟最低，功耗恒定。

**同步**：WuRx 也用占空比（如每 10 秒醒 50ms 检查），进一步降功耗但增延迟。适合预算极严苛场景。

## 9. 与占空比的详细对比

| 指标 | 占空比（1%） | WuRx（10μW） |
|------|------------|-------------|
| 空闲平均功耗 | 200 μW | 10 μW |
| 响应延迟 | 100ms-数秒 | 5-20ms |
| 空唤醒浪费 | 有（每次周期唤醒） | 无（仅真实事件唤醒） |
| 突发事件响应 | 可能错过 | 立即响应 |
| 适合流量模式 | 周期性/可预测 | 事件驱动/突发 |
| 硬件复杂度 | 单射频 | 双射频（额外成本） |
| 标准化程度 | 成熟 | 新兴（仅802.11ba） |

占空比的隐性成本：RTC 时钟维护功耗、同步协议开销、射频启动过渡态（1-5ms 高峰值电流）、上下文恢复时间。

WuRx 优势最显著的场景：事件极不频繁但响应要求高（火灾报警）、流量完全不可预测（按需查询）、延迟敏感的下行通信、长期部署（10+ 年电池目标）。

## 10. 当前研究挑战

**灵敏度瓶颈**：在 <10 μW 功耗下实现 >-80 dBm 灵敏度仍困难。研究方向：不确定性放大器（Uncertain-IF）、注入锁定振荡器、MEMS 谐振器增强选频。

**误唤醒（False Wake-up）**：WiFi、蓝牙、微波炉等非唤醒信号可能触发 WuRx。OOK 包络检测器无法区分信号来源。解决方案：更长地址码、时域特征校验、频率选择性滤波器。权衡：更强抗干扰 → 更高功耗和延迟。

**集成挑战**：额外 PCB 面积（天线+匹配+检测电路）、两套射频间天线耦合隔离、额外 BOM 成本（虽然器件本身便宜）、微瓦级功耗精确测量需要专用仪器。

**标准化缺失**：除 IEEE 802.11ba 外无通用 IoT WuRx 标准。无统一唤醒信号格式、无跨厂商互操作性、3GPP/802.15.4/LoRa Alliance 均无 WuRx 规范。阻碍大规模商业化。

**商用芯片有限**：AS3933（ams AG，125kHz，2.7μA，-69dBm）是少数商用选择。大部分 WuRx 停留在学术原型阶段，缺乏完整参考设计和开发工具链。

---

## 11. IEEE 802.11ba：WiFi 唤醒无线电

IEEE 802.11ba（2021 年发布）是首个正式纳入唤醒接收器的无线标准：

- 工作频率：2.4GHz，OOK 调制，4MHz 带宽
- 数据速率：62.5 kbps 或 250 kbps
- 支持地址唤醒（选择性唤醒特定设备）
- 功耗目标：<1 mW（比学术 WuRx 高，但对 WiFi 场景足够）
- 唤醒距离与 WiFi 数据距离匹配
- 帧格式兼容现有 WiFi 基础设施

意义：使 WiFi IoT 设备可能实现年级别电池寿命（传统 WiFi 仅天/周级别）+ 始终可达（AP 随时唤醒设备）+ 高带宽通信的组合。这对智能家居、企业 IoT 场景意义重大。

## 12. 实践案例：桥梁结构健康监测

### 12.1 场景描述

大型桥梁部署 100 个振动传感器用于结构健康监测（SHM, Structural Health Monitoring）。这是 WuRx 技术最具说服力的应用场景之一，因为它完美匹配了 WuRx 的核心优势：极低频率事件 + 极高响应速度要求。

具体需求：
- 正常状态：桥梁结构健康，传感器无需上报数据，可以睡眠数小时甚至数天
- 异常事件：地震、超重车辆通过、结构裂缝扩展产生的异常振动
- 关键约束：事件发生时所有 100 个传感器需在 50ms 内同步启动高速振动数据采集
- 电池寿命：安装在桥梁结构内部，更换困难，要求 10+ 年无维护运行

### 12.2 为什么占空比不可行

- 事件频率极低（每月 1 次），99.99% 的唤醒都是空唤醒
- 50ms 响应要求占空比需 >2%，在 10 年寿命目标下功耗不可接受
- 异步占空比无法保证 100 个传感器同时唤醒
- 同步占空比需要持续的信标广播，增加系统复杂度

### 12.3 WuRx 解决方案

**架构**：每个传感器配备加速度计 + MCU + 主射频 + WuRx。一个"哨兵节点"持续低功耗监测振动基线。

**唤醒链反应**：哨兵检测到异常振动 → 广播 WuRx 唤醒信号 → 覆盖范围内传感器唤醒（~10ms）→ 被唤醒节点转发唤醒信号给更远节点 → 链式传播 → 全网 30-50ms 内全部唤醒 → 同步高速数据采集 → 数据回传网关 → 全体深睡。

### 12.4 性能分析

**电池寿命**：WuRx 持续 10 μW + 年均 12 次事件（每次 5 秒通信）→ 平均功耗 ≈ 11 μW。1000mAh 纽扣电池理论寿命 > 15 年。

**响应时间**：唤醒信号传输 3ms + MCU 唤醒和射频启动 5ms + 链式传播 3 跳 × 8ms = 24ms → 总响应 ~32ms < 50ms 要求。

**核心结论**：10+ 年电池寿命 + <50ms 即时响应——这个性能组合在纯占空比方案中**不可能实现**。WuRx 打破了延迟与功耗的线性耦合。

### 12.5 其他适用场景

同样的设计模式可推广到：
- **森林火灾监测**：烟雾/温度传感器网络，平时完全静默，一旦检测到异常立即唤醒周围节点进行协作确认
- **地震预警网络**：分布式地震仪阵列，P 波到达时快速唤醒全网同步采集
- **工厂紧急停机**：安全传感器检测到危险条件时，瞬间唤醒所有执行器执行停机序列
- **军事/安防**：入侵检测传感器网络，平时零功耗运行，触发时快速组网上报

---

## 总结

唤醒接收器代表 IoT 超低功耗 MAC 层的范式转变：从"定期醒来看看有没有事"到"有事才醒来"。核心价值在于打破占空比方案中延迟与功耗的线性耦合关系。

当前瓶颈集中在灵敏度、抗干扰性和商业化成熟度上。随着 802.11ba 发布和半导体工艺进步，WuRx 有望在未来 5-10 年从学术原型走向大规模商用，特别是在事件驱动型和延迟敏感型 IoT 应用中。
