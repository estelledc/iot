# Sigfox 0G网络架构与超窄带技术
> **难度**: 中级 | **领域**: LPWAN | **阅读时间**: 约 20 分钟

## 引言

想象你在一个嘈杂的体育场里,需要传递一条简短消息给对面看台的朋友。你不需要喊整段对话,只需要用一种极其尖锐、独特的音调喊出几个字——即使周围万人呐喊,朋友也能辨认出你的声音。Sigfox的超窄带(Ultra-Narrowband)技术就是这个原理: 把信号压缩到极窄的频带里,用"尖锐"换"穿透力",让微弱的信号能在巨大的噪声中被识别出来。

Sigfox自称"0G网络"——比2G/3G/4G/5G都更简单的一代。它追求的不是更快、更大带宽,而是更简单、更便宜、更省电。本文深入解析这种"极简主义"网络的技术架构和设计哲学。

## 1. Sigfox与0G理念

### 1.1 0G的含义

0G不是"没有网络",而是"零复杂度网络"——从设备端的视角来看:

- 零连接建立(No connection setup): 设备不需要握手、注册、认证
- 零协商(No negotiation): 不需要协商速率、信道、参数
- 零同步(No synchronization): 设备不需要与基站保持时间同步
- 零管理(No network management): 设备不需要处理任何网络管理消息

设备要做的事情极其简单: 有数据? 直接发射到空中。就这样。

### 1.2 设计哲学对比

```
传统蜂窝网络(如NB-IoT)设备需要做的事:
1. 搜索网络 -> 2. 同步时钟 -> 3. 随机接入 -> 4. 认证
-> 5. 附着网络 -> 6. 建立连接 -> 7. 发送数据
-> 8. 确认接收 -> 9. 释放连接

Sigfox设备需要做的事:
1. 发送数据

这就是"0G"的含义: 设备侧复杂度趋近于零
```

### 1.3 极简主义的代价

极简带来的限制是显而易见的:

- 上行每天最多140条消息
- 每条消息最大12字节有效载荷
- 下行每天最多4条消息
- 没有确认机制(设备不知道消息是否送达)
- 无法实时控制设备

但对于"每天报个温度/位置/状态"的场景,这些限制完全可以接受。

## 2. 超窄带物理层技术

### 2.1 UNB(Ultra-Narrowband)原理

Sigfox物理层的核心创新是超窄带调制:

| 参数 | Sigfox UNB | 对比: LoRa | 对比: NB-IoT |
|------|-----------|-----------|-------------|
| 信号带宽 | 100Hz | 125kHz | 180kHz |
| 调制方式 | DBPSK/GFSK | CSS(啁啾扩频) | OFDM/SC-FDMA |
| 数据速率 | 100bps/600bps | 300bps-50kbps | 最高250kbps |
| 接收灵敏度 | -142dBm | -137dBm | -141dBm |
| 链路预算 | 163dB | 157dB | 164dB |
| 频段 | 868MHz/902MHz | Sub-GHz | 运营商授权频段 |

100Hz带宽意味着什么? 整个Sigfox的一条消息占用的频谱宽度,只有LoRa信号的1/1250,NB-IoT的1/1800。这就像在一条千车道的高速公路上,Sigfox只占用了一根自行车道的宽度。

### 2.2 DBPSK调制

Sigfox上行链路使用DBPSK(差分二进制相移键控): 数据"0"时载波相位不变,数据"1"时载波相位翻转180度。接收端只需比较相邻符号的相位差,无需恢复绝对载波相位。这种调制方式对频率偏移容忍度高(适合低成本晶振),解调简单,在极窄带宽内性能优异。

### 2.3 为什么窄带宽能提升灵敏度

这是Sigfox最核心的物理原理: 热噪声功率 = kTB,其中k是玻尔兹曼常数,T是温度,B是带宽。带宽越窄,接收器中的噪声功率越低。

```
噪声比较:
WiFi (20MHz):    噪声功率 = kT x 20,000,000 = -101 dBm
LoRa (125kHz):   噪声功率 = kT x 125,000   = -123 dBm
Sigfox (100Hz):  噪声功率 = kT x 100       = -154 dBm

Sigfox接收器的噪底比WiFi低53dB!
这意味着Sigfox能检测到WiFi无法察觉的极微弱信号
```

### 2.4 频率随机接入

Sigfox设备发射时不选择固定信道,而是在整个可用频段(如EU868: 868.034-868.226MHz, 192kHz宽)内随机选择发射频率。每条消息只占100Hz,理论上192kHz频段可同时容纳1920条消息。基站使用宽带SDR接收整个频段,通过FFT检测所有活动信号并逐一解调。

这种ALOHA式随机接入完全消除了设备端的信道分配复杂度。碰撞概率通过重复发送来对冲。

## 3. 消息格式与协议设计

### 3.1 上行消息结构

每条Sigfox上行消息格式固定:

```
Sigfox上行帧格式:

+----------+----------+---------+----------+-------+
| 前导码   | 帧同步字 | 设备ID  | 有效载荷 | MAC   |
| 4字节    | 2字节    | 4字节   | 0-12字节 | 2字节 |
+----------+----------+---------+----------+-------+
                                     |
                               最大12字节!

总帧长: 12-26字节(含开销)
空中时间: 约2秒(100bps, EU频段) / 约0.5秒(600bps, US频段)
```

### 3.2 三次重复发送机制

为了在没有ACK的情况下保证可靠性,Sigfox使用三重冗余: 每条消息在不同随机频率上发送3次(时间分集+频率分集)。基站只要收到任意一次即可。若单次丢包率为10%,三次全丢的概率仅为0.1%,有效送达率达99.9%。

### 3.3 下行链路设计

Sigfox下行链路(基站到设备)非常受限:

- 每天最多4条下行消息
- 只在设备主动请求时才能下发
- 设备发送上行后打开短暂接收窗口(类似LoRaWAN Class A)
- 下行消息最大8字节有效载荷
- 使用GFSK调制,600bps

这种不对称设计反映了Sigfox的核心理念: IoT设备主要是"说"(上报数据),很少需要"听"(接收命令)。

### 3.4 消息配额与容量

上行140条/天(每条最大12字节),下行4条/天(每条最大8字节)。日上行数据容量仅1,680字节(1.64KB),相当于8条纯文字微信消息。这个限制决定了Sigfox只适合极简数据上报场景。

## 4. 网络架构

### 4.1 极简拓扑

Sigfox网络架构有意设计得极其简单:

```
Sigfox网络拓扑:

                          +------------------+
[设备] ---无线--->  [基站] ---IP网络--->  | Sigfox Cloud    |
[设备] ---无线--->  [基站] ---IP网络--->  | (巴黎/新加坡)   |
[设备] ---无线--->  [基站] ---IP网络--->  |                 |
                          | - 解调确认       |
                          | - 去重           |
                          | - 设备注册       |
                          | - API暴露        |
                          +--------+---------+
                                   |
                                   v
                          [用户应用服务器]
                          (通过回调/API获取数据)
```

关键设计决策: 所有智能都在云端,基站尽量"笨",设备最"笨"。

### 4.2 基站设计

Sigfox基站与传统蜂窝基站有根本区别:

| 特性 | Sigfox基站 | 传统蜂窝基站 |
|------|-----------|-------------|
| 接收方式 | 宽带SDR接收 | 信道化接收 |
| 处理方式 | 转发原始信号到云端 | 本地解调解码 |
| 天线 | 全向或三扇区 | 多扇区MIMO |
| 发射功率 | 仅下行需要(低) | 高(上下行) |
| 安装复杂度 | 低(类似WiFi AP) | 高(需要机房) |
| 成本 | 约$3000-5000 | $50,000-200,000 |

### 4.3 覆盖模型

一个Sigfox基站的覆盖范围:

- 农村/开阔地: 30-50km
- 郊区: 10-20km
- 密集城区: 3-5km

这得益于极高的链路预算(163dB)。作为对比,一个典型WiFi热点的覆盖范围只有30-50米。

### 4.4 云端处理

Sigfox将大量处理推到云端: 多基站接收同一消息时进行去重;基于多基站RSSI进行TDOA定位(无需GPS);消息认证和设备身份验证;通过回调(Callback)将数据推送给用户应用。

## 5. Sigfox公司历史与UnaBiz收购

### 5.1 发展历程

| 年份 | 事件 |
|------|------|
| 2010 | Sigfox在法国图卢兹成立 |
| 2012 | 首次商用部署(法国) |
| 2015 | 累计融资超过3亿欧元 |
| 2018 | 覆盖72个国家,声称"全球最大IoT网络" |
| 2020 | 财务困难开始显现 |
| 2021 | 进入法国司法重组程序 |
| 2022 | UnaBiz(新加坡)完成收购 |
| 2023 | UnaBiz整合运营,网络继续服务 |

### 5.2 失败原因分析

Sigfox作为独立公司的失败有多重原因:

商业模式问题: Sigfox试图做"全球IoT运营商",在每个国家建设和运营网络。但IoT数据收入增长远慢于基础设施投资速度。与LoRaWAN(设备商/用户自建)和NB-IoT(运营商已有基础设施)相比,Sigfox的重资产模式压力巨大。

技术限制: 12字节有效载荷和每天140条消息的硬限制,使很多应用场景无法满足。竞争对手(特别是NB-IoT)提供了更灵活的能力。

### 5.3 UnaBiz收购后的变化

UnaBiz是一家新加坡IoT公司,收购Sigfox后:

- 维持现有网络运营
- 推出"融合LPWAN"策略(Sigfox + LoRaWAN双模)
- 聚焦亚太市场(新加坡、日本、台湾)
- 降低运营成本,寻找可持续商业模式

## 6. 与LoRaWAN的深入对比

### 6.1 架构哲学对比

| 维度 | Sigfox | LoRaWAN |
|------|--------|---------|
| 设计哲学 | 极简(设备零复杂度) | 灵活(设备有一定智能) |
| 网络控制 | 全球统一运营 | 各方可自建网络 |
| 设备端协议栈 | 极小(几KB代码) | 较大(几十KB) |
| 双向通信 | 极度受限(4条/天下行) | 支持(Class B/C) |
| 自适应速率 | 无(固定100bps) | ADR自动调整 |
| 有效载荷 | 最大12字节 | 最大242字节 |
| 确认机制 | 无(靠重复发送) | 可选ACK |
| 设备成本 | 略低(协议栈更简单) | 略高 |
| 网络部署权 | 仅Sigfox/合作运营商 | 任何人 |

### 6.2 适用场景分化

Sigfox更适合: 数据量极小(几字节状态/位置)、完全不需要下行控制、设备成本极度敏感、不需要自建网络、报告频率低(每小时或更低)。

LoRaWAN更适合: 数据量较大或变化频繁、需要下行命令(固件更新、参数调整)、需要自主可控的网络、需要确认送达、灵活的数据速率需求。

## 7. 典型应用案例

### 7.1 物流资产追踪

这是Sigfox最经典的应用场景:

```
Sigfox物流追踪器工作模式:

设备构成:
  - Sigfox射频模块(成本约$2)
  - GPS模块(成本约$3)或WiFi扫描定位
  - 加速度计(检测运动)
  - 纽扣电池(CR2477, 约$1)
  - 总BOM: 约$8-12

工作流程:
  1. 加速度计检测到物品移动
  2. 唤醒GPS获取坐标(约30秒, 30mA)
  3. 将坐标编码为12字节:
     纬度(4字节) + 经度(4字节) + 电量(1字节) + 状态(3字节)
  4. 通过Sigfox发射(2秒, 50mA)
  5. 设备回到深度睡眠(1uA)

电池寿命: 每天定位4次,约3-5年
设备零月租: 使用Sigfox包年套餐约$1/年/设备
```

### 7.2 漏水检测

公用事业场景中,Sigfox非常适合部署在难以维护的位置:

- 传感器放置在地下管道或建筑夹层
- 检测到水时发送12字节报警(位置ID + 水位 + 时间戳)
- 每天心跳1次确认设备存活
- 电池寿命10年+(极少发射)
- 部署后数年不需要任何维护

### 7.3 环境监测

大面积部署环境传感器:

- 温湿度每小时上报一次
- 12字节载荷: 温度(2字节) + 湿度(2字节) + 气压(2字节) + 电量(1字节) + 预留(5字节)
- 利用TDOA基站定位,无需GPS(节省成本和功耗)
- 适合农田、森林、城市街道等分布式部署

## 8. 设备端实现与代码示例

### 8.1 最小硬件配置

Sigfox设备的硬件可以极其简单:

```
最小Sigfox设备BOM:

核心组件:
  MCU:     STM32L0系列 (ARM Cortex-M0+, 超低功耗)   $1.5
  射频:    S2-LP (ST Sub-GHz收发器)                   $1.0
  天线:    PCB板载天线或弹簧天线                      $0.1
  晶振:    TCXO 26MHz (温补晶振,保证频率精度)        $0.5
  匹配网络: 几个电阻电容                              $0.1

总射频前端BOM: 约$3-4
```

### 8.2 发射流程代码示例

```c
// Sigfox设备端发送示例(伪代码)
#include "sigfox_api.h"

typedef struct {
    int16_t temperature;  // 温度 x 100
    uint16_t humidity;    // 湿度 x 100
    uint32_t latitude;    // 纬度 x 1000000
    uint32_t longitude;   // 经度 x 1000000
} sensor_payload_t;       // 正好12字节

void send_sensor_data(void) {
    sensor_payload_t payload;
    payload.temperature = read_temperature() * 100;
    payload.humidity = read_humidity() * 100;
    payload.latitude = get_latitude() * 1000000;
    payload.longitude = get_longitude() * 1000000;

    // 发送: 无需连接/握手/认证,直接发射
    SIGFOX_API_send_frame(
        (uint8_t*)&payload, sizeof(payload),
        NULL, 0  // 不请求下行,不需要ACK
    );
    enter_deep_sleep();  // 立即回到睡眠
}
```

### 8.3 软件复杂度对比

| 协议 | Flash占用 | RAM占用 | 主要功能 |
|------|-----------|---------|----------|
| Sigfox | 5-10KB | 1-2KB | 帧组装 + 射频驱动 |
| LoRaWAN | 30-60KB | 5-10KB | MAC状态机 + ADR + 加密 |
| NB-IoT | 模组内部数百KB | 对主MCU透明 | 完整3GPP协议栈 |

Sigfox设备的固件可以极小,适合最低成本MCU运行。

## 9. 网络性能与定位能力

### 9.1 容量分析

单个Sigfox基站的理论容量:

```
容量计算(EU868频段):

可用频段: 192kHz
单消息带宽: 100Hz
同时可容纳: 1920条消息(理论)
消息空中时间: 2秒

实际考虑碰撞和三次重复:
  实际容量约为理论值的30-50%
  对于每天发送6条消息的设备:
  单基站可服务约数万-数十万台设备
```

### 9.2 延迟与定位

Sigfox消息的端到端延迟典型为10-30秒(含空中传输2秒、三次重复6-10秒、云端处理1-5秒),绝对不适合实时应用,但适合"最终一致性"的数据上报。

Sigfox还提供免费的设备定位服务(无需GPS): 基于多基站RSSI的三角定位,精度城区1-5km、郊区5-10km。无需设备端任何额外硬件,适合粗略区域定位(如"包裹在哪个城市")。

## 10. Sigfox在当今IoT格局中的位置

### 10.1 存量部署

尽管Sigfox公司经历了收购,但全球仍有大量存量部署:

- 全球约7000万注册设备
- 约75个国家有网络覆盖
- 欧洲(特别是法国、西班牙)是最大市场
- 公用事业(水表、气表)是最大垂直行业

### 10.2 UnaBiz的融合战略

UnaBiz收购后推动"0G + LoRaWAN"双模策略: 单芯片同时支持Sigfox和LoRaWAN,根据场景自动选择最优网络。Sigfox用于极简上报,LoRaWAN用于需要灵活性的场景,降低用户选择焦虑。

### 10.3 适用场景判断

2024年新项目是否还应该选择Sigfox:

适合考虑的情况: 目标市场在欧洲(覆盖好)、数据量极小且纯上行、设备成本极度敏感、已有Sigfox网络覆盖的区域。

不建议的情况: 需要双向通信、目标市场在Sigfox覆盖弱的地区、对网络长期可用性有严格要求、数据量超过12字节/条、需要开放标准和多供应商选择。

## 总结

Sigfox代表了IoT连接技术中的"极简主义"流派: 通过超窄带(100Hz)物理层实现极高的接收灵敏度(-142dBm),通过"无连接"协议设计将设备端复杂度降到最低,通过三次重复发送在没有ACK的情况下保证可靠性。

这种设计哲学在特定场景下依然有价值——当你只需要每天发送几个字节的状态数据,当设备成本必须压到极致,当你不需要任何下行控制能力。然而,Sigfox公司的商业失败提醒我们: 技术的优雅不等于商业的成功。重资产的全球运营商模式在IoT早期市场难以为继,而LoRaWAN的开放生态和NB-IoT的运营商支撑,提供了更可持续的路径。

UnaBiz的接手给Sigfox技术带来了新的生命,双模策略也是务实之选。对于已有Sigfox覆盖的区域和极简数据上报场景,这仍然是一个值得考虑的选项。
