# NB-IoT省电模式PSM与eDRX配置优化
> **难度**：🟡 中级 | **领域**：蜂窝IoT功耗 | **阅读时间**：约 20 分钟

## 引言

想象你在一个很无聊的会议里,但老板可能随时点你的名。如果你必须时刻盯着老板(传统LTE idle模式),你会累得不行。聪明的做法是: 要么跟老板说"我每小时来报到一次,其他时间别找我"(PSM模式),要么"我每10分钟看一眼有没有人找我"(eDRX模式)。NB-IoT设备面临同样的问题: 蜂窝协议要求定期"签到",但IoT设备的电池要撑10年,必须想办法少"醒来"。

本文深入分析NB-IoT的两大省电机制: PSM和eDRX,以及如何根据具体场景优化配置。

## 1. NB-IoT功耗挑战

### 1.1 蜂窝协议的固有开销

传统LTE设备在空闲模式下需要定期监听寻呼(DRX周期1.28-2.56秒唤醒一次)、周期性TAU以及监听系统信息更新。这些操作对IoT设备是灾难性的:

```
传统LTE空闲态功耗估算:
- DRX周期: 1.28s
- 每次唤醒: 持续约50ms, 电流约50mA
- 睡眠电流: 3uA
- 平均电流: (50mA*0.05s + 0.003mA*1.23s) / 1.28s = 约1.95mA

CR2032电池(225mAh): 仅能支撑约5天  -- 完全不可接受!
```

### 1.2 10年电池寿命的目标

NB-IoT的设计目标是AA电池支撑10年以上:

```
目标反推:
- AA电池容量: 约3000mAh
- 10年 = 87600小时
- 允许平均电流: 3000mAh / 87600h = 约34uA
- 必须将空闲态功耗降到uA级别
```

## 2. PSM(省电模式)详解

### 2.1 PSM工作原理

PSM的核心思想: 数据发完就关闭射频,进入深度睡眠,不再监听网络寻呼。

```
PSM状态机:
                    T3324到期
[数据传输] --> [Active Timer] --> [PSM深睡眠]
   ^                                    |
   |          T3412到期(周期性TAU)       |
   +------------------------------------+

PSM期间:
- 射频完全关闭, 不监听寻呼(网络无法主动联系设备)
- 电流降到3-5uA(仅MCU保持时钟)
- 设备仍注册在网络(无需重新附着)
```

### 2.2 PSM关键定时器

**T3324(Active Timer)**: 数据交换完成后设备保持可达状态的时长。

```
T3324编码(GPRS Timer 2):
- 单位可选: 2s / 1min / 6min
- 取值范围: 2s 到 186min
- 常用值: 10s / 30s / 1min
- 设为0: 立即进入PSM(最省电但完全不可达)
```

**T3412(Extended TAU Timer)**: 周期性跟踪区更新间隔。

```
T3412扩展编码(GPRS Timer 3):
- 单位可选: 10min / 1h / 10h / 2s / 30s / 1min / 320h
- 取值范围: 10min 到 413天
- 常用值: 1h / 4h / 24h
- 注意: 设置过长可能导致网络侧注销
```

### 2.3 PSM功耗剖面

```
电流(mA)
400|  *
   | ***  TX突发(200-400mA, 1-5s)
200| ****
   |     *
100|      **  Active Timer(50-100mA)
 50|        ********
   |                *
  0|                 _________________________ PSM(3-5uA)
   +--+---+--+-----+--------------------------> 时间
   0  1s  3s 13s   T3324到期               T3412到期
```

### 2.4 PSM平均功耗计算

假设每小时上报一次,Active Timer设为10秒:

```
实际PSM功耗计算:
- TX阶段: 300mA * 4s = 1200 mAs
- Active Timer: 60mA * 10s = 600 mAs
- PSM: 0.005mA * 3586s = 17.9 mAs
- 总电荷/周期: 1817.9 mAs
- 平均电流: 1817.9/3600 = 0.505mA

AA电池(3000mAh)寿命: 3000/0.505 = 5941小时 = 约248天
注意: 主要瓶颈在TX阶段,而非睡眠阶段
若改为每4小时上报: 寿命约4倍 = 约2.7年
```

## 3. eDRX(扩展非连续接收)详解

### 3.1 eDRX工作原理

eDRX将寻呼监听周期从1.28秒扩展到最长10485.76秒:

```
传统DRX (周期1.28s):
|醒|睡|醒|睡|醒|睡|醒|睡|  <-- 频繁唤醒

eDRX (周期20.48s):
|醒|_______________睡________________|醒|  <-- 很少唤醒

NB-IoT eDRX最大周期 (10485.76s = 约2.9小时):
|醒|________________________长时间睡眠________________________|醒|
```

### 3.2 eDRX周期选项

| 周期值 | 时间 | 适用场景 |
|--------|------|---------|
| 20.48s | 20秒 | 需要较快响应的执行器 |
| 81.92s | 82秒 | 一般监控设备 |
| 327.68s | 5.5分钟 | 非紧急下行需求 |
| 1310.72s | 21.8分钟 | 低优先级设备 |
| 5242.88s | 87.4分钟 | 接近PSM但保留可达性 |
| 10485.76s | 174.8分钟 | 最大省电 |

### 3.3 寻呼时间窗口(PTW)

每个eDRX周期内有一个寻呼时间窗口:

```
|<------------- eDRX周期 ------------->|
|<-PTW->|                              |
|DRX|DRX|DRX|DRX|______深睡眠__________|

PTW长度(NB-IoT): 2.56s 到 40.96s
PTW内DRX周期: 1.28s 或 2.56s
```

## 4. PSM vs eDRX: 选择策略

### 4.1 核心区别

| 特性 | PSM | eDRX |
|------|-----|------|
| 设备可达性 | 不可达 | 可达(有延迟) |
| 下行延迟 | 等设备下次唤醒 | 最大一个eDRX周期 |
| 睡眠电流 | 3-5uA | 3-5uA(相同) |
| 额外唤醒 | 无(仅TAU) | 每周期监听寻呼 |

### 4.2 组合使用

PSM和eDRX可以组合: Active Timer期间用eDRX保持可达性,T3324到期后进入PSM深度睡眠。

### 4.3 按场景选择配置

```
场景1: 每小时上报温度的传感器(无下行需求)
- PSM: T3412=1小时, T3324=10秒, 不需要eDRX
- 理由: 纯上行,无需保持可达

场景2: 智能门锁(需要接收开锁命令)
- eDRX: 周期=20.48秒, 不使用PSM
- 理由: 需要在20秒内响应开锁命令

场景3: 智能水表(每天上报+偶尔接收配置更新)
- PSM: T3412=24小时, T3324=5分钟
- eDRX: T3324期间使用, 周期=5.12秒
- 理由: 上报后5分钟内可接收响应,之后深睡眠
```

## 5. 网络协商机制

### 5.1 请求与授权流程

设备通过NAS信令请求参数,但网络有最终决定权:

```
设备                           网络(MME)
  |--- Attach Request ------->|
  |    (请求T3412=24h,        |
  |     T3324=10s,            |
  |     eDRX=20.48s)          |
  |<-- Attach Accept ---------|
  |    (授权T3412=12h,        |  <-- 网络可能缩短
  |     T3324=30s,            |  <-- 网络可能延长
  |     eDRX=40.96s)          |  <-- 网络可能调整
  |                           |
设备必须使用网络授权的值!
```

### 5.2 AT命令配置

```
// 配置PSM
AT+CPSMS=1,,,"01000011","00000101"
// T3412="01000011": 01=1小时单位, 000011=3 -> 3小时
// T3324="00000101": 000=2秒单位, 00101=5 -> 10秒

// 配置eDRX
AT+CEDRXS=1,5,"0010"
// 1=启用, 5=NB-IoT, "0010"=周期20.48秒

// 查询实际生效值
AT+CPSMS?
AT+CEDRXRDP
```

## 7. 常见陷阱与优化

### 7.1 常见问题

**问题1: 网络不支持请求的定时器值**
- 现象: 请求T3412=24小时,网络只授权4小时
- 对策: 查询并使用网络授权值,不要假设请求值生效

**问题2: TAU定时器过长导致注销**
- 现象: 长时间PSM后无法通信
- 对策: T3412不超过网络允许的最大值(通常24小时以内)

**问题3: 未计入RRC建立时间**
- 现象: 应用层延迟比预期长
- 对策: PSM唤醒后需重新建立RRC连接(1-10秒)

**问题4: eDRX和PSM交互不当**
- 现象: Active Timer内无法收到下行数据
- 对策: 确保eDRX周期远小于T3324

### 7.2 优化建议

```
1. 最小化Active Timer(T3324)
   - 不需要即时下行响应时设为最小值
   - 每多1秒 = 多消耗约50-80mA*1s

2. 对齐报告间隔与TAU定时器
   - T3412设为报告间隔的整数倍
   - 让TAU与数据上报合并执行

3. 仅在需要时启用eDRX
   - 纯上行设备不需要eDRX

4. 在目标网络实测
   - 不同运营商授权的参数不同
   - 覆盖差的环境TX时间更长

5. 考虑最坏情况
   - CE Level 2: RRC建立可能30秒+
   - 首次附着可能需要数分钟
```

## 8. 电池寿命估算

### 8.1 计算公式

```
寿命(小时) = 电池容量(mAh) / 平均电流(mA)
平均电流 = 每周期总电荷(mAs) / 周期时长(s)
每周期总电荷 = I_tx*t_tx + I_active*t_active + I_sleep*t_sleep
```

### 8.2 实际案例

```
场景: 每4小时上报一次, CE Level 0
- TX阶段: 250mA * 3s = 750 mAs
- Active Timer: 60mA * 10s = 600 mAs
- PSM: 0.005mA * 14387s = 72 mAs
- 总电荷: 1422 mAs
- 平均电流: 1422/14400 = 0.099mA = 99uA
- AA电池寿命: 3000/0.099 = 30303小时 = 约3.5年
- 实际(含自放电和温度): 约2.4年
```

## 总结

PSM和eDRX是NB-IoT实现超低功耗的核心机制。PSM通过彻底关闭射频监听实现最低功耗(3-5uA),代价是设备不可达; eDRX通过扩展寻呼周期在省电和可达性之间取平衡。两者可以组合使用。

实际部署的关键点: 理解网络协商机制(请求值不一定被接受)、在目标网络上实测功耗、根据应用需求选择合适的模式组合。设计良好的NB-IoT终端配合合理的PSM/eDRX配置,完全可以实现5-10年电池寿命。
