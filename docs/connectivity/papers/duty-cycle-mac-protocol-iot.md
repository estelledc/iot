# 占空比MAC协议在IoT低功耗通信中的设计
> **难度**: 高级 | **领域**: MAC协议 | **阅读时间**: 约 22 分钟

## 引言

想象你是一个保安, 负责看守一栋大楼的大门。如果24小时不间断站岗, 你很快就会累垮。聪明的做法是: 每隔10分钟出来巡视一次, 每次只看30秒。如果有人来访, 他们需要在门口等一小会儿, 但你可以精力充沛地工作很长时间。

这就是占空比(duty cycling)的核心思想。IoT设备的无线电收发器就像这个保安, 大部分时间处于休眠状态, 只在短暂的活跃窗口醒来检查是否有数据需要收发。通过这种方式, 电池寿命可以从几天延长到数年。

但这也引入了新问题: 发送方不知道接收方什么时候醒来, 如何保证消息不被错过? 如何平衡节能与延迟? 本文将深入探讨占空比MAC协议的设计原理和主要方案。

## 1. 占空比基本概念

### 1.1 为什么需要占空比

无线电收发器是IoT设备上最耗电的组件。关键事实:

- 发送功耗: 典型10-100mW
- 接收/监听功耗: 与发送相当(这是很多人的误解)
- MCU运行功耗: 1-10mW
- 传感器采样功耗: 0.1-1mW
- 深度睡眠功耗: 1-10uW

如果无线电始终开启监听信道, 即使没有数据通信, 电池也会在几天内耗尽。这就是idle listening(空闲监听)问题, 是低功耗无线网络的头号能耗杀手。

### 1.2 占空比定义

```
占空比 = 活跃时间 / 总周期时间

示例:
- 周期: 1秒
- 活跃时间: 10ms
- 占空比: 10ms / 1000ms = 1%

能耗降低:
- 始终开启: 功耗 P
- 1%占空比: 平均功耗约 0.01P (降低100倍)
```

典型IoT应用的占空比范围:

- 较高: 1-10% (低延迟要求, 如工业控制)
- 中等: 0.1-1% (一般传感器网络)
- 极低: 小于0.1% (超低功耗, 如环境监测)

### 1.3 基本监听-休眠周期

最简单的占空比模型:

```
时间线:
|--活跃--|--------休眠--------|--活跃--|--------休眠--------|
  T_on        T_sleep           T_on        T_sleep

T_on: 活跃时间, 检查信道是否有数据
T_sleep: 休眠时间, 无线电关闭
周期: T_cycle = T_on + T_sleep
占空比: DC = T_on / T_cycle
```

核心trade-off: 更长的T_sleep意味着更低功耗但更高延迟, 更短的T_on意味着更低功耗但可能错过数据包。需要在节能和通信质量之间找到平衡。

## 2. 前导码采样方法

### 2.1 基本问题

占空比引入的核心挑战: 发送方不知道接收方什么时候醒来。如果发送方在接收方休眠时发送数据, 数据就会丢失。如何解决这个异步问题?

### 2.2 长前导码方案(LPL)

Low-Power Listening的解决思路很直接: 发送方在数据包前面加一个很长的前导码(preamble), 长度至少等于接收方的休眠周期。这样无论接收方在哪个时刻醒来, 都能检测到前导码, 然后保持活跃等待数据包到来。

```
发送方:
|<-------- 长前导码 (>= T_sleep) -------->|<-- 数据包 -->|

接收方(周期性唤醒):
      __|__          __|__          __|__
     |check|        |check|        |check|.........|接收数据|
     休眠    休眠    休眠    检测到前导码  保持活跃

关键: 前导码长度 >= T_sleep, 保证接收方至少醒来一次能检测到
```

### 2.3 LPL的能耗分析

**接收方能耗**: 每次唤醒执行CCA约128us。信道空闲则立即回到休眠。检测到前导码则保持活跃直到数据包结束。平均情况下大部分唤醒只消耗极短时间, 接收方非常节能。

**发送方能耗**: 必须发送完整的长前导码, 持续时间等于接收方休眠周期。如果休眠周期为500ms, 前导码也要500ms。发送能耗被大幅转嫁到发送方, 这是LPL的主要缺点。

### 2.4 B-MAC协议

B-MAC是LPL的经典实现(加州大学伯克利分校, TinyOS平台):

```python
# B-MAC接收方伪代码
def receiver_loop():
    while True:
        sleep(T_sleep)
        # 周期性唤醒检查
        radio.on()
        energy = radio.cca(duration_us=128)
        if energy > threshold:
            # 检测到信号, 保持活跃接收数据
            packet = radio.receive(timeout=T_max)
            process(packet)
        radio.off()
```

核心机制: CCA检测信道能量, 长前导码持续时间可配置, 随机退避避免碰撞, 无RTS/CTS保持简单。B-MAC在传感器网络领域影响深远, 证明了简单设计的有效性。

### 2.5 短前导码优化(X-MAC)

长前导码的问题是发送方能耗太高。X-MAC将长前导码拆分为多个短前导码:

```
原始长前导码:
|================== 连续前导码 ==================|数据|

X-MAC短前导码:
|前导|gap|前导|gap|前导|gap|前导|gap|前导|gap|前导|数据|

改进点:
- 每个短前导码包含目标地址信息
- 非目标节点收到后可立即回到休眠
- 目标节点收到后发送ACK
- 发送方收到ACK后立即发送数据, 不必等完整周期
```

X-MAC显著降低了发送方能耗, 同时也减少了非目标节点的干扰。

## 3. 同步式占空比

### 3.1 基本思想

如果所有节点同步醒来, 就不需要长前导码:

```
节点A: |活跃|------休眠------|活跃|------休眠------|
节点B: |活跃|------休眠------|活跃|------休眠------|
节点C: |活跃|------休眠------|活跃|------休眠------|
         ^                    ^
      同步唤醒              同步唤醒(此时通信)
```

优势: 无需前导码, 通信效率高, 双方都知道何时通信。劣势: 需要时间同步机制, 时钟漂移需要周期性校正, 调度表较为刚性。

### 3.2 S-MAC协议

S-MAC(Sensor MAC)是同步式占空比的代表:

**虚拟簇**: 相邻节点自动形成簇, 采用相同的醒-睡调度表。

**SYNC报文**: 节点周期性广播SYNC报文, 包含调度信息让邻居对齐。

**活跃期结构**:

```
|<------------ 活跃期 ------------>|<---- 休眠期 ---->|
|SYNC|  RTS/CTS/DATA/ACK 通信窗口  |                  |
```

**自适应监听**: 节点在活跃期末尾侦听到邻居通信时, 额外延长一小段监听时间, 提高接收机会。

### 3.3 T-MAC协议

T-MAC(Timeout MAC)在S-MAC基础上引入动态活跃期:

```
S-MAC(固定活跃期):
|<-- 固定活跃时间 -->|<---- 休眠 ---->|
|通信|  空闲等待...  |                |

T-MAC(动态活跃期):
|<->|<---- 休眠(提前, 因为超时无活动) ---->|
|通信|
```

如果在活跃期内超过一定时间没有检测到任何活动, 提前进入休眠。T-MAC在低负载时比S-MAC节省更多能量, 但高负载时可能因提前休眠增加延迟(early sleeping problem)。

## 4. ContikiMAC: 现代RDC设计

### 4.1 设计理念

ContikiMAC是Contiki OS的默认RDC(Radio Duty Cycling)层, 结合了前导码采样和相位优化:

- 接收方: 周期性唤醒做CCA
- 发送方: 连续发送数据包直到收到ACK
- 相位锁定: 学习邻居的唤醒时刻, 精确发送

### 4.2 双CCA唤醒检查

```
每次唤醒执行两次CCA: CCA1 -- 间隔 -- CCA2

为什么两次?
- 单次CCA可能恰好落在数据包间隙
- 两次间隔等于数据包传输时间的一半
- 确保至少有一次CCA落在数据包传输期间
- 两次都检测到能量才认为信道忙
```

### 4.3 相位锁定优化

ContikiMAC最重要的优化是相位锁定(phase-lock):

```
首次发送: 不知道接收方唤醒时间
|数据|数据|数据|数据|数据|数据|数据|数据|ACK收到!|
                                    ^
                              接收方此时醒来

记录相位后, 后续发送:
             |数据|数据|ACK|
                    ^
              接收方准时醒来, 很快收到

节省: 从发送8个副本减少到2-3个, 能耗降低75%
```

### 4.4 性能数据

ContikiMAC在典型配置下的性能: 唤醒间隔125ms(每秒唤醒8次), 每次CCA约0.4ms, 占空比约0.6%, 有相位锁定时发送延迟小于125ms, 吞吐量约4-8 packets/second。

## 5. WiseMAC协议

### 5.1 基本原理

WiseMAC将前导码采样与同步思想结合: 接收方周期性采样(与LPL相同), 发送方记录接收方的唤醒相位(与同步式相同), 只在接收方即将醒来前发送短前导码。

```
初次通信(不知道相位):
|<---- 长前导码(= 周期) ---->|数据|

记录相位后:
       |短前导码|数据|
前导码从周期长度缩短到最小安全余量
```

### 5.2 适用场景

WiseMAC特别适合: 基础设施网络(固定接入点), 节点唤醒周期稳定(相位估计准确), 单跳通信为主(相位信息只对直接邻居有效), 数据流量单向(传感器向基站上报)。

## 6. 对比分析

### 6.1 方案分类

```
占空比MAC协议
|-- 异步(无需时间同步)
|   |-- 发送方长前导码: B-MAC, LPL
|   |-- 发送方短前导码: X-MAC
|   |-- 接收方采样加相位优化: ContikiMAC, WiseMAC
|-- 同步(需要时间同步)
|   |-- 固定调度: S-MAC
|   |-- 自适应调度: T-MAC
|   |-- TDMA调度: TSCH (见TDMA专题)
```

### 6.2 关键指标对比

| 方案 | 同步需求 | 发送能耗 | 接收能耗 | 延迟 | 复杂度 |
|------|----------|----------|----------|------|--------|
| B-MAC/LPL | 无 | 高(长前导码) | 低 | 中 | 低 |
| X-MAC | 无 | 中 | 低 | 中 | 低 |
| ContikiMAC | 无(相位锁定) | 低到中 | 低 | 低到中 | 中 |
| WiseMAC | 无(相位学习) | 低 | 低 | 低 | 中 |
| S-MAC | 需要 | 低 | 中 | 中 | 中 |
| T-MAC | 需要 | 低 | 低到中 | 中到高 | 中 |

### 6.3 选择建议

- 极简实现: B-MAC(代码最少, 最容易理解和调试)
- 通用平衡: ContikiMAC(综合性能好, 实际部署多)
- 基础设施型: WiseMAC(基站-节点通信, 相位稳定)
- 同步可用: S-MAC/T-MAC(有时间同步基础设施时)
- 确定性要求: TSCH(工业场景, 另文讨论)

## 7. 实际案例: 森林火灾监测

### 7.1 场景需求

500个温度传感器分布在10平方公里森林中, 电池供电目标寿命2年以上, 正常模式每10分钟采集上报, 报警模式需要5秒内通知网关。

### 7.2 ContikiMAC RDC方案设计

```
参数配置:
- 唤醒间隔: 125ms
- CCA时间: 0.4ms x 2
- 占空比: 约0.5%

正常模式能耗计算:
- 无线电监听: 0.5% x 60mW = 0.3mW (平均)
- MCU加传感器: 约0.1mW (平均)
- 总平均功耗: 约0.4mW
- 2节AA电池(约18000mWh)
- 含发送和自放电: 实际寿命约2.5-3年, 满足需求
```

### 7.3 报警传播分析

```
火灾报警场景(最远传感器距网关5跳):
1. 传感器检测到温度异常
2. 立即发送报警, 连续发送直到邻居ACK
3. 邻居最迟125ms内唤醒
4. 邻居收到后立即转发
5. 最大传播延迟: 5 x 125ms + 5 x 10ms = 675ms
6. 远小于5秒要求
```

ContikiMAC同时实现了: 长电池寿命(0.5%占空比, 3年), 快速报警(最大延迟小于1秒), 无需时间同步(异步方案部署简单), 相位优化(重复通信邻居间效率更高)。

## 8. 占空比的挑战与未来

### 8.1 当前挑战

**延迟不确定性**: 占空比天然引入延迟抖动, 对实时性要求高的应用需要权衡。

**休眠节点的可达性**: 泛洪消息可能在目标节点休眠时到达并被错过, 需要重复发送或调度配合。

**多跳累积延迟**: 每跳都可能增加一个唤醒周期的延迟, 多跳网络中累积显著。

**流量突变**: 突发流量时固定占空比可能响应不及, 自适应占空比增加复杂度。

### 8.2 发展趋势

**唤醒无线电(Wake-up Radio)**: 专用超低功耗接收器持续监听, 主无线电完全关闭。收到唤醒信号时才打开主无线电, 可实现接近零延迟加极低功耗, 是占空比的终极进化方向。

**AI自适应调度**: 机器学习预测流量模式, 动态调整占空比参数, 在不同时段自动选择最优唤醒间隔。

**异构占空比**: 不同节点根据角色采用不同策略。中继节点更高占空比保证转发可用性, 叶子节点极低占空比最大化电池寿命。

## 总结

占空比MAC是IoT低功耗通信的基石技术。核心思想很简单: 让无线电大部分时间关闭, 只在需要时短暂开启。但实现细节中有丰富的设计空间:

- 异步方案(LPL/ContikiMAC): 不需要同步, 靠前导码或相位优化解决时序问题
- 同步方案(S-MAC/T-MAC): 节点协调唤醒, 通信效率高但需要维护同步
- 核心trade-off: 占空比越低越省电, 但延迟越高, 可靠性越需要额外机制保证

理解占空比MAC的设计原则, 是理解为什么IoT设备能在纽扣电池上运行数年的关键。
