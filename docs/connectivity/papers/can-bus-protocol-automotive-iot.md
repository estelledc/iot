# CAN总线协议在车联网IoT中的应用
> **难度**：🟡 中级 | **领域**：车载通信 | **阅读时间**：约 20 分钟

## 引言

想象一个班级里30个同学要互相传纸条, 但只有一条走廊。如果两个人同时把纸条扔进走廊, 就会撞在一起。CAN总线的解决方案很聪明: 每张纸条上写了优先级编号, 如果两张纸条同时出发, 编号更小(优先级更高)的自动通过, 另一张自动退回等待重发。而且整个过程中没有纸条损坏 -- 这就是CAN总线的"非破坏性仲裁"机制。

CAN(Controller Area Network, 控制器局域网)是1986年由Bosch开发的串行通信总线, 最初为汽车设计, 如今已成为车辆内部通信的事实标准。随着车联网的发展, CAN总线数据通过网关连接到云端, 成为车辆IoT生态的数据源头。本文将从物理层到应用层全面解析CAN协议及其在车联网中的角色。

## 1. CAN总线概述

### 1.1 历史与基本特性

CAN总线1986年由Bosch发布, 1991年Mercedes-Benz首次量产使用, 1993年成为ISO 11898国际标准。至今全球有数十亿个CAN节点在运行, 覆盖汽车、工业、医疗等领域。

CAN的核心设计理念:

- 多主架构: 没有固定主节点, 任何节点都可以发起通信
- 消息导向: 消息用ID标识, 不是节点地址
- 差分信号: 抗干扰能力强, 适合汽车电磁环境
- 广播通信: 所有节点看到所有消息, 按需过滤
- 非破坏性仲裁: 总线冲突时自动解决, 不丢失数据

### 1.2 CAN网络拓扑

```
  120ohm                                    120ohm
  终端电阻                                   终端电阻
    |                                         |
    +---[ECU-A]---[ECU-B]---[ECU-C]---[ECU-D]---+
                        |
                     CAN_H (高线)
                     CAN_L (低线)
```

所有节点并联连接到两根线(CAN_H和CAN_L), 总线两端各一个120欧姆终端电阻防止信号反射。这种简单的两线式总线结构是CAN在汽车中大规模应用的基础 -- 布线成本低, 可靠性高。

## 2. CAN物理层

### 2.1 差分信号原理

CAN使用差分信号传输, 通过两根线的电压差表示数据:

```
显性状态(Dominant, 逻辑0):
  CAN_H = 3.5V
  CAN_L = 1.5V
  差值 = 2.0V

隐性状态(Recessive, 逻辑1):
  CAN_H = 2.5V
  CAN_L = 2.5V
  差值 = 0V
```

差分信号的关键优势: 外部电磁噪声同时耦合到两根线上, 电压差保持不变, 因此CAN在汽车发动机舱、电机附近等强干扰环境中仍能可靠通信。

"显性"和"隐性"这两个名字也暗示了仲裁机制: 当一个节点发显性(0), 另一个发隐性(1)时, 总线上呈现的是显性状态 -- 显性覆盖隐性。

### 2.2 总线速率与距离

CAN 2.0的速率与总线长度关系:

| 波特率 | 最大总线长度 | 典型应用 |
|--------|-------------|----------|
| 1 Mbps | 25m | 车内动力总成 |
| 500 kbps | 100m | 车身控制 |
| 250 kbps | 250m | 诊断 |
| 125 kbps | 500m | 工业设备 |
| 50 kbps | 1000m | 农业机械 |

速率越高, 一个位的时间越短, 信号在总线上的传播延迟占比越大。如果远端节点还没收到当前位就开始下一位的判断, 仲裁就会出错。这就是高速时总线必须短的根本原因。

### 2.3 终端电阻

总线两端各需120欧姆终端电阻, 用于阻抗匹配防止信号反射, 确保隐性状态时两线电压一致。缺少终端电阻会导致通信不稳定, 这是CAN网络调试中最常见的硬件问题之一。

## 3. CAN帧格式

### 3.1 标准帧(CAN 2.0A)

```
| SOF | ID(11bit) | RTR | IDE | r0 | DLC | Data(0-8B) | CRC | ACK | EOF |
  1      11        1    1    1    4     0-64       16    2     7  (bits)
```

各字段含义:

- SOF: 帧起始, 1个显性位, 用于同步
- ID: 11位标识符, 决定优先级和消息类型
- RTR: 远程传输请求位(请求其他节点发送数据)
- DLC: 数据长度码, 指示0-8字节有效载荷
- Data: 有效载荷, 最多8字节(64位)
- CRC: 15位循环冗余校验, 错误检测
- ACK: 接收确认, 收到正确帧的节点回复显性位
- EOF: 帧结束, 7个隐性位

### 3.2 扩展帧与载荷限制

扩展帧(CAN 2.0B)使用29位ID(11位基础+18位扩展), 可标识约5亿种不同消息, 用于需要更多消息类型的系统如J1939重型车辆。

CAN的8字节载荷限制在设计时足够(一个传感器值+状态), 但现代应用中固件更新等大数据传输需要通过ISO-TP(ISO 15765-2)协议分多帧传输, 效率较低。这也是后来CAN FD诞生的动力。

## 4. 仲裁机制

### 4.1 消息导向通信

CAN总线没有节点地址概念, 消息以ID标识并广播:

```
传统网络: "节点A发送消息给节点B" (点对点)
CAN总线:  "发送ID=0x100的消息" (广播, 各节点按需接收)
```

这种设计的优势: 增删节点不需要修改其他节点配置, 一条消息可被多个节点同时使用(如车速信号被仪表盘、ABS、定速巡航同时需要), 系统灵活性高。

### 4.2 非破坏性仲裁详解

当多个节点同时开始发送时, 逐位比较ID:

```
节点A发送ID: 0x100 = 0001 0000 0000
节点B发送ID: 0x120 = 0001 0010 0000

逐位比较(从高位MSB开始):
  位10-6: 相同, 两者继续
  位5: A发送0(显性), B发送1(隐性)
       总线呈现显性(0)
       B读回总线值=0, 与自己发的1不同
       B知道仲裁失败, 停止发送
       A继续发送剩余帧内容
```

关键原则: ID值越小优先级越高, 显性覆盖隐性, 胜出帧完整传输无损坏。整个仲裁过程在位时间级别完成, 不浪费任何带宽。

### 4.3 优先级设计

汽车中典型的CAN ID分配策略:

| ID范围 | 优先级 | 消息类型 |
|--------|--------|----------|
| 0x000-0x0FF | 最高 | 安全关键(ABS, 气囊) |
| 0x100-0x1FF | 高 | 动力系统(发动机, 变速箱) |
| 0x200-0x3FF | 中 | 底盘(转向, 悬挂) |
| 0x400-0x5FF | 较低 | 车身(车窗, 车灯) |
| 0x600-0x7FF | 最低 | 娱乐/诊断 |

安全相关消息使用最小ID, 确保在总线繁忙时优先传输。

## 5. 汽车CAN网络架构

### 5.1 多总线域划分

一辆现代汽车有多条独立CAN总线, 按功能域划分:

```
[动力CAN 500kbps]        [底盘CAN 500kbps]
  - 发动机ECU               - ABS/ESP模块
  - 变速箱ECU               - 电动助力转向
       |                         |
       +----[中央网关ECU]--------+
       |                         |
[车身CAN 250kbps]        [信息娱乐CAN 500kbps]
  - 车窗/空调/车灯           - 中控屏/音响/导航
```

### 5.2 网关ECU

中央网关是不同CAN总线之间的桥梁, 负责消息路由转发、安全过滤(防止外部接口的非法消息进入核心总线)和协议转换(连接CAN与LIN、FlexRay等)。现代中高端车有70-100个ECU, 经济型也有30-50个。

## 6. CAN在车联网IoT中的角色

### 6.1 车联网数据流

CAN总线是车辆数据的源头:

```
[车内CAN总线] --> [网关ECU] --> [T-Box/OBD] --> [4G/5G] --> [云平台]
```

可采集数据: 发动机转速/车速/油耗/冷却液温度、电池SOC/温度/电流(EV)、车门/安全带/灯光状态、轮胎压力等。

### 6.2 OBD-II标准接口

OBD-II是法规要求的16针标准诊断接口, PIN 6(CAN_H)和PIN 14(CAN_L)提供CAN访问。通过标准PID读取通用数据, 第三方OBD诊断器借此实现远程诊断、车队管理、UBI保险和预测性维护。

## 7. CAN总线安全问题

### 7.1 协议安全缺陷

CAN设计于1986年, 完全没有安全机制: 无认证(任何节点可发任何消息), 无加密(明文广播), 无源地址(无法追溯来源)。一个节点被攻破, 整条总线暴露。

### 7.2 攻击与防护

主要攻击方式: 消息注入(伪造控制消息)、消息重放(录制后不当时机回放)、DoS攻击(高优先级消息洪泛)、诊断命令滥用。

防护方案:

- AUTOSAR SecOC: 在CAN消息中添加MAC(消息认证码)
- 网关防火墙: 域间网关过滤异常消息
- 入侵检测(IDS): 监控总线流量检测异常模式
- HSM: ECU内部硬件安全模块存储密钥
- 安全启动: 防止ECU固件被篡改

SecOC的挑战: 8字节载荷中MAC占用数据空间(如用3字节MAC则只剩5字节数据), 这也是CAN FD(64字节载荷)对安全的间接帮助。

## 8. CAN在工业IoT中的应用

### 8.1 CANopen

CANopen是基于CAN的工业应用层协议, 由CiA(CAN in Automation)组织维护:

- 对象字典: 设备所有参数的标准化组织方式, 索引+子索引定位
- PDO(Process Data Object): 实时数据交换, 事件触发或定时发送
- SDO(Service Data Object): 参数读写, 请求-响应模式
- NMT(Network Management): 设备状态控制(初始化/运行/停止)
- 设备描述(EDS文件): 类似PROFIBUS的GSD, 标准化设备能力描述

广泛用于医疗设备、电梯控制、工程机械和协作机器人。

### 8.2 J1939

SAE J1939是重型车辆和农业机械的CAN应用标准:

- 基于CAN 2.0B(29位扩展ID)
- 固定250kbps速率
- 标准化参数组(PGN): 如发动机转速PGN=61444
- 标准化信号定义(SPN): 如SPN 190=发动机转速
- 覆盖: 卡车、客车、农机、工程机械、船舶
- 车队管理IoT中, J1939是商用车数据采集的主要来源

### 8.3 DeviceNet

Allen-Bradley(Rockwell)的工业CAN协议:

- 北美工业自动化市场广泛使用
- 连接PLC到传感器/执行器
- 总线供电(24V), 设备无需单独电源线
- 简单主从通信, 配置直观

## 9. CAN与车载以太网的对比

| 特性 | CAN 2.0 | 车载以太网100BASE-T1 |
|------|---------|---------------------|
| 速率 | 1Mbps | 100Mbps |
| 有效载荷 | 8字节 | 1500字节 |
| 通信模式 | 事件驱动/广播 | 请求-响应/流 |
| 确定性 | 优先级仲裁 | 需要TSN |
| 布线 | 差分双绞线 | 单对非屏蔽双绞线 |
| 复杂度 | 低 | 高 |
| 成本 | 低 | 中等 |

未来趋势: 以太网作为高速骨干连接域控制器(承载摄像头/ADAS等大数据流), CAN继续服务于简单传感器和执行器的"最后一米", 两者互补共存。

## 10. 实际案例: 车队管理IoT系统

### 10.1 系统架构

```
[车辆端]
  OBD-II诊断器(硬件) --> 插入标准16针接口
  连接CAN总线(500kbps)
  内置蜂窝模块(4G LTE)
  GPS定位模块

[云端平台]
  数据接收网关 --> 消息队列 --> 时序数据库
  分析引擎 --> 规则引擎(告警) --> 可视化仪表盘
```

### 10.2 数据采集流程

OBD-II诊断器读取CAN数据的标准流程:

```
1. 诊断器插入OBD-II接口(驾驶员侧仪表板下方)
2. 连接CAN总线, 自动检测波特率(250k/500k)
3. 发送标准OBD请求帧(ID: 0x7DF, 功能寻址)
4. 接收ECU响应帧(ID: 0x7E8-0x7EF)
5. 按PID规范解析数据:
   PID 0x0C: 发动机转速 = (A*256+B)/4 RPM
   PID 0x0D: 车速 = A km/h
   PID 0x05: 冷却液温度 = A-40 摄氏度
6. 定时(每10秒)打包上传到云端
```

### 10.3 业务价值

车队管理IoT系统基于CAN数据的实际收益:

- 油耗优化: 分析急加速/急刹车频率, 通过驾驶培训节省5-15%燃油
- 维护降本: 基于发动机参数趋势预测性维护, 减少30%非计划停机
- 安全提升: 实时监控驾驶行为, 生成驾驶评分和排名
- 合规管理: 自动记录行驶里程和工时满足法规要求
- 调度优化: 实时位置+车辆状态(空载/满载)支持智能派单

## 总结

CAN总线作为简洁、可靠、低成本的串行通信协议, 近40年来证明了它在汽车电子中的价值。非破坏性仲裁确保了确定性通信, 消息导向模型提供了灵活的系统架构, 差分信号保障了强电磁环境下的可靠性。

在车联网IoT时代, CAN总线是所有车辆数据的源头 -- 通过OBD/T-Box采集后经网关上传云端, 支撑远程诊断、车队管理、预测维护等应用。同时联网也带来了安全挑战, 需要SecOC、IDS等新机制。架构上正向以太网骨干+CAN边缘的分层模式演进, 但CAN在简单传感器连接场景中将长期存在。
