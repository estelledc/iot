# NB-IoT 规模部署：从标准到实践

> 难度：🟡 进阶 | 领域：低功耗广域网 · 蜂窝物联网 | 更新：2025-06

---

## 一句话总结

NB-IoT（窄带物联网）是运营商在已有 LTE 基站上通过软件升级部署的 LPWAN 技术。中国已建成全球最大的 NB-IoT 网络（150 万+ 基站，4 亿+ 连接），智能抄表和智能烟感是两大杀手级应用。本文从协议栈到真实部署数据，讲清楚 NB-IoT 从标准到实践中的关键细节。

---

## 从日常场景说起

你家的水表和燃气表是怎么"抄表"的？在过去，是抄表员挨家挨户上门读数。后来有了"远传表"，表上装个通信模块，数据自动传到水务/燃气公司。

问题是：水表在地下管井里，燃气表在厨房角落的铁柜子里——信号穿透要求极高。而且这些表用电池供电，换一次电池成本几十元（含人工），电池至少要撑 6-8 年。

WiFi？信号穿不过管井的钢筋混凝土。蓝牙？传不了几百米到最近的基站。4G？功耗太大，电池半年就没了。

NB-IoT 正是为这种"深覆盖 + 超低功耗 + 海量连接"场景设计的。它的最大耦合损耗（MCL）达到 164 dB——比传统 LTE 多 20 dB，意味着信号能多穿透 1-2 面混凝土墙或几米地下土层。

## NB-IoT 协议栈架构

### 与 LTE 的关系

NB-IoT 不是一个独立的通信系统，而是 LTE 的一个"简化子集"。它复用了 LTE 的核心网（EPC）和部分接入网（eNodeB），但物理层和 MAC 层做了大幅简化。

```
┌─────────────────────────────────────┐
│           应用层 (CoAP/MQTT)        │
├─────────────────────────────────────┤
│           传输层 (UDP/TCP)          │
├─────────────────────────────────────┤
│           网络层 (IP)               │
├─────────────────────────────────────┤
│           PDCP (分组数据汇聚)       │
├─────────────────────────────────────┤
│           RLC (无线链路控制)         │
├─────────────────────────────────────┤
│           MAC (媒体接入控制)         │
├─────────────────────────────────────┤
│      物理层 (NPBCH/NPDCCH/         │
│       NPDSCH/NPUSCH/NPRACH)         │
└─────────────────────────────────────┘
```

NB-IoT 物理层有自己独立的信道名称（前缀 N 表示 Narrowband）：
- **NPBCH**（窄带物理广播信道）：广播系统信息
- **NPDCCH**（窄带物理下行控制信道）：调度和控制指令
- **NPDSCH**（窄带物理下行共享信道）：下行数据传输
- **NPUSCH**（窄带物理上行共享信道）：上行数据传输
- **NPRACH**（窄带物理随机接入信道）：设备发起连接请求

### 关键协议特性

**单载波传输**：NB-IoT 上行使用 SC-FDMA（单载波频分多址），每个设备只占用一个 15 kHz 子载波（单 tone 模式）或 12 个子载波（多 tone 模式，180 kHz）。单 tone 模式下，一个 180 kHz NB-IoT 载波可以同时服务 12 个设备的上行——虽然每个设备的速率极低（~15 kbps），但足够传一个水表读数。

**重复传输（Repetition）**：这是 NB-IoT 实现深覆盖的核心机制。同一个数据包可以重复发送最多 2048 次。接收方将所有重复的副本做"软合并"（soft combining），信噪比随重复次数近似线性增长（以 dB 计）。2048 次重复 ≈ 33 dB 的额外增益。代价是传输时间拉长（一个包可能要几秒甚至几十秒才能送完），但对于水表这种非实时场景完全可以接受。

**PSM（省电模式）**：设备完成数据传输后，通知核心网"我要去睡了"，然后进入极深度睡眠（功耗 < 1 μA）。核心网保留设备的上下文（不需要重新注册），但设备在 PSM 期间完全不可达——网络侧无法唤醒它。设备按照预设的定时器（如每小时、每天）自己醒来，发数据，再睡。

**eDRX（扩展不连续接收）**：介于 PSM（完全不可达）和持续在线之间。设备周期性醒来（eDRX 周期最长 2.91 小时），短暂监听是否有下行数据，没有则继续睡。eDRX 使得设备在大部分时间睡眠的同时，仍保留一定的下行可达性。

## 三种部署模式

NB-IoT 可以在运营商的 LTE 频段中以三种方式部署：

### 带内部署（In-band）

在 LTE 载波的带宽内"借"一个 180 kHz 的 PRB（物理资源块）给 NB-IoT 使用。

```
LTE 载波 (20 MHz = 100 PRBs)
┌──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┐
│  │  │  │NB│  │  │  │  │  │  │  │  │ ← NB-IoT 占一个 PRB
│  │  │  │  │  │  │  │  │  │  │  │  │
└──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┘
```

优点：不需要额外频谱，利用已有 LTE 频谱。
缺点：NB-IoT 和 LTE 数据竞争同一频谱资源，在 LTE 负载高时 NB-IoT 性能可能下降。

### 保护带部署（Guard-band）

利用 LTE 载波边缘的保护带（guard band）部署 NB-IoT。LTE 每侧保护带有 ~100 kHz 空闲频谱（20 MHz LTE 实际只用 18 MHz），刚好够放一个 NB-IoT 载波。

优点：不占用 LTE 的有效频谱资源，NB-IoT 和 LTE 互不干扰。
缺点：保护带频谱有限，每个 LTE 载波最多部署 1-2 个 NB-IoT 载波。

### 独立部署（Standalone）

在独立的频谱上部署 NB-IoT（如退网的 GSM 频段，每个 GSM 载波 200 kHz 刚好容纳一个 NB-IoT 的 180 kHz）。

优点：完全独立，不受 LTE 影响。
缺点：需要专用频谱（GSM 退网释放的频谱最常用）。

### 中国三大运营商的部署选择

| 运营商 | 部署模式 | 频段 | 理由 |
|--------|---------|------|------|
| 中国移动 | 带内/独立 | Band 8 (900 MHz) | 复用 GSM 900 频段 |
| 中国电信 | 独立 | Band 5 (850 MHz) | 利用退网 CDMA 800 频段 |
| 中国联通 | 保护带 | Band 8 (900 MHz) | LTE 900 保护带 |

## 覆盖增强技术详解

NB-IoT 的"深覆盖"不是一个单一技术，而是多种增强手段叠加的结果：

### 技术组合与覆盖增益

| 增强技术 | 覆盖增益 | 代价 |
|----------|---------|------|
| 重复传输 (最大 2048 次) | 最高 ~33 dB | 延迟大幅增加 |
| 功率增强 (Power Boosting, 6 dB) | 6 dB | 占用更多下行功率预算 |
| 低阶调制 (QPSK → π/2-BPSK) | 3-4 dB | 速率降低 |
| 单 tone 上行 (窄带集中能量) | 3-6 dB | 只支持一个设备 |
| 接收分集 (基站侧多天线) | 3-5 dB | 基站侧成本 |
| **总计** | **≈ 52+ dB** | 速率和延迟大幅牺牲 |

在最极端的覆盖增强模式下（所有技术全开），NB-IoT 的等效速率可能降到 **几十 bps**，单次传输时间可达 **数十秒到数分钟**。但对于水表抄表（每小时发一个几字节的读数），这完全够用。

### 实测覆盖案例

华为和中国移动在 2018-2019 年做过大规模覆盖测试：

- 地下车库 B3 层（地下约 9 米）：NB-IoT 信号成功穿透，RSRP -130 dBm，数据传输正常
- 地下管井（金属盖板+混凝土，地下 1.5 米）：NB-IoT 可达，传统 LTE 不可达
- 室内深处（距基站 1.5 km，穿过 3 面承重墙）：NB-IoT 正常通信

## 真实部署数据

### 中国 NB-IoT 网络规模（截至 2024 年底）

| 指标 | 中国移动 | 中国电信 | 中国联通 | 合计 |
|------|---------|---------|---------|------|
| NB-IoT 基站数 | 70 万+ | 50 万+ | 30 万+ | 150 万+ |
| NB-IoT 连接数 | 2.5 亿+ | 1.2 亿+ | 0.5 亿+ | 4 亿+ |
| 网络覆盖率 | 全国乡镇级 | 全国县级 | 城市为主 | — |
| 主要应用 | 智能表计 | 智能烟感 | 资产追踪 | — |

### 杀手级应用 1：智能水表/气表

中国智能水表/气表的 NB-IoT 化是全球最大的 LPWAN 应用：

- 累计部署 2 亿+ NB-IoT 智能水表
- 典型产品：宁波水表 NB-IoT 超声波水表、新天科技 NB-IoT 燃气表
- 通信参数：每 15 分钟上报一次读数（~50 字节），每天上报 96 次
- 电池寿命要求：≥ 6 年（AA 锂电池，3.6V 8500mAh）
- 实际电池寿命实测：使用 PSM 模式，5 年后剩余电量 > 50%
- 网络覆盖率：地下管井场景覆盖率 > 98%（使用覆盖增强模式）

### 杀手级应用 2：智能烟感

中国消防法要求所有出租房屋安装烟感报警器，NB-IoT 智能烟感是合规方案之一：

- 累计部署 5000 万+ 台
- 典型产品：赛特威尔 NB-IoT 烟感
- 工作模式：平时每天心跳上报 1-2 次（确认设备在线），报警时即时上报
- 电池寿命：≥ 3 年（要求较低，因为烟感本身 3 年更换一次）
- 关键要求：报警延迟 < 10 秒（从烟雾触发到云平台收到报警）

### 其他应用场景

| 应用 | 典型部署规模 | 上报频率 | 数据量/次 | 关键要求 |
|------|-------------|---------|----------|---------|
| 智能路灯 | 数百万 | 每 10 分钟 | ~100 字节 | 双向控制 (开/关/调光) |
| 智能停车 | 数十万 | 车辆进出时 | ~50 字节 | 低延迟检测 |
| 共享单车锁 | 数百万 | 用户扫码时 | ~200 字节 | 移动场景 (NB-IoT 弱项) |
| 农业土壤监测 | 数万 | 每小时 | ~30 字节 | 偏远地区覆盖 |
| 物流追踪 | 数十万 | 每 30 分钟 | ~100 字节 | 需要定位 |

## NB-IoT 的局限与挑战

### 延迟

NB-IoT 不适合对延迟敏感的应用。从设备唤醒到数据成功上传，典型延迟：

| 场景 | 延迟 |
|------|------|
| 正常覆盖，PSM 唤醒 | 2-5 秒 |
| 边缘覆盖，重复传输 | 10-30 秒 |
| 极端覆盖增强 | 30-120 秒 |
| eDRX 周期结束 → 接收下行 | 0 到 2.91 小时不等 |

这意味着 NB-IoT 不适合做实时控制（如电机调速）、不适合做紧急报警（延迟不可预测）。智能烟感之所以能用，是因为报警走的是 "立即唤醒发送"路径（不等 PSM/eDRX），延迟可控在 5-10 秒。

### 移动性

NB-IoT 不支持小区切换（handover）。设备在移动中从一个基站覆盖区域进入另一个时，需要重新进行小区搜索和连接建立，导致数据中断。

这使得 NB-IoT 不适合移动场景——共享单车的 NB-IoT 锁经常在骑行中断连，到目的地后才重新上报。需要移动中持续通信的场景应该选 LTE-M 或 RedCap。

### 下行通信限制

NB-IoT 的设计重点是上行（设备 → 云端）。下行通信有较多限制：

- PSM 期间完全不可达
- eDRX 期间只有醒来窗口可达
- 下行速率低（26 kbps）
- 不支持大块下行数据传输（如 OTA 升级困难）

如果应用需要频繁的下行控制（如远程开关灯），NB-IoT 可以工作但体验不佳。

### 功耗优化的实践经验

理论上 NB-IoT 电池可以撑 10 年，但实际中有很多"坑"：

1. **注册/附着消耗**：设备首次开机或从 PSM 醒来后的注册流程消耗较大（信令交互多），如果网络覆盖差导致注册频繁失败重试，功耗会急剧上升
2. **基站寻呼风暴**：运营商网络配置不当可能导致频繁寻呼，打断设备的 PSM 睡眠
3. **eDRX 周期选择**：周期太短则频繁唤醒耗电，太长则下行延迟大
4. **信号边缘区**：覆盖差的地方发送功率拉满 + 重复传输次数增多，单次上报功耗可能是正常覆盖的 10-50 倍

实际经验：在信号好的位置（RSRP > -100 dBm），AA 电池确实能撑 6-8 年；在信号差的位置（RSRP < -130 dBm），电池可能 2-3 年就耗尽。

## NB-IoT 的演进方向

### 3GPP 持续优化

| 版本 | 年份 | 关键改进 |
|------|------|---------|
| Release 13 | 2016 | NB-IoT 首版：基础功能 |
| Release 14 | 2017 | 多载波、定位（OTDOA）、多播 |
| Release 15 | 2018 | TDD 支持、WUS（唤醒信号）、功耗优化 |
| Release 16 | 2020 | 更小数据包传输、群播增强 |
| Release 17 | 2022 | 卫星回传（NTN）、覆盖增强、RedCap 共存 |
| Release 18 | 2024 | 与 5G 核心网融合、进一步功耗优化 |

### NB-IoT over Satellite（NTN）

Release 17 引入了 NB-IoT 的非地面网络（NTN）支持——让 NB-IoT 设备可以通过 LEO 卫星通信。这是 NB-IoT 最激动人心的演进方向之一。

技术挑战：
- **传播延迟**：LEO 卫星高度 ~600 km，往返延迟 ~4 ms（比地面 NB-IoT 的 < 1 ms 大得多）→ 需要修改 HARQ 定时
- **多普勒频移**：卫星高速移动（~7.6 km/s），频率偏移可达 ±24 kHz → 设备需要预补偿
- **大覆盖区**：一颗卫星覆盖几百公里直径的地面区域，同时服务大量设备 → NPRACH 冲突增加

详见 [卫星物联网](satellite-iot.md)。
