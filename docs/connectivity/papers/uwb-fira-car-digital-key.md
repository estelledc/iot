# UWB FiRa数字车钥匙技术方案
> **难度**：🔴 高级 | **领域**：UWB应用 | **阅读时间**：约 22 分钟

## 引言

想象你到了一个高档酒店，房间门不需要刷卡也不需要掏钥匙——你只是走到门前，门就自动识别你并打开。你不用担心别人站在旁边就能蹭你的房间，因为系统精确知道"你本人就站在门前"而不是"你的信号在附近"。这就是 UWB 数字车钥匙的核心体验：当你带着手机走向自己的车，车门在你到达时精确解锁，坐进驾驶座时允许启动，而不是只要手机在车附近就全部打开。

传统车钥匙从纯机械钥匙发展到遥控钥匙，再到 BLE 智能钥匙，每一代都增加了便利性。但 BLE 钥匙有一个严重安全漏洞——中继攻击(Relay Attack)。UWB 数字车钥匙通过精确测距从物理层面解决了这个问题，成为汽车行业安全无钥匙进入的首选技术方案。

本文深入解析 UWB 数字车钥匙的技术架构、安全机制和实现方案。

## 1. 车钥匙技术演进

### 1.1 四代车钥匙

```
第1代: 机械钥匙
  插入 -> 转动 -> 启动
  安全性: 物理复制难度

第2代: 遥控钥匙(RF Fob)
  按键 -> 315/433MHz信号 -> 车辆解锁
  安全性: 滚动码防重放

第3代: BLE智能钥匙(被动进入)
  靠近 -> BLE自动识别 -> 解锁(无需按键)
  安全性: 加密认证, 但不防中继攻击

第4代: UWB数字钥匙
  靠近 -> BLE认证 + UWB精确测距 -> 按区域解锁
  安全性: 物理距离验证, 中继攻击无效
```

### 1.2 每代的进步与局限

机械钥匙需要物理接触，安全但不便利。遥控钥匙增加了远程控制，但早期固定码容易被复制，滚动码部分解决了这个问题。BLE 智能钥匙实现了无感进入——钥匙在口袋里就能自动解锁，但引入了中继攻击风险。UWB 数字钥匙在保持无感便利的同时，从物理层面证明钥匙确实在车旁。

## 2. 中继攻击问题

### 2.1 什么是中继攻击

中继攻击(Relay Attack)利用两个协作设备将车与钥匙之间的通信信号远程转发：

```
正常场景:
  [车] <--BLE 2m--> [手机/钥匙]  --> 解锁(合理)

中继攻击:
  [车] <--BLE--> [攻击设备A]  ~~~~无线转发~~~~
                              [攻击设备B] <--BLE--> [手机/钥匙]
  |<----- 实际距离可能100米 ------>|

  车认为钥匙在旁边 --> 解锁(非法!)
```

攻击者 A 站在车旁，攻击者 B 跟在车主身后(比如在超市里)。A 的设备将车的 BLE 信号通过无线(WiFi/4G)转发给 B，B 再将信号传给车主的手机；手机的响应原路返回。整个过程对车和手机都透明，车被骗以为钥匙就在旁边。

### 2.2 为什么BLE无法防御

BLE 的安全机制(加密、认证)验证的是"身份"而非"距离"。中继攻击不破解加密，不伪造身份，只是把合法通信原样转发。BLE 没有可靠的测距能力(RSSI 误差太大，容易伪造)，因此无法检测中继。

### 2.3 UWB如何解决

UWB 的物理层测距精度为厘米级，中继转发会引入可检测的时间延迟：

```
UWB测距验证距离:
  信号往返时间(合法): ~6.7ns (对应1m距离)
  中继额外延迟: >100ns (对应>15m误差)

UWB接收端检测:
  测得距离 = 1m, 合法阈值 = 5m --> 通过
  测得距离 = 20m(中继), 合法阈值 = 5m --> 拒绝
```

中继设备接收、处理和转发 UWB 信号至少引入 100 纳秒延迟，对应 15 米以上的距离误差。UWB 系统设置 5 米的合法阈值，中继攻击会被立即检测到。

## 3. 标准化组织与规范

### 3.1 FiRa联盟

FiRa(Fine Ranging)联盟成立于 2019 年，由 NXP、三星、索尼、博世等公司发起，专注 UWB 测距技术的标准化和互操作性：

```
FiRa联盟角色:
- 定义UWB测距协议和安全配置文件
- 制定设备互操作性测试规范
- 发布用例规范(数字车钥匙、门禁、寻物等)
- 认证程序: 确保不同厂商的UWB设备能互通

主要成员:
  芯片厂商: NXP, Qorvo, Samsung, Apple, STMicro
  汽车厂商: BMW, VW, Hyundai, Continental
  手机厂商: Apple, Samsung, Google, Xiaomi
```

### 3.2 CCC数字钥匙3.0

CCC(Car Connectivity Consortium)专注汽车连接标准，其 Digital Key Release 3.0 是 UWB 车钥匙的核心规范：

```
CCC Digital Key 3.0架构:

+-------------------+     +-------------------+
|     手机端         |     |     车端           |
|                   |     |                   |
| +------+ +-----+ |     | +------+ +------+ |
| | BLE  | | UWB | |     | | BLE  | | UWB  | |
| +------+ +-----+ |     | +------+ +------+ |
| +---------------+ |     | +------+ +------+ |
| | Secure Element| |     | | SE   | | ECU  | |
| +---------------+ |     | +------+ +------+ |
+-------------------+     +-------------------+
        |                         |
    BLE: 身份认证            UWB: 距离验证
    UWB: 安全测距            动作: 解锁/启动
```

CCC 3.0 要求: 测距精度优于 30cm、支持 STS 安全测距(防中继)、支持被动进入(Passive Entry)和被动启动(Passive Start)、跨厂商互操作(不同品牌手机开不同品牌的车)。

## 4. 系统架构

### 4.1 车端硬件

车端需要部署多个 UWB 锚点以实现精确的手机定位：

```
车辆UWB锚点部署(典型4-8个):

        前方
   [A1]------[A2]       前保险杠区域(迎宾感应)
    |          |
    |   车厢   |
   [A3]      [A4]       B柱位置(车门区域)
    |          |
    |          |
   [A5]------[A6]       后保险杠区域(后备箱)
    |          |
   [A7]      [A8]       车内(驾驶座/副驾判断)

每个锚点包含: UWB收发器 + MCU + 天线
锚点通过CAN总线连接车辆ECU
```

此外车端还需要 BLE 模块(用于发现和认证通道)以及安全元件 SE(存储加密密钥和证书)。

### 4.2 手机端实现

```
手机端组件:
+------------------------------------------+
|  数字车钥匙应用(CarKey/Digital Key)       |
|  +--------------------------------------+|
|  | 钥匙凭证管理 | 安全通道 | 位置服务   ||
|  +--------------------------------------+|
|  +----------+ +--------+ +------------+ |
|  | BLE协议栈 | | UWB API| | SE访问接口 | |
|  +----------+ +--------+ +------------+ |
|  +----------+ +--------+ +------------+ |
|  | BLE芯片  | | UWB芯片| | 安全元件SE | |
|  +----------+ +--------+ +------------+ |
+------------------------------------------+

Apple: U1/U2芯片 + CarKey(Wallet集成)
Samsung: Exynos UWB + Digital Key
Google: Android UWB API + Digital Car Key框架
```

### 4.3 安全元件(SE)的角色

安全元件是一个防篡改硬件芯片，存储和处理敏感数据。在数字车钥匙方案中，SE 存储钥匙凭证(私钥)，执行加密运算(签名、验签)，保护密钥免受物理攻击(侧信道、故障注入)，提供安全的密钥生命周期管理(发放、更新、撤销)。

## 5. 操作流程

### 5.1 完整的解锁流程

```
手机接近车辆的完整交互:

阶段1 - BLE发现(距离~10m):
  手机 <--BLE广播--> 车辆
  建立BLE连接

阶段2 - 身份认证(BLE通道):
  手机SE <--加密挑战/响应--> 车辆SE
  双向认证: 车确认手机身份, 手机确认车身份
  协商STS密钥(用于UWB安全测距)

阶段3 - UWB安全测距:
  手机UWB <--STS保护的测距帧--> 车辆UWB锚点
  多个锚点同时测距 -> 三角定位手机位置
  验证距离在合法范围内

阶段4 - 位置判断与动作执行:
  车辆ECU根据手机位置执行对应动作
  (详见下节分区动作)

阶段5 - 持续跟踪:
  UWB持续测距跟踪手机移动
  手机离开 -> 自动落锁
```

### 5.2 STS安全测距

STS(Scrambled Timestamp Sequence)是 IEEE 802.15.4z 定义的关键安全特性:

```
STS工作原理:

1. BLE认证阶段协商共享密钥K
2. 双方用K生成相同的伪随机STS序列
3. UWB测距帧中嵌入STS

发送方:
  [前导码] [SFD] [STS(伪随机)] [PHR] [数据]
                  |
           只有知道K才能生成正确的STS

接收方:
  用本地生成的STS与接收信号相关
  - 匹配: 合法测距, 提取精确到达时间
  - 不匹配: 伪造/重放攻击, 拒绝
```

STS 保证了只有预共享密钥的合法设备才能参与测距，攻击者无法伪造或篡改测距结果。

## 6. 分区动作机制

### 6.1 车辆周围分区

UWB 多锚点定位可以将车辆周围划分为多个精确区域，每个区域触发不同动作：

```
车辆分区示意(俯视图):

                  5m
         +------------------+
         |   迎宾区(3-5m)    |
         |  唤醒车辆系统     |
    +----+--+          +--+----+
    | 左门区 |   车辆   | 右门区 |
    | <1m   |          | <1m   |
    | 解锁   |          | 解锁   |
    | 左门   |          | 右门   |
    +----+--+          +--+----+
         |   后备箱区(<1m) |
         |   打开后备箱     |
         +------------------+

         车内分区:
         +--------+--------+
         | 驾驶座  | 副驾座  |
         | 允许启动 |        |
         +--------+--------+
         | 后排左  | 后排右  |
         +--------+--------+
```

### 6.2 分区动作对应表

| 区域 | 距离阈值 | 触发动作 |
|------|---------|---------|
| 迎宾区 | 3-5m | 唤醒车辆系统、点亮迎宾灯 |
| 车门区 | 小于1m | 解锁对应侧车门 |
| 后备箱区 | 小于1m | 解锁后备箱(可配合脚踢感应) |
| 驾驶座 | 车内 | 允许一键启动 |
| 离开区 | 大于5m | 自动落锁(延迟数秒确认) |

这种分区策略比传统的"全开全锁"更安全更节能: 你走向左门，只有左门解锁；坐进驾驶座，才允许启动；走开后自动落锁。

## 7. BLE与UWB的协同

### 7.1 为什么需要双协议

UWB 和 BLE 各有所长，数字车钥匙方案充分利用两者优势：

```
BLE的角色:
- 低功耗始终监听: 车辆BLE持续扫描等待手机出现
- 设备发现: 第一时间发现手机接近
- 认证通道: 双向身份验证和密钥交换
- 数据交换: 传输钥匙凭证、配置参数

UWB的角色:
- 精确测距: 厘米级距离测量
- 中继防护: 物理距离验证
- 位置感知: 确定手机在车辆的哪个区域
- 实时跟踪: 持续更新手机位置
```

### 7.2 协同时序

```
状态机:

[车辆待机] -- BLE扫描(低功耗, <1mA) -->
  |
发现手机BLE广播
  |
[BLE连接] -- 建立加密通道 -->
  |
身份认证通过
  |
[UWB激活] -- 协商STS密钥, 启动测距 -->
  |
手机在有效范围内
  |
[动作执行] -- 根据位置解锁/启动 -->
  |
手机离开范围
  |
[UWB休眠] -- 停止测距, 落锁 -->
  |
BLE断开
  |
[车辆待机] -- 恢复BLE扫描
```

这种设计使车辆在等待状态的功耗极低(仅 BLE 扫描)，只在需要精确定位时才激活 UWB。

### 9.2 推广挑战

数字车钥匙的全面普及面临几个实际挑战: 手机覆盖率方面并非所有手机都有 UWB 芯片，虽然高端机型采用率在增加但中低端机型仍缺失；车辆需要 4-8 个 UWB 锚点增加了硬件成本约 50-100 美元/车；多手机冲突问题即车内有多部手机时系统需要判断谁是驾驶员；标准化仍在演进中部分高级功能(如远程泊车授权)尚未完全标准化。

## 10. 备用方案与边界场景

### 10.1 手机没电怎么办

这是用户最关心的问题。CCC 3.0 规范要求支持 NFC 备用模式：

```
NFC备用模式(Express Mode):
- 手机电池完全耗尽后, SE仍可从NFC读卡器获取能量
- 车门把手内置NFC读卡器
- 用户将手机贴近门把手
- NFC供能给SE -> SE响应认证 -> 解锁
- 类似Apple Pay的Express Transit模式

限制:
- 只能解锁, 不能启动(需要完整系统)
- 需要物理接触(NFC范围<5cm)
- 安全性: NFC范围极短, 中继攻击不实际
```

### 10.2 手机壳对UWB的影响

厚重的手机壳(特别是金属材质)可能衰减 UWB 信号，影响测距精度。CCC 规范要求系统在信号质量下降时自动增加测距次数以提高精度，并在极端情况下回退到 BLE 距离估算加用户确认的模式。

### 10.3 多用户场景

当多个家庭成员的手机同时在车附近时，车辆需要判断哪部手机在驾驶座位置来决定允许谁启动。UWB 的精确定位能力使这成为可能——系统可以分辨"在驾驶座"和"在副驾驶座"的手机。

## 总结

UWB 数字车钥匙通过 BLE 身份认证和 UWB 精确测距的双层架构，在保持无感便利的同时从物理层面解决了中继攻击问题。FiRa 联盟和 CCC Digital Key 3.0 规范推动了跨品牌互操作，使用户的手机能够成为通用车钥匙。随着 UWB 芯片成本下降和手机覆盖率提升，数字车钥匙有望在未来几年内成为标配。技术上，BLE 负责低功耗发现和身份认证，UWB 负责精确测距和中继防护，安全元件保护密钥安全——三者协同构成了完整的安全无钥匙进入方案。
